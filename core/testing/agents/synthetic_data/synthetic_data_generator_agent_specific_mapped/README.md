# Synthetic Data Generator ‚Äî Mapping-Based Agent

Generates BigQuery-ready synthetic SQL data from design team mapping files.
No sample data required ‚Äî Claude generates purely from column definitions.

---

## Architecture

```
Upstream UI
    ‚îÇ  User enters: "model_SCRUM-5_20260413_23_mapping.csv"
    ‚îÇ  Hits "Generate Synthetic Data"
    ‚ñº
trigger_config.yaml  ‚Üê updated by upstream UI
    ‚îÇ
    ‚ñº
trigger.py           ‚Üê upstream team calls this
    ‚îÇ  reads trigger_config.yaml (which files to process)
    ‚îÇ  reads config.yaml (our settings)
    ‚îÇ  calls main.py
    ‚ñº
src/main.py          ‚Üê our orchestrator
    ‚îÇ
    ‚îú‚îÄ‚îÄ mapping_loader.py   reads design team's mapping CSV
    ‚îÇ                       groups columns by target_table
    ‚îÇ
    ‚îú‚îÄ‚îÄ schema_builder.py   builds column profile from mapping
    ‚îÇ                       (BQ types, transformation logic, flags)
    ‚îÇ
    ‚îú‚îÄ‚îÄ generator_agent.py  Claude generates realistic rows
    ‚îÇ                       using core/utilities/llm factory
    ‚îÇ                       (complete_with_context + prompt caching)
    ‚îÇ
    ‚îî‚îÄ‚îÄ sql_writer.py       writes BQ INSERT SQL
                            one file per target_table
```

## Config Files

### `config.yaml` ‚Äî Our Team


## Input ‚Äî Mapping CSV Format

Mapping files come from: `core/design/agents/data_model/output/`

| Column | Used for |
|---|---|
| `target_table` | BQ table name ‚Üí SQL output filename |
| `target_column` | Column name in INSERT |
| `target_data_type` | BQ native type (INT64, STRING, NUMERIC, DATE, TIMESTAMP) |
| `transformation_logic` | Claude understands how data is cast/transformed |
| `is_partition_column` | Noted in SQL header, dates kept in range |
| `is_cluster_column` | Noted in SQL header, good cardinality ensured |
| `notes` | Claude uses this for realistic value generation |

One mapping file can have **multiple target_tables** ‚Äî each produces its own SQL file.

## Upstream Integration

The upstream team only needs two things:

**1. Update `trigger_config.yaml`** when user hits "Generate":
```yaml
mapping_files:
  - "model_SCRUM-5_20260413_23_mapping.csv"
```

**2. Call `trigger.py`:**
```python
# Option A ‚Äî import and call
from trigger import run_trigger
run_trigger()

# Option B ‚Äî shell
import subprocess
subprocess.run(["python", "trigger.py"], check=True)
```

Our code never changes regardless of how upstream evolves.

## LLM Utility

Uses `core/utilities/llm` factory ‚Äî consistent with team standards:
- Provider: `anthropic` (claude-sonnet-4-6 default)
- Method: `complete_with_context()` with prompt caching
- Column profile is cached server-side ‚Äî batches 2+ reuse cache (~10x cheaper)
- Swap provider: change `model` in `config.yaml` ‚Äî zero code changes
