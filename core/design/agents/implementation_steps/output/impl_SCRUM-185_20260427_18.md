## ⚠️ Open Blockers

> **These blockers must be resolved before implementation is marked complete. Do not promote to production until each item has a recorded resolution.**

1. **Latency SLA Undefined** — No maximum acceptable data lag is defined for either the hourly or daily cadence. If any table requires near-real-time freshness under five minutes, the pattern must be revisited from batch to hybrid (Datastream). **Action required: Confirm acceptable lag per table with Shruthi B before architecture is finalised.**

2. **Table Expansion Scope Unconfirmed** — Requirements identify a single source table (`agentichub.employees`). If additional tables on the Cloud SQL instance will be onboarded, per-table watermark column identification, schema validation configuration registries, Composer DAG parameterisation, and a table onboarding runbook must be added to scope. **Action required: Confirm the complete agreed-upon table list before implementation begins.**

3. **Data Sensitivity Classification Missing** — No PII, PCI-DSS, or confidentiality classification has been provided for the `employees` table. Given that the table contains names, contact details, and compensation data, CMEK encryption, BigQuery column-level security policies, and a VPC Service Controls perimeter may be mandatory. **Action required: Obtain data sensitivity classification before the pipeline is deployed to production.**

## ⚠️ Risks & Mitigations

> *Reference material only — not implemented as steps.*

| Risk | Mitigation |
|---|---|
| **Schema Drift** | Pre-load schema validation in the Cloud Run Job detects type mismatches and missing fields before every load; new-column detection triggers the hold-and-alert policy, halting the load and notifying the pipeline owner via Cloud Monitoring; BigQuery schema definitions are managed as Terraform resources enabling controlled schema migration through code review and approval workflows. |
| **Historical Load Data Loss** | Idempotent chunked extraction partitioned by `employee_id` range with per-chunk row count validation; chunk completion state persisted in `pipeline_control_log` (keyed by chunk range and `load_run_id`); re-run logic resumes from the last successfully completed chunk, preventing re-loading of completed ranges and duplicate data. |
| **Watermark Accuracy & Incremental Completeness** | Confirm MySQL-side application enforcement of `updated_date` on all write paths before go-live; implement a weekly reconciliation Cloud Run Job comparing BigQuery and MySQL total record counts and a random sample checksum; alert via Cloud Monitoring when discrepancy exceeds 0.1% of total row count. |

## Phase 3: Historical Load — Cloud Run Ingestion Job

**Objective:** Implement an idempotent, chunked full-snapshot extraction from `agentichub.employees` to GCS as Parquet, with per-chunk state tracking.

1. **Implement Cloud SQL Auth Proxy connection layer** — In the Python 3.11 Cloud Run Job, configure the SQLAlchemy engine using PyMySQL dialect over the Cloud SQL Auth Proxy Unix socket. Parameterise `host`, `port`, `database`, `username`, and `instance_connection_name` via environment variables. Do not hardcode credentials.

2. **Implement employee_id range chunking logic** — Determine `MIN(EMPLOYEE_ID)` and `MAX(EMPLOYEE_ID)` from the source at the start of each historical load run. Divide the range into chunks of a configurable size (default: 10,000 rows). For each chunk, record a `PENDING` entry in `pipeline_control_log` keyed by `load_run_id` and chunk bounds before extraction begins.

3. **Implement idempotent chunk extraction** — Before extracting a chunk, query `pipeline_control_log` for the current `load_run_id` and chunk bounds. If `chunk_status = COMPLETE`, skip the chunk. If `IN_PROGRESS` or `PENDING`, execute:
   ```sql
   SELECT * FROM agentichub.employees
   WHERE EMPLOYEE_ID >= :chunk_start AND EMPLOYEE_ID < :chunk_end;
   ```
   Update `chunk_status` to `IN_PROGRESS` before extraction and `COMPLETE` on success.

4. **Implement per-chunk row count validation** — After extracting each chunk, compare the row count returned by SQLAlchemy against:
   ```sql
   SELECT COUNT(*) FROM agentichub.employees
   WHERE EMPLOYEE_ID >= :chunk_start AND EMPLOYEE_ID < :chunk_end;
   ```
   If counts diverge, mark the chunk `FAILED` in `pipeline_control_log`, log the discrepancy, and raise an exception to halt the job.

5. **Serialize to Parquet via PyArrow** — Convert each chunk's result set to a PyArrow Table. Apply explicit schema casting to align MySQL types with BigQuery target types (e.g., `decimal(10,0)` → `int64`, `decimal(8,2)` → PyArrow decimal128). Write the Parquet file with snappy compression.

6. **Upload to GCS** — Write each chunk's Parquet file to `gs://<bucket>/employees/load_date=<YYYY-MM-DD>/chunk_<start>_<end>.parquet`. Use the `load_run_id` as a prefix to distinguish re-runs.

7. **Write summary to `pipeline_audit_log`** — On historical load job completion (all chunks `COMPLETE`), write a row to `pipeline_audit_log` with `load_type=HISTORICAL`, `records_extracted` = total rows across all chunks, `validation_status=PASSED`, and `schema_diff_detected=FALSE` (schema diff is set by the validation job in Phase 4).

## Phase 4: Cloud Run Validation Job

**Objective:** Implement pre-load data quality checks and load validated Parquet files to the BigQuery staging table via the Storage Write API.

1. **Implement pre-load schema validation** — Read all Parquet files in the GCS path for the current `load_run_id`. For each column in the Parquet schema:
   - Verify the column exists in the target `stg_employees` schema. Any extra column triggers the **hold-and-alert policy** (step 4 below).
   - Verify the column's Arrow data type is compatible with the target BigQuery type.
   - Verify null values only appear in columns designated `NULLABLE` in the target schema.

2. **Implement row count reconciliation** — Sum the total rows across all Parquet files for the `load_run_id`. Compare against the MySQL source count recorded in Phase 1 (for historical loads) or the extraction window count (for incremental runs). If the counts differ, update `pipeline_audit_log` with `validation_status=FAILED` and `error_details` containing the divergence, then raise an exception to halt the job.

3. **Implement PK uniqueness check on `employee_id`** — Load `employee_id` values from all Parquet files into memory (or use a temporary BigQuery table for large volumes). Execute a group-by count. If any `employee_id` appears more than once, update `pipeline_audit_log` with `validation_status=FAILED` and halt the job.

4. **Implement new-column detection with hold-and-alert** — Compare the Parquet schema against the target schema. If any column is present in the Parquet but absent from the target schema definition, immediately:
   - Update `pipeline_audit_log` with `schema_diff_detected=TRUE` and `validation_status=FAILED`.
   - Emit a structured log entry to Cloud Logging classifying the event as `SCHEMA_DRIFT_DETECTED`.
   - Trigger a Cloud Monitoring custom metric increment for `pipeline/schema_drift_events`.
   - Halt the job without loading any data to BigQuery. Do **not** auto-evolve the schema.

5. **Load to BigQuery via Storage Write API** — If all validation checks pass, use the BigQuery Storage Write API (default stream or pending stream for atomicity) to load validated Parquet data into `stg_employees`. Do not use `bq load` or the legacy streaming API.

6. **Update `pipeline_audit_log` on validation pass** — After a successful BigQuery load, update the audit log row with `validation_status=PASSED` and the final `records_extracted` count.

## Phase 5: BigQuery Stored Procedure — sp_mysqltobq_load.sql

**Objective:** Implement the MERGE stored procedure that upserts staged records into the production `stg_employees` table using the composite merge key `employee_id` AND `updated_date`.

1. **Author the MERGE statement** — Implement `sp_mysqltobq_load` as a BigQuery stored procedure containing a single MERGE statement with the following logic:

   ```sql
   MERGE `agentichub.stg_employees` AS T
   USING `agentichub.stg_employees_staging` AS S  -- or the staging load table
   ON T.employee_id = S.employee_id
     AND T.updated_date = S.updated_date
   WHEN MATCHED THEN
     UPDATE SET
       T.first_name      = S.first_name,
       T.last_name       = S.last_name,
       T.email           = S.email,
       T.phone_number    = S.phone_number,
       T.hire_date       = S.hire_date,
       T.job_id          = S.job_id,
       T.salary          = S.salary,
       T.commission_pct  = S.commission_pct,
       T.manager_id      = S.manager_id,
       T.department_id   = S.department_id,
       T.status          = S.status,
       T.created_date    = S.created_date
   WHEN NOT MATCHED BY TARGET THEN
     INSERT ROW;
   ```

   Accept `load_run_id STRING` as an input parameter to scope the MERGE to the current run's staging data if a run-scoped staging table pattern is used.

2. **Verify partition pruning** — After authoring the MERGE, run `EXPLAIN` or inspect the query execution plan in the BigQuery console. Confirm that the `updated_date` predicate in the merge key causes BigQuery to prune partitions rather than scan the full table. Document the partition scan count for baseline comparison.

3. **Test stored procedure manually** — Load a small dataset of 10–20 rows into the staging source. Execute `CALL agentichub.sp_mysqltobq_load(<load_run_id>)`. Verify:
   - New `employee_id` + `updated_date` combinations are inserted.
   - Existing `employee_id` + `updated_date` combinations are updated with changed column values.
   - `records_merged` output (rows affected) is returned to the caller for writing to `pipeline_audit_log`.

4. **Deploy stored procedure via Terraform** — Ensure the stored procedure DDL is managed as a Terraform `google_bigquery_routine` resource. Confirm `terraform plan` shows no diff after deployment.

## Phase 6: Incremental Load — Cloud Run Ingestion Job

**Objective:** Implement watermark-based delta extraction that retrieves only changed records since the last successful run, without performing full table re-scans.

1. **Implement watermark retrieval** — At the start of each incremental run, query `pipeline_control_log` (or `pipeline_audit_log`) for the maximum `updated_date` and `employee_id` from the last successfully completed incremental run. If no successful incremental run exists, the ExternalTaskSensor in Phase 7 will have blocked this job from running, so a missing watermark is an error condition.

2. **Implement watermark-filtered extraction** — Extract delta records using a composite watermark filter to avoid missing records at timestamp boundaries:
   ```sql
   SELECT * FROM agentichub.employees
   WHERE updated_date > :last_watermark_ts
      OR (updated_date = :last_watermark_ts AND EMPLOYEE_ID > :last_watermark_pk)
   ORDER BY updated_date ASC, EMPLOYEE_ID ASC;
   ```
   This prevents full table re-scans. Confirm with the source team that `updated_date` is enforced on all write paths (inserts, updates, batch jobs) before go-live.

3. **Reuse Parquet serialisation and GCS upload** — Apply the same PyArrow serialisation and GCS upload pattern as the historical load job (`gs://<bucket>/employees/load_date=<YYYY-MM-DD>/incremental_<run_id>.parquet`). Write chunk state to `pipeline_control_log` using `load_type=INCREMENTAL`.

4. **Reuse validation job** — Trigger the Phase 4 validation job after the incremental extraction completes, passing the current `load_run_id`. All four validation checks (schema, row count reconciliation, PK uniqueness, new-column detection) apply equally to incremental loads.

5. **Update watermark after successful MERGE** — After `sp_mysqltobq_load` completes successfully, write the new high-water mark (`MAX(updated_date)`, `MAX(employee_id)` from the current run's data) to `pipeline_control_log` for use by the next incremental run.

6. **Write to `pipeline_audit_log`** — Record `load_type=INCREMENTAL` and all required audit fields on completion.

## Phase 7: Cloud Composer Orchestration

**Objective:** Implement two Airflow DAGs — one for the historical load and one for incremental loads — with the incremental DAG gated on historical load SUCCESS via ExternalTaskSensor.

1. **Author the historical load DAG** — Create `dag_employees_historical_load.py` in the Composer DAGs folder. The DAG must be triggered manually (no schedule). Task sequence:
   1. `trigger_historical_ingestion` — Trigger the Phase 3 Cloud Run Job via `CloudRunJobOperator` (or `GKEStartJobOperator` if applicable). Wait for job completion.
   2. `trigger_validation_job` — Trigger the Phase 4 Cloud Run validation and BQ load job. Wait for completion.
   3. `execute_merge_procedure` — Call `sp_mysqltobq_load` via `BigQueryInsertJobOperator` or `BigQueryOperator`.
   4. `write_audit_log` — Write final execution summary to `pipeline_audit_log` using `BigQueryInsertJobOperator`.
   - On any task failure, the DAG must fail immediately and propagate `FAILED` status — this is the status that `ExternalTaskSensor` in the incremental DAG will observe.

2. **Author the incremental load DAG** — Create `dag_employees_incremental_load.py`. Apply the schedule per the confirmed SLA (hourly or daily — confirm when Blocker 1 is resolved). Task sequence:
   1. `gate_on_historical_load` — `ExternalTaskSensor` pointing to `dag_employees_historical_load` and its final task (`write_audit_log`). Set `mode='reschedule'` and a configurable `timeout`. The incremental DAG must not proceed past this task until the historical DAG task reaches `SUCCESS` state.
   2. `trigger_incremental_ingestion` — Trigger the Phase 6 Cloud Run ingestion job.
   3. `trigger_validation_job` — Trigger the Phase 4 validation and BQ load job.
   4. `execute_merge_procedure` — Call `sp_mysqltobq_load` via BigQuery operator.
   5. `write_audit_log` — Write final execution summary to `pipeline_audit_log`.

3. **Configure DAG-level defaults** — Set `retries=1`, `retry_delay=timedelta(minutes=5)`, `email_on_failure=True` (routing to pipeline owner), and `catchup=False` on both DAGs.

4. **Validate DAG import errors** — After deploying both DAGs to the Composer environment, confirm zero import errors via Airflow UI → DAGs → import errors pane. Zero errors is the pass condition.

5. **Perform end-to-end dry run** — In the non-production Composer environment, manually trigger `dag_employees_historical_load`. Confirm all tasks reach `SUCCESS`. Then trigger `dag_employees_incremental_load` and confirm `gate_on_historical_load` sensor resolves immediately. Proceed through all incremental tasks.

6. **Confirm ExternalTaskSensor blocking behaviour** — Reset the historical load DAG to `FAILED` state. Trigger the incremental DAG. Confirm `gate_on_historical_load` sensor does not clear and the incremental DAG remains in `running` state (sensor polling), without executing downstream tasks.

## Phase 8: Monitoring, Observability & Reconciliation

**Objective:** Implement alerting, logging, the Looker Studio stakeholder dashboard, and the weekly reconciliation job to ensure ongoing operational visibility.

1. **Configure Cloud Monitoring alert policies** — Create the following alert policies in Cloud Monitoring:
   - **Job Failure Alert**: Trigger when a Cloud Run Job execution completes with a non-zero exit code for any job in the employee pipeline. Route to pipeline owner.
   - **Schema Drift Alert**: Trigger on custom metric `pipeline/schema_drift_events` count > 0 within a 5-minute window. Route to pipeline owner and data engineering lead.
   - **Row Count Discrepancy Alert**: Trigger when the weekly reconciliation job logs a discrepancy exceeding 0.1% of total row count. Route to pipeline owner.

2. **Verify Cloud Logging audit trail** — Confirm that each Cloud Run Job execution emits structured JSON logs containing: `load_run_id`, `load_type`, `chunk_id` (where applicable), `records_processed`, `validation_status`, and any `error_details`. Verify logs are queryable in Cloud Logging using the pipeline's resource labels.

3. **Implement weekly reconciliation Cloud Run Job** — Create a standalone Cloud Run Job that:
   - Queries `SELECT COUNT(*) FROM agentichub.employees` in MySQL via Cloud SQL Auth Proxy.
   - Queries `SELECT COUNT(*) FROM agentichub.stg_employees` in BigQuery.
   - Computes the absolute and percentage difference.
   - If the percentage difference exceeds 0.1%, emits a structured log event and increments the `pipeline/row_count_discrepancy` custom metric.
   - Executes a random sample checksum of 1,000 `employee_id` values, comparing `salary` and `updated_date` between MySQL and BigQuery for those IDs.
   - Writes results to `pipeline_audit_log` with `load_type=RECONCILIATION`.
   - Schedule this job via Cloud Composer (weekly trigger).

4. **Build Looker Studio dashboard** — Connect Looker Studio directly to the `pipeline_audit_log` BigQuery table. The dashboard must display — without engineering involvement — the following per run:
   - Load status (PASSED / FAILED)
   - Records extracted
   - Records merged
   - Last refresh timestamp
   - Validation pass/fail indicator
   - Schema drift detected flag
   - Filter by `load_type` (HISTORICAL / INCREMENTAL / RECONCILIATION)

5. **Validate alert routing** — Trigger a test alert by publishing a synthetic `pipeline/schema_drift_events` metric data point via the Cloud Monitoring API. Confirm the alert fires and the notification reaches the configured recipients within 5 minutes.

- **What is tested:** That re-running a failed historical load does not produce duplicate rows.

- **What is tested:** That an unrecognised column in the source triggers the hold-and-alert policy.

### TC-08: MERGE — New Record Insertion

### TC-10: MERGE — Composite Merge Key (New updated_date)
