## Phase 2: BigQuery Target Schema Provisioning

**Goal:** Create all BigQuery tables required by the pipeline in the `verizon_data_deah` dataset.

1. **Create the permanent staging table `stg_employees`** with DAY-granularity partitioning on `updated_date` and clustering on `[department_id, status, job_id]`. Apply the following DDL:

   ```sql
   CREATE TABLE IF NOT EXISTS `verizon_data_deah.stg_employees`
   (
     employee_id   INT64     NOT NULL,
     first_name    STRING    NOT NULL,
     last_name     STRING    NOT NULL,
     email         STRING    NOT NULL,
     phone_number  STRING,
     hire_date     DATE      NOT NULL,
     job_id        STRING    NOT NULL,
     salary        NUMERIC   NOT NULL,
     commission_pct NUMERIC,
     manager_id    INT64,
     department_id INT64,
     status        STRING    NOT NULL,
     created_date  TIMESTAMP NOT NULL,
     updated_date  TIMESTAMP
   )
   PARTITION BY DATE(updated_date)
   CLUSTER BY department_id, status, job_id
   OPTIONS (require_partition_filter = FALSE);
   ```

2. **Create the transient load buffer `employees_staging`** with the same column definitions as `stg_employees` but without partitioning or clustering, as it is truncated before each load cycle:

   ```sql
   CREATE TABLE IF NOT EXISTS `verizon_data_deah.employees_staging`
   (
     employee_id    INT64,

     hire_date      DATE,
     job_id         STRING,
     salary         NUMERIC,
     commission_pct NUMERIC,
     manager_id     INT64,
     department_id  INT64,
     status         STRING,
     created_date   TIMESTAMP,
     updated_date   TIMESTAMP
   );
   ```

3. **Create the `etl_watermark` control table** to persist the per-table incremental state:

   ```sql
   CREATE TABLE IF NOT EXISTS `verizon_data_deah.etl_watermark`
   (
     table_name     STRING    NOT NULL,
     last_watermark TIMESTAMP,
     last_max_pk    INT64,
     updated_at     TIMESTAMP NOT NULL
   );

   -- Seed the initial watermark row for employees
   INSERT INTO `verizon_data_deah.etl_watermark`
     (table_name, last_watermark, last_max_pk, updated_at)
   ```

4. **Create the `etl_audit_log` table** to record every pipeline run:

   ```sql
     run_id              STRING    NOT NULL,
   ```

## Phase 4: Historical Full Load

**Goal:** Execute a one-time, complete export of `agentichub.employees` into `verizon_data_deah.stg_employees` with 100% data completeness and no data loss.

> **Prerequisite:** Open Blockers 1 and 2 must be resolved before executing this phase. Confirm `etl_control.full_load_complete = FALSE` for `employees` before starting.

1. **Truncate `employees_staging`** to ensure a clean load buffer:

   ```sql
   TRUNCATE TABLE `verizon_data_deah.employees_staging`;
   ```

2. **Trigger Cloud SQL Admin API export** (full) via the Airflow `CloudSQLExportInstanceOperator` (or equivalent API call), targeting the `cloud_sql_export_pool` Airflow Pool with `pool_size=1`. Export query:

   ```sql
   SELECT
     employee_id, first_name, last_name, email, phone_number,
     hire_date, job_id, salary, commission_pct, manager_id,
     department_id, status, created_date, updated_date
   FROM agentichub.employees;
   ```

   Output destination: `gs://[project]-etl-staging/employees/full/employees_full_{run_date}.csv` using the agreed CSV format contract (Phase 3, Step 2).

3. **Execute pre-load schema compatibility check.** Before loading the CSV, compare `agentichub.employees` columns from Cloud SQL `INFORMATION_SCHEMA.COLUMNS` against `verizon_data_deah.employees_staging` from BigQuery `INFORMATION_SCHEMA.COLUMNS`. Check: column names, data type compatibility, and nullable flags. On any mismatch: fail the DAG task, write a structured discrepancy record to `etl_audit_log` (status = `failed`, validation_results = discrepancy detail), and halt â€” do not proceed to the load job.

4. **Execute BigQuery Native Load Job** from the exported GCS file into `employees_staging`. Configuration: source format CSV, skip leading rows = 0 (no header) or 1 (if header included per contract), write disposition `WRITE_TRUNCATE`, schema autodetect off (use explicit schema matching `employees_staging` DDL).

5. **Execute post-load row count validation.** Query the source row count scoped to the full export window:

   ```sql
   -- Source count (via Cloud SQL connector or previously captured export metadata)
   SELECT COUNT(*) FROM agentichub.employees;

   -- Target count
   SELECT COUNT(*) FROM `verizon_data_deah.employees_staging`;
   ```

   If counts do not match, fail the DAG task and write to `etl_audit_log` (status = `failed`, source_row_count, target_row_count).

6. **Execute null constraint violation scan** on all `NOT NULL` columns in `employees_staging`: `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date`. Fail the task and write to `etl_audit_log` if any nulls are found.

7. **Execute MERGE from `employees_staging` into `stg_employees`** to enforce PK uniqueness and apply upsert semantics:

   ```sql
   MERGE `verizon_data_deah.stg_employees` T
   USING `verizon_data_deah.employees_staging` S
   ON T.employee_id = S.employee_id
   WHEN MATCHED THEN UPDATE SET
     first_name     = S.first_name,
     last_name      = S.last_name,
     email          = S.email,
     phone_number   = S.phone_number,
     hire_date      = S.hire_date,
     job_id         = S.job_id,
     salary         = S.salary,
     commission_pct = S.commission_pct,
     manager_id     = S.manager_id,
     department_id  = S.department_id,
     status         = S.status,
     created_date   = S.created_date,
     updated_date   = S.updated_date
   WHEN NOT MATCHED THEN INSERT VALUES (
     S.employee_id, S.first_name, S.last_name, S.email,
     S.phone_number, S.hire_date, S.job_id, S.salary,
     S.commission_pct, S.manager_id, S.department_id,
     S.status, S.created_date, S.updated_date
   );
   ```

8. **Advance the watermark** after a successful MERGE:

   ```sql
   UPDATE `verizon_data_deah.etl_watermark`
   SET
     last_watermark = (SELECT MAX(updated_date) FROM `verizon_data_deah.stg_employees`),
     last_max_pk    = (SELECT MAX(employee_id)  FROM `verizon_data_deah.stg_employees`),
     updated_at     = CURRENT_TIMESTAMP()
   WHERE table_name = 'employees';
   ```

9. **Write audit log record** for the completed full load run:

   ```sql
   INSERT INTO `verizon_data_deah.etl_audit_log`
     (run_id, table_name, load_type, status, source_row_count,
      target_row_count, validation_results, start_time, end_time)
   VALUES
     ({run_id}, 'employees', 'full', 'success',
      {source_count}, {target_count}, {validation_json},
      {start_time}, CURRENT_TIMESTAMP());
   ```

10. **Set the historical load sign-off gate** upon successful completion and manual review:

    ```sql
    UPDATE `verizon_data_deah.etl_control`
    SET
      full_load_complete = TRUE,
      signed_off_by      = '{approver_name}',
      signed_off_at      = CURRENT_TIMESTAMP()
    WHERE table_name = 'employees';
    ```

## Phase 5: Incremental Load & Watermark Logic

**Goal:** Implement the scheduled incremental load DAG that processes only new and changed rows using the watermark strategy confirmed in Phase 1.

> **Prerequisite:** `etl_control.full_load_complete = TRUE` for `employees`. Do not enable the incremental schedule until the historical load is signed off.

1. **Read the current watermark** at the start of each DAG run:

   ```sql
   SELECT last_watermark, last_max_pk
   FROM `verizon_data_deah.etl_watermark`
   WHERE table_name = 'employees';
   ```

   Store `last_watermark` and `last_max_pk` as Airflow XCom values for use in downstream tasks.

2. **Truncate `employees_staging`** to clear the transient buffer before each incremental load.

3. **Trigger Cloud SQL Admin API export (incremental)** via the `cloud_sql_export_pool`. Apply the delta filter based on the confirmed watermark strategy from Open Blocker 2:

   - **Primary strategy** (if `updated_date` is reliably populated):

     ```sql
     SELECT employee_id, first_name, last_name, email, phone_number,
            hire_date, job_id, salary, commission_pct, manager_id,
            department_id, status, created_date, updated_date
     FROM agentichub.employees
     WHERE updated_date > '{last_watermark}'
        OR (updated_date IS NULL AND created_date > '{last_watermark}');
     ```

   - **Fallback strategy** (if PK-delta is required):

     ```sql
     SELECT employee_id, first_name, last_name, email, phone_number,
            hire_date, job_id, salary, commission_pct, manager_id,
            department_id, status, created_date, updated_date
     FROM agentichub.employees
     WHERE employee_id > {last_max_pk};
     ```

   Output destination: `gs://[project]-etl-staging/employees/incremental/employees_incremental_{run_timestamp}.csv`.

4. **Execute pre-load schema compatibility check** (same logic as Phase 4, Step 3). Fail the DAG and alert on any mismatch before loading.

5. **Execute BigQuery Native Load Job** from the incremental GCS file into `employees_staging` with `WRITE_TRUNCATE`.

6. **Execute post-load row count validation.** Query the source row count scoped to the same delta window used in the export query. Compare to `COUNT(*) FROM employees_staging`. Fail and alert on mismatch.

7. **Execute null constraint violation scan** on all `NOT NULL` columns in `employees_staging` (same columns as Phase 4, Step 6).

8. **Execute MERGE from `employees_staging` into `stg_employees`** using the identical MERGE statement defined in Phase 4, Step 7.

9. **Advance the watermark** using the identical UPDATE statement defined in Phase 4, Step 8.

10. **Write audit log record** to `etl_audit_log` with `load_type = 'incremental'` and all run metrics.
