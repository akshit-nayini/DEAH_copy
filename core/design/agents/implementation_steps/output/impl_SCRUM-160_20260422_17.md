## ⚠️ Open Blockers

The following blockers are unresolved and must be signed off before the indicated phases can be finalised. Development may proceed in parallel up to the gated point, but no phase depending on a blocker may be closed without resolution.

1. **Latency SLA undefined** — Requirements permit hourly or daily cadence but no maximum acceptable delay from MySQL commit to BigQuery availability has been specified. If a sub-hour freshness SLA is introduced, the batch Dataflow pattern must be re-evaluated against Datastream CDC before the Cloud Composer DAG schedule is locked. *Gates: Phase 5 (DAG schedule configuration).*

2. **Table inventory incomplete** — Only `employees` is confirmed as an initial ingestion target. The full list of `agentichub` tables must be agreed with stakeholder Yash and documented in a load inventory matrix before historical load begins. Each additional table requires a Composer DAG branch, stored procedure extension, `watermark_control` entry, and schema definition registration. *Gates: Phase 5 and all subsequent phases for any table beyond `employees`.*

3. **Per-table load cadence not finalised** — A load frequency matrix mapping each table to its required cadence (hourly vs. daily) must be signed off before Composer DAG schedules and trigger intervals are configured. *Gates: Phase 5 (DAG schedule configuration).*

## ⚠️ Risks & Mitigations

*Reference material only — not embedded as implementation steps.*

| Risk | Mitigation Summary |
|---|---|
| **Data Completeness — Historical Load Gate** | The Composer DAG must gate incremental load activation on a confirmed `PASS` status in `audit_log`, comparing a pre-Dataflow MySQL source row count (stored in a Composer XCom variable) against the BigQuery post-load count. Dataflow job success status alone is not a sufficient gate condition. |
| **Schema Evolution — New MySQL Columns** | A schema comparison pre-flight task in the Composer DAG must run `INFORMATION_SCHEMA` queries against MySQL (via Cloud SQL Auth Proxy) and BigQuery before each Dataflow job submission. Mismatches write an `ERROR` record to `audit_log`, halt the pipeline, and trigger a Cloud Monitoring alert. Nullable additions may use automatic `ALTER TABLE`; type changes and `NOT NULL` additions require a manual engineering review gate per the approved runbook. |
| **Source System Load — Cloud SQL Saturation** | Schedule the historical full load during off-peak hours. Configure JdbcIO `numPartitions` conservatively (2–4) for the initial historical run and tune based on Cloud SQL CPU utilisation and connection count metrics. Evaluate promoting to a Cloud SQL read replica if source load becomes a recurring concern as additional tables are onboarded. |

4. **Execute schema comparison pre-flight.** Query `INFORMATION_SCHEMA.COLUMNS` against `agentichub.employees` in MySQL and compare the result against the registered 15-column target schema definition for `stg_employees`. Confirm column names, ordinal positions, and data types match expectations. Log the comparison result. Any mismatch must be resolved and documented before proceeding to Phase 2.

5. **Verify BigQuery dataset reachability.** Confirm that the Dataflow service account can write to project `verizon-data`, dataset `verizon_data_dea` by running a dry-run query against the dataset. Confirm that the Cloud Composer service account can submit BigQuery jobs to the same project and dataset.

## Phase 2: BigQuery Target Schema Deployment

1. **Create the `employees_staging` table.** Deploy the raw Dataflow output staging table in `verizon_data_dea` using the 15-column schema defined in the data model for `stg_employees`. Apply TIME partitioning on `updated_date` at DAY granularity with a 60-day partition expiry. Apply clustering on `department_id`, `status`, `job_id` (in that order). Set `employee_id` as the primary key constraint (non-enforced, BigQuery semantics). Confirm that the `__NULL__` partition is available for rows where `updated_date IS NULL` and that those rows are exempt from the 60-day expiry.

2. **Create the `employees` production table.** Deploy the post-MERGE production table in `verizon_data_dea` using the identical 15-column schema. Apply the same TIME partitioning, 60-day expiry, and clustering configuration as `employees_staging`. This table is the authoritative output of `sp_mysqltobq_load.sql`.

3. **Create the `audit_log` table.** Deploy the `audit_log` table in `verizon_data_dea` with columns sufficient to capture: `load_id`, `table_name`, `load_mode` (`HISTORICAL` / `INCREMENTAL`), `status` (`PASS` / `ERROR`), `source_row_count`, `target_row_count`, `load_start_ts`, `load_end_ts`, `watermark_start`, `watermark_end`, `error_detail`. This table is written exclusively by `sp_mysqltobq_load.sql`.

4. **Create the `watermark_control` table.** Deploy the `watermark_control` table in `verizon_data_dea` with columns: `table_name` (STRING, PK), `last_loaded_watermark` (TIMESTAMP), `updated_ts` (TIMESTAMP). Insert the initial row for `employees` with `last_loaded_watermark = NULL` to represent the pre-load state.

5. **Register the schema definition artefact.** Store the 15-column schema definition (column names, data types, nullable flags) as a versioned JSON artefact in the project artefact store. This is the reference schema used by the Composer pre-flight schema comparison task in all subsequent load executions.

## Phase 3: Dataflow Flex Template Development

1. **Implement the Apache Beam JDBC ingestion pipeline.** Develop the Beam pipeline in the project-standard language using `JdbcIO` with the MySQL JDBC driver. The pipeline must accept the following runtime parameters: `source_table`, `target_bq_table`, `load_mode` (`FULL` / `INCREMENTAL`), `watermark_value` (used only in `INCREMENTAL` mode), `partition_column` (`employee_id`), `num_partitions`, `lower_bound`, `upper_bound`.

2. **Configure the historical (FULL) load branch.** For `load_mode=FULL`, the pipeline executes a full table scan of `agentichub.employees` with no filter predicate. JdbcIO partitioning parameters (`partitionColumn=employee_id`, `numPartitions=2–4`, `lowerBound`, `upperBound`) must be set to enable parallel reads without overwhelming the source. Write all rows in parallel to `verizon_data_dea.employees_staging`.

3. **Configure the incremental load branch.** For `load_mode=INCREMENTAL`, the pipeline appends a `WHERE updated_date > :last_watermark` predicate to the JDBC read query, reading only changed and new records since the last successful run. The `last_watermark` value is passed as a runtime parameter sourced from `watermark_control.last_loaded_watermark` by the Composer DAG.

4. **Containerise the pipeline as a Flex Template.** Package the pipeline into a container image and publish the Flex Template specification to the designated GCS path in `verizon-data`. The template must be parameterisable for all runtime parameters listed in step 1, supporting reuse across both load modes and future tables.

5. **Version-control the template artefact.** Commit the Beam pipeline source, Dockerfile, and Flex Template JSON specification to the project repository under a versioned path. Tag the initial release as `v1.0.0`.

## Phase 4: Stored Procedure Development (`sp_mysqltobq_load.sql`)

1. **Implement the MERGE statement.** Author the `sp_mysqltobq_load.sql` stored procedure in BigQuery. The MERGE must use `employee_id` AND `updated_date` as the composite merge key, matching rows in `employees_staging` against `employees`. On match, update all non-key columns. On no match (new record), insert. On source delete (if applicable), leave in place per the batch pattern.

2. **Implement pre-load row count reconciliation.** At the start of the stored procedure execution, read the pre-load MySQL source row count from the Composer XCom variable (passed in as a stored procedure parameter) and store it as `source_row_count` for the audit record. After the MERGE, execute `SELECT COUNT(*) FROM employees` and store as `target_row_count`.

3. **Implement schema consistency validation.** Within the stored procedure, query `INFORMATION_SCHEMA.COLUMNS` for the `employees` table in `verizon_data_dea` and compare against the registered schema definition artefact (passed in or referenced via a BigQuery-accessible metadata table). On mismatch, write an `ERROR` record to `audit_log` and raise an exception to halt the procedure.

4. **Implement primary key uniqueness validation.** After the MERGE, execute a `GROUP BY employee_id HAVING COUNT(*) > 1` check on `employees`. On any result, write an `ERROR` record to `audit_log` and raise an exception.

5. **Implement null and data type constraint validation.** Verify that all `NOT NULL` columns (`employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date`) contain no null values in the post-MERGE `employees` table. On violation, write an `ERROR` record to `audit_log` and raise an exception.

6. **Enforce 60-day partition expiry.** Confirm the partition expiry policy on `employees` is set to 60 days on `updated_date`. Document in the runbook that rows where `updated_date IS NULL` land in the `__NULL__` partition and are explicitly exempt from partition expiry.

7. **Advance the watermark.** On `PASS` completion, update `watermark_control.last_loaded_watermark` for `table_name = 'employees'` to `MAX(updated_date)` of the current load batch.

8. **Write the audit log record.** On completion (PASS or ERROR), write a complete record to `audit_log` including `load_id`, `table_name`, `load_mode`, `status`, `source_row_count`, `target_row_count`, `load_start_ts`, `load_end_ts`, `watermark_start`, `watermark_end`, and `error_detail` (null on PASS). The PASS status written here is the gate condition read by the Composer DAG before incremental load activation.

9. **Version-control and test the stored procedure.** Commit `sp_mysqltobq_load.sql` to the project repository. Execute the procedure against `employees_staging` with known test data before wiring it into the DAG.

## Phase 5: Cloud Composer DAG Development

1. **Implement the schema comparison pre-flight task.** Author a PythonOperator or custom operator that, before each Dataflow job submission, queries `INFORMATION_SCHEMA.COLUMNS` against MySQL (via Cloud SQL Auth Proxy) and BigQuery, compares column count and names against the registered schema definition artefact, and writes an `ERROR` record to `audit_log` and triggers a Cloud Monitoring alert on mismatch. The task must halt the DAG on any schema discrepancy.

2. **Capture and store the pre-load MySQL row count.** Implement a PythonOperator that executes `SELECT COUNT(*) FROM employees` against the MySQL source and pushes the result to a Composer XCom variable (`mysql_pre_load_row_count`). This task runs after the schema comparison pre-flight and before the Dataflow job submission.

3. **Implement the historical load DAG branch.** Wire the following task sequence for `load_mode=FULL`:
   - Schema comparison pre-flight
   - MySQL row count capture (XCom push)
   - Dataflow Flex Template submission (`load_mode=FULL`, `num_partitions=2–4`, `lower_bound` and `upper_bound` derived from source `employee_id` range)
   - Dataflow job status sensor (poll until `JOB_STATE_DONE`)
   - `BigQueryInsertJobOperator` calling `sp_mysqltobq_load.sql` with `load_mode=HISTORICAL` and the XCom row count

4. **Implement the incremental load DAG branch with historical gate.** Wire the following task sequence for `load_mode=INCREMENTAL`:
   - `BigQueryCheckOperator` asserting that a `PASS` record exists in `audit_log` for `table_name='employees'` and `load_mode='HISTORICAL'` — this is the hard gate condition; the DAG must not proceed if this check fails
   - Schema comparison pre-flight
   - Read `last_loaded_watermark` from `watermark_control` for `employees` (XCom push)
   - MySQL row count capture for the incremental window (`WHERE updated_date > last_watermark`) (XCom push)
   - Dataflow Flex Template submission (`load_mode=INCREMENTAL`, `watermark_value` from XCom)
   - Dataflow job status sensor
   - `BigQueryInsertJobOperator` calling `sp_mysqltobq_load.sql` with `load_mode=INCREMENTAL` and the XCom row count

5. **Configure the DAG schedule.** Set the DAG schedule interval once the per-table load cadence blocker (Open Blocker 3) is resolved and the load frequency matrix is signed off. Until resolved, deploy the DAG with `schedule_interval=None` (manual trigger only) for historical load execution.

6. **Configure DAG-level retry and alerting.** Set task-level retries (minimum 2) with exponential backoff. Configure an `on_failure_callback` on all tasks to emit a structured error event to Cloud Logging and trigger the Cloud Monitoring alert policy.

## Phase 6: Historical Load Execution

1. **Schedule the historical load during off-peak hours.** Confirm a maintenance window with stakeholder Yash. Trigger the historical load DAG branch manually with `load_mode=FULL` during the agreed off-peak window to minimise Cloud SQL saturation.

2. **Monitor Dataflow job progress.** During the historical load, monitor Cloud Monitoring for Cloud SQL CPU utilisation and active connection count. If either metric approaches saturation thresholds, pause the Dataflow job and reduce `numPartitions` before resuming.

3. **Validate `audit_log` PASS record.** After the stored procedure completes, query `audit_log` for `table_name='employees'`, `load_mode='HISTORICAL'`, `status='PASS'`. Confirm `source_row_count` equals the pre-load MySQL count captured in XCom. Confirm `target_row_count` matches `source_row_count`. This record is the gate condition for incremental load activation.

4. **Validate `watermark_control` advancement.** Query `watermark_control` and confirm `last_loaded_watermark` for `employees` has been updated from `NULL` to `MAX(updated_date)` of the loaded data.

5. **Sign off on historical load completion.** Record the historical load results (row counts, timestamps, watermark value, audit log ID) in the project artefact store and obtain sign-off from stakeholder Yash before enabling the incremental load schedule.

## Phase 7: Incremental Load Activation

1. **Verify the historical gate condition.** Confirm that the `BigQueryCheckOperator` in the incremental DAG branch correctly reads the `PASS` record from `audit_log` for the completed historical load. Execute a dry run of the gate check task to confirm it evaluates to true.

2. **Execute the first incremental load manually.** Trigger the incremental DAG branch manually for the first run. Confirm that the watermark value passed to Dataflow equals the `last_loaded_watermark` set during the historical load.

3. **Validate incremental MERGE correctness.** Confirm that only records with `updated_date > last_watermark` were read from MySQL and merged into `employees`. Verify the `audit_log` record shows `load_mode='INCREMENTAL'` with correct `source_row_count` and `target_row_count`. Verify the watermark has advanced.

4. **Activate the scheduled DAG.** Once the per-table cadence blocker is resolved and the load frequency matrix is signed off, update the DAG `schedule_interval` to the agreed value and enable the scheduler.

## Phase 8: Monitoring & Observability Configuration

1. **Configure Cloud Monitoring alert policies.** Create alert policies for: Dataflow job state transition to `JOB_STATE_FAILED`; Cloud Composer task failure events; custom log-based metrics derived from `ERROR`-severity entries written by the stored procedure to Cloud Logging. Configure notification channels (email and/or PagerDuty) as agreed with stakeholder Yash.

2. **Configure the schema mismatch alert.** Create a dedicated log-based metric and alert policy that fires when the schema comparison pre-flight task writes an `ERROR` record to `audit_log`. This alert must route to the engineering escalation channel, not only to stakeholder Yash.

3. **Deploy the Looker Studio operational dashboard.** Connect the Looker Studio report to `verizon_data_dea.audit_log` and reconciliation result tables. The dashboard must surface: load execution history, `source_row_count` vs. `target_row_count` per run, data quality check outcomes, pipeline status, and watermark progression. Confirm Yash can access the report without engineering involvement.
