## ⚠️ Risks & Mitigations

> Reference material only — not embedded as implementation steps.

| Risk | Mitigation |
|---|---|
| **Data Volume Unknown** | Execute `SELECT COUNT(*), AVG(LENGTH(CONCAT_WS(',', <columns>))) FROM employees` against MySQL before finalising extraction design. If uncompressed size exceeds 20 GB, implement PK-range partitioned extraction with configurable batch size and parallel Cloud Run Job task instances. Schedule formal volume review at 6 months post-production. |
| **Schema Drift** | Pre-load schema compatibility check (MySQL `information_schema.COLUMNS` vs BigQuery table schema) is a mandatory, non-skippable step. New nullable columns: auto-add to BigQuery as `STRING`, raise alert. Type mismatch or column removal: block load and page on-call immediately. Document and obtain sign-off on this policy before first production run. |
| **Incremental Delta Strategy Dependency** | Confirmed by Blocker #2. If no reliable timestamp exists, evaluate in order: (1) add `updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` to MySQL schema; (2) enable binlog capture via Datastream; (3) full-table hash checksum comparison (highest cost). Resolution must be locked in sprint 0. |

## Phase 1: Connectivity & Environment Validation

1. **Connectivity check — MySQL source.** From within a Cloud Run Job container using the Cloud SQL Auth Proxy sidecar, open a TCP connection to `verizon-data:us-central1:mysql-druid-metadatastore` (host `34.70.79.163`, port `3306`, database `agentichub`, user `sa`). Verify connection is established successfully and the session can execute a `SELECT 1`.

2. **Source table existence check.** Execute `SELECT TABLE_NAME FROM information_schema.TABLES WHERE TABLE_SCHEMA = 'agentichub' AND TABLE_NAME = 'employees'` and confirm the table is returned.

3. **Source schema snapshot.** Execute `SELECT COLUMN_NAME, DATA_TYPE, IS_NULLABLE, CHARACTER_MAXIMUM_LENGTH, NUMERIC_PRECISION, NUMERIC_SCALE FROM information_schema.COLUMNS WHERE TABLE_SCHEMA = 'agentichub' AND TABLE_NAME = 'employees' ORDER BY ORDINAL_POSITION` and capture the full result set. Persist this snapshot as a JSON artifact in GCS for use during pre-load schema compatibility checks.

4. **Volume sizing query.** Execute `SELECT COUNT(*) AS row_count, ROUND(AVG(LENGTH(CONCAT_WS(',', EMPLOYEE_ID, FIRST_NAME, LAST_NAME, EMAIL, PHONE_NUMBER, HIRE_DATE, JOB_ID, SALARY, COMMISSION_PCT, MANAGER_ID, DEPARTMENT_ID, STATUS, CREATED_DATE, UPDATED_DATE))), 2) AS avg_row_bytes FROM agentichub.employees`. Record results and compare against the 20 GB threshold defined in the architecture risk register. **This result resolves Blocker #1 — update SCRUM-123 before proceeding to Phase 2.**

5. **Delta column confirmation.** Execute `SELECT COLUMN_NAME, IS_NULLABLE, EXTRA FROM information_schema.COLUMNS WHERE TABLE_SCHEMA = 'agentichub' AND TABLE_NAME = 'employees' AND COLUMN_NAME IN ('updated_date', 'updated_at', 'modified_at')`. Verify `updated_date` is present, non-null on writes, and populated by the application. **This result resolves Blocker #2 — update SCRUM-123 before proceeding to Phase 4.**

6. **BigQuery dataset and GCS bucket accessibility check.** Confirm the Cloud Run Job service account has `bigquery.dataEditor` on dataset `verizon_data_deah` and `storage.objectAdmin` on the GCS staging bucket. Execute a `bq ls verizon_data_deah` and a `gsutil ls gs://<staging-bucket>` to validate permissions.

## Phase 2: Extraction Layer — Full Historical Load

> **Prerequisite:** Blocker #1 resolved and volume confirmed ≤ 20 GB for single-container design. If volume exceeds 20 GB, implement PK-range partitioned extraction before proceeding.

1. **Implement the extraction Cloud Run Job (Python 3.12).** Create a containerised Python job using `mysql-connector-python` and Cloud SQL Auth Proxy sidecar. Accept a `--mode` argument accepting values `full` and `incremental`. For `full` mode, execute `SELECT EMPLOYEE_ID, FIRST_NAME, LAST_NAME, EMAIL, PHONE_NUMBER, HIRE_DATE, JOB_ID, SALARY, COMMISSION_PCT, MANAGER_ID, DEPARTMENT_ID, STATUS, CREATED_DATE, UPDATED_DATE FROM agentichub.employees ORDER BY EMPLOYEE_ID` with configurable fetch batch chunk size (default 50,000 rows).

2. **Implement Parquet serialisation and GCS staging write.** Convert each fetched batch to a Parquet file using `pyarrow`. Apply the following type mappings: `INT` → `INT64`, `VARCHAR` → `STRING` (UTF-8), `DATE` → `DATE32`, `DECIMAL(8,2)` / `DECIMAL(4,2)` → `DECIMAL128` with matching precision/scale, `TIMESTAMP` → `TIMESTAMP[us, UTC]`. Write each Parquet file to GCS path `gs://<staging-bucket>/employees/full/<run_id>/part-<batch_seq>.parquet`.

3. **Implement pre-load schema compatibility check.** Before executing the MySQL `SELECT`, compare the live `information_schema.COLUMNS` result against the persisted schema snapshot from Phase 1 Step 3. Apply the following rules: new nullable column → log warning, auto-provision column in BigQuery as `STRING`, continue load; type mismatch → log error, abort load, write `SCHEMA_CHECK_FAILED` to `audit_log`; column removal or non-nullable column added → log critical, abort load, page on-call.

4. **Implement null constraint violation scan.** After extraction and before GCS write, scan the in-memory batch for `NULL` values in columns declared `nullable: false` (`employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date`). Log each violation with row identifier and column name. If any violations are found, write a structured summary to `audit_log.null_violations` and abort the load.

5. **Capture source row count.** After the full extraction loop completes, record `rows_extracted` from the MySQL `COUNT(*)` result obtained in Phase 1 Step 4. This value is the reconciliation baseline for post-load validation.

6. **Build and push the container image.** Build the Docker image for the extraction job targeting Python 3.12, tag as `gcr.io/<project>/mysql-bq-ingestion:full-<git-sha>`, and push to Artifact Registry.

## Phase 3: BigQuery Target Setup & Full Load Processing

1. **Create BigQuery staging table `stg_employees`.** Define the table in dataset `verizon_data_deah` with the following schema, partitioning, and clustering:
   - **Columns:** `employee_id INT64 NOT NULL`, `first_name STRING NOT NULL`, `last_name STRING NOT NULL`, `email STRING NOT NULL`, `phone_number STRING`, `hire_date DATE NOT NULL`, `job_id STRING NOT NULL`, `salary NUMERIC NOT NULL`, `commission_pct NUMERIC`, `manager_id INT64`, `department_id INT64`, `status STRING NOT NULL`, `created_date TIMESTAMP NOT NULL`, `updated_date TIMESTAMP`.
   - **Partitioning:** `DAY` on `created_date`.
   - **Clustering:** `department_id`, `status`, `job_id` (in this order).

2. **Create BigQuery `watermark` table.** Schema: `table_name STRING NOT NULL`, `last_successful_watermark TIMESTAMP NOT NULL`, `pipeline_mode STRING NOT NULL`, `updated_at TIMESTAMP NOT NULL`. This table persists the extraction high-water mark per table for incremental loads.

3. **Create BigQuery `audit_log` table.** Schema: `run_id STRING NOT NULL`, `pipeline_mode STRING NOT NULL`, `table_name STRING NOT NULL`, `rows_extracted INT64`, `rows_loaded INT64`, `validation_status STRING NOT NULL`, `null_violations INT64`, `schema_check_result STRING NOT NULL`, `run_timestamp TIMESTAMP NOT NULL`.

4. **Implement BigQuery native LOAD JOB.** Trigger a BigQuery Load Job targeting a transient `stg_employees_staging_<run_id>` table (same schema as `stg_employees`, no partitioning). Source URIs: `gs://<staging-bucket>/employees/full/<run_id>/part-*.parquet`. Format: `PARQUET`. Write disposition: `WRITE_TRUNCATE` for full load.

5. **Implement MERGE into `stg_employees`.** After the Load Job completes, execute a BigQuery `MERGE` statement using `employee_id` as the match key: on match, `UPDATE SET` all non-key columns; on no match, `INSERT` the full row. This enforces PK uniqueness on every load execution.

6. **Implement post-load row count validation.** Query `SELECT COUNT(*) FROM verizon_data_deah.stg_employees` and compare against `rows_extracted` captured in Phase 2 Step 5. If counts match, write `PASSED` to `audit_log.validation_status`. If counts diverge by more than 0 rows, write `FAILED` and emit a Cloud Logging `ERROR` entry with the delta.

7. **Write run record to `audit_log`.** Persist all captured run metadata (`run_id`, `pipeline_mode = 'full'`, `table_name = 'employees'`, `rows_extracted`, `rows_loaded`, `validation_status`, `null_violations`, `schema_check_result`, `run_timestamp`) as a single row insert.

8. **Drop transient staging table.** Delete `stg_employees_staging_<run_id>` after successful MERGE and audit write to avoid accumulation of transient tables.

9. **Apply GCS lifecycle policy.** Confirm that the 7-day object lifecycle policy is active on the GCS staging bucket. Verify that the Parquet files written in Phase 2 are subject to this policy.

## Phase 4: Incremental Load Pipeline

> **Prerequisite:** Blocker #2 resolved and `updated_date` column confirmed as reliable delta column.

1. **Implement watermark read logic.** At job start in `incremental` mode, execute `SELECT last_successful_watermark FROM verizon_data_deah.watermark WHERE table_name = 'employees'`. If no row exists (first incremental run), fall back to the minimum `created_date` value observed in `stg_employees`.

2. **Implement delta extraction query.** Execute the following MySQL query using the watermark `ts` resolved in Step 1:
   ```sql
   SELECT EMPLOYEE_ID, FIRST_NAME, LAST_NAME, EMAIL, PHONE_NUMBER,
          HIRE_DATE, JOB_ID, SALARY, COMMISSION_PCT, MANAGER_ID,
          DEPARTMENT_ID, STATUS, CREATED_DATE, UPDATED_DATE
   FROM   agentichub.employees
   WHERE  COALESCE(UPDATED_DATE, CREATED_DATE) >= :last_watermark
   ORDER  BY EMPLOYEE_ID
   ```
   Apply the same chunked Parquet serialisation and GCS write pattern as Phase 2, targeting path `gs://<staging-bucket>/employees/incremental/<run_id>/part-<batch_seq>.parquet`.

3. **Apply the same pre-load schema compatibility check and null scan** as Phase 2 Steps 3 and 4, using the same abort conditions.

4. **Execute BigQuery Load Job for delta files.** Target the same transient staging table pattern (`stg_employees_staging_<run_id>`). Write disposition: `WRITE_TRUNCATE` on the transient table.

5. **Execute MERGE into `stg_employees`** using the same MERGE statement defined in Phase 3 Step 5. This is the mechanism that prevents duplication across incremental cycles.

6. **Update watermark table.** After a successful MERGE, execute an `INSERT OR UPDATE` into `verizon_data_deah.watermark` setting `last_successful_watermark = MAX(COALESCE(updated_date, created_date))` from the rows loaded in this run. Only update the watermark on confirmed success — never on partial or failed runs.

7. **Write run record to `audit_log`** with `pipeline_mode = 'incremental'` and all captured metrics, using the same pattern as Phase 3 Step 7.

8. **Drop transient staging table** following the same pattern as Phase 3 Step 8.

## Phase 5: Orchestration — Cloud Composer 2 DAG

1. **Implement the Airflow DAG.** Create a DAG named `mysql_bq_employees_ingestion` in Cloud Composer 2. Define two modes of execution: (a) initial full load run (one-time, manual trigger), (b) scheduled incremental runs at the agreed cadence (to be confirmed when Blocker #3 is resolved).

2. **Define DAG task sequence for full load.** Tasks in order:
   - `check_connectivity` — calls Phase 1 connectivity check endpoint.
   - `run_schema_snapshot` — triggers extraction Cloud Run Job in schema-snapshot-only mode.
   - `run_full_extraction` — triggers extraction Cloud Run Job with `--mode full`.
   - `run_bq_load_and_merge` — triggers processing Cloud Run Job to execute Load Job + MERGE.
   - `run_post_load_validation` — triggers validation Cloud Run Job; fails task on count mismatch.
   - `write_audit_log` — triggers audit write step; marks DAG run complete.

3. **Define DAG task sequence for incremental load.** Tasks in order:
   - `read_watermark` — retrieves last successful watermark from BigQuery.
   - `run_schema_snapshot` — same as full load task.
   - `run_incremental_extraction` — triggers extraction Cloud Run Job with `--mode incremental`.
   - `run_bq_load_and_merge` — same as full load task.
   - `run_post_load_validation` — same as full load task.
   - `update_watermark` — updates BigQuery watermark table on success only.
   - `write_audit_log` — same as full load task.

4. **Configure task retries and timeouts.** Set `retries=2`, `retry_delay=timedelta(minutes=5)` on extraction and load tasks. Set a task-level `execution_timeout` appropriate to the expected run duration established during volume sizing.

5. **Configure DAG-level failure callback.** On DAG failure, emit a structured Cloud Logging `CRITICAL` entry with `dag_id`, `run_id`, `failed_task`, and `run_timestamp`. This log entry is the trigger source for the Cloud Monitoring alert policy defined in Phase 6.

## Phase 6: Monitoring & Alerting

> **Prerequisite:** Blocker #3 resolved before SLA thresholds are set on alert policies.

1. **Create Cloud Monitoring log-based alert for DAG failure.** Configure a log-based metric filtering on `severity=CRITICAL` and `jsonPayload.dag_id="mysql_bq_employees_ingestion"`. Create an alert policy with notification threshold of 0 occurrences (alert on first event). Set the SLA notification window per the agreed values from Blocker #3.

2. **Create Cloud Monitoring alert for Cloud Run Job non-zero exit code.** Configure a metric alert on `run.googleapis.com/job/completed_task_attempt_count` filtered by `exit_code != 0` and job name matching the ingestion job. Set the same SLA notification window.

3. **Configure notification channels.** Wire both alert policies to the agreed email and/or PagerDuty channels. Confirm delivery by triggering a test notification.

4. **Configure Looker Studio audit dashboard.** Connect a Looker Studio report directly to `verizon_data_deah.audit_log`. Include scorecards for: last run `validation_status`, last run `rows_extracted` vs `rows_loaded` delta, `null_violations` trend, `schema_check_result` per run, and `run_timestamp` of most recent successful load. Share the dashboard with stakeholder Yash and confirm read access.

5. **Validate Cloud Logging structured output.** Confirm that each Cloud Run Job emits structured JSON logs covering: `run_id`, `pipeline_mode`, `table_name`, `step`, `status`, `rows_processed`, `error_message` (when applicable). Verify logs are queryable in Cloud Logging using `jsonPayload.run_id` as a filter.

## Phase 8: Handover

1. **Update runbook — pipeline design summary.** Document the following in the team runbook under SCRUM-123:
   - Architecture pattern: Cloud Run Jobs (Python 3.12, Cloud SQL Auth Proxy sidecar) → GCS Parquet staging (7-day lifecycle) → BigQuery native Load Job → MERGE into `verizon_data_deah.stg_employees`.
