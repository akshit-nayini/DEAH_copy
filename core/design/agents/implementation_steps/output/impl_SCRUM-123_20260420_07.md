## ⚠️ Open Blockers

The following blockers are unresolved and must be answered before the corresponding phases are finalised. Work may proceed up to the gate indicated, but must pause until the blocker is resolved.

> **BLOCKER-1 — Data Volume (Gate: Phase 4, Step 1)**
> The approximate row count and byte size of the MySQL `employees` table is unknown. This is the most critical missing input: it directly determines whether Cloud Run Jobs (< 50 GB) or Dataflow (≥ 50 GB) is the correct compute tier and whether cost estimates can be produced. Historical full load must not begin until this is confirmed and the architecture validated against the threshold.

> **BLOCKER-2 — Monitoring Alert SLA (Gate: Phase 6, Step 4)**
> The quantified SLA for pipeline failure alerting (e.g. alert within 15 min, 30 min, or 1 hr of job failure) has not been provided. Cloud Monitoring alerting policies cannot be finalised without it.

> **BLOCKER-3 — Source `updated_date` Reliability (Gate: Phase 5, Step 1)**
> It must be confirmed that the MySQL `employees.updated_date` column is reliably server-side updated on every INSERT and UPDATE. If it is not, the incremental strategy cannot rely on `MAX(updated_date)` as the high-water mark and must fall back exclusively to primary key delta, which will not detect in-place updates to existing rows.

## ⚠️ Risks & Mitigations *(Reference only — not implementation steps)*

| Risk | Mitigation |
|---|---|
| **Data Volume Uncertainty** | Instrument the first historical full load with explicit row count and byte volume logging to Cloud Logging. If volume exceeds 50 GB, implement chunked pagination with a configurable page size, or escalate to Dataflow migration. |
| **Watermark State Integrity** | Enable GCS Object Versioning on the watermark state bucket. Implement a checksum integrity check at Cloud Run Job startup before reading the state file. Maintain a recovery runbook for state file corruption that triggers a bounded re-load from the last known good watermark. |
| **Schema Evolution** | Pre-load schema diff task compares MySQL `INFORMATION_SCHEMA.COLUMNS` against the BigQuery table schema before every load. Auto-add new nullable columns; apply default value strategy and alert for new NOT NULL columns; halt and alert on data type conflicts. |

## Phase 1: Source Connectivity & Schema Validation

**Objective:** Confirm end-to-end network access to the MySQL source and validate that the source schema matches the expected data model before any code is written.

1. **Connectivity check — MySQL source via Cloud SQL Auth Proxy.**
   From a Cloud Run Job test container (or a Cloud Shell session within the same VPC), start the Cloud SQL Auth Proxy targeting instance `verizon-data:us-central1:mysql-druid-metadatastore`. Establish a SQLAlchemy connection to `agentichub` at `34.70.79.163:3306` with username `sa`. Confirm the connection succeeds and the `employees` table is visible.

2. **Source schema introspection.**
   Execute `SELECT COLUMN_NAME, DATA_TYPE, IS_NULLABLE, CHARACTER_MAXIMUM_LENGTH, NUMERIC_PRECISION, NUMERIC_SCALE, COLUMN_KEY FROM INFORMATION_SCHEMA.COLUMNS WHERE TABLE_SCHEMA = 'agentichub' AND TABLE_NAME = 'employees' ORDER BY ORDINAL_POSITION;`. Capture the full result as the authoritative source schema baseline and persist it to a JSON file in the GCS watermark/state bucket under a `schema_snapshots/` prefix.

3. **Source-to-target schema diff — manual pre-code review.**
   Compare the introspected MySQL schema against the target `stg_employees` column definitions (16 columns: `employee_id`, `first_name`, `last_name`, `email`, `phone_number`, `hire_date`, `job_id`, `salary`, `commission_pct`, `manager_id`, `department_id`, `status`, `created_date`, `updated_date`). Confirm type mappings: MySQL `decimal(10,0)` → BigQuery `INT64` for `employee_id`/`manager_id`/`department_id`; `decimal(8,2)` → `NUMERIC(8,2)` for `salary`; `decimal(4,2)` → `NUMERIC(4,2)` for `commission_pct`; `datetime`/`timestamp` → `TIMESTAMP`; `date` → `DATE`; `varchar` columns → `STRING`. Document any discrepancies.

4. **Resolve BLOCKER-3 — `updated_date` reliability.**
   Execute `SELECT COUNT(*) FROM employees WHERE updated_date IS NULL;` and cross-check with `created_date` population. Confirm whether `updated_date` is server-managed (e.g. `ON UPDATE CURRENT_TIMESTAMP`) or application-managed. Record the outcome and feed it into Phase 5 watermark strategy selection.

5. **Record source row count baseline.**
   Execute `SELECT COUNT(*) AS total_rows FROM agentichub.employees;`. Log the result to Cloud Logging and document it as the authoritative baseline for historical load validation. Use this count to resolve BLOCKER-1 by also running `SELECT data_length + index_length AS bytes FROM information_schema.TABLES WHERE table_schema = 'agentichub' AND table_name = 'employees';`.

## Phase 3: ETL Container Development (Cloud Run Job)

**Objective:** Build the parameterised Python 3.11 Cloud Run Job container that handles full and incremental loads, schema diffing, watermark management, and post-load assertions.

1. **Implement `schema_diff.py` module.**
   On every job invocation, query `INFORMATION_SCHEMA.COLUMNS` for `agentichub.employees` and compare against the live BigQuery `stg_employees` schema via `client.get_table()`. Implement the following decision tree:
   - **New nullable column in MySQL:** auto-execute `ALTER TABLE` equivalent (`client.update_table`) to add the column to BigQuery with `NULLABLE` mode; log the addition to the audit log.
   - **New NOT NULL column in MySQL:** apply a configured default value, add the column to BigQuery, insert a `WARNING` entry in the audit log, and emit a structured alert log entry for stakeholder notification.
   - **Data type conflict:** log the conflict, write `FAILED` status to the audit log, and raise a non-zero exit code to halt the job before any data is loaded.
   - **No changes:** log `SCHEMA_OK` and proceed.

2. **Implement `watermark.py` module.**
   At job startup: (a) read `gs://<watermark-bucket>/state/employees_watermark.json`; (b) compute `sha256` of the file content and compare against the embedded `checksum` field — if they differ, log a `WATERMARK_CORRUPT` error and exit non-zero; (c) return the parsed `watermark_ts` and `watermark_pk` values. At job completion (success only): write the new watermark values and updated checksum back to GCS, which creates a new object version for rollback safety.

3. **Implement `extract.py` module — full load path.**
   Using SQLAlchemy with the Cloud SQL Auth Proxy connection string, execute `SELECT * FROM agentichub.employees` (no delta filter). Stream results into a pandas DataFrame in configurable page sizes (default: 100,000 rows). Log total rows extracted and bytes processed to Cloud Logging. If byte volume exceeds 50 GB (per BLOCKER-1 resolution), raise a warning and exit — pending architectural re-evaluation.

4. **Implement `extract.py` module — incremental load path.**
   Parameterised by `load_type=INCREMENTAL`. Apply the delta filter: `WHERE updated_date > :watermark_low AND updated_date <= :watermark_high` (primary strategy, contingent on BLOCKER-3 resolution) with `watermark_high = NOW() - interval 5 MINUTE` to avoid capturing in-flight transactions. If `updated_date` is confirmed unreliable per BLOCKER-3, switch to `WHERE employee_id > :watermark_pk` and document the limitation (in-place updates will not be detected).

5. **Implement `transform.py` module.**
   Cast MySQL `decimal(10,0)` columns (`employee_id`, `manager_id`, `department_id`) to Python `int`. Cast `decimal(8,2)` `salary` and `decimal(4,2)` `commission_pct` to `decimal.Decimal` to preserve precision. Cast `date` and `datetime` columns to their Python equivalents for BigQuery type compatibility. Apply no other transformations — do not rename, derive, or enrich columns.

6. **Implement `load.py` module.**
   For **full load**: truncate `stg_employees` using a `TRUNCATE TABLE` DML statement before loading, then execute a BigQuery load job via `client.load_table_from_dataframe`. For **incremental load**: execute a BigQuery `MERGE` statement using `employee_id` as the merge key — `UPDATE` matched rows on all non-key columns, `INSERT` unmatched rows — to prevent duplication and support updates.

7. **Implement `assertions.py` module.**
   After every load: (a) query `SELECT COUNT(*) FROM agentichub.employees` (full load) or the delta-bounded count (incremental) and compare against `SELECT COUNT(*) FROM verizon_data_deah.stg_employees` or the affected partition; (b) execute `SELECT COUNT(*) - COUNT(DISTINCT employee_id) AS pk_violations FROM verizon_data_deah.stg_employees` — if result > 0, log `PK_VIOLATION`, write to audit log, and exit non-zero. Write final row counts and assertion results to the `pipeline_audit_log` table.

8. **Implement `main.py` entrypoint.**
   Accept environment variables: `LOAD_TYPE` (`FULL`|`INCREMENTAL`), `PROJECT_ID`, `DATASET_ID`, `WATERMARK_BUCKET`, `PAGE_SIZE`. Orchestrate: watermark read → schema diff → extract → transform → load → assertions → watermark write → audit log write. Ensure all unhandled exceptions are caught, written to the audit log with `FAILED` status, and re-raised to produce a non-zero exit code for Cloud Run Job failure signalling.

9. **Build and push container image.**
   Build the Docker image using Python 3.11 slim base with `requirements.txt` pinning `sqlalchemy`, `pandas`, `google-cloud-bigquery`, `google-cloud-storage`, `pymysql`, and `cryptography`. Push to Artifact Registry at `us-central1-docker.pkg.dev/verizon-data/etl-images/mysql-bq-employees:<git-sha>`.

## Phase 4: Historical Full Load

**Objective:** Execute and validate the one-time full historical load of the `employees` table. This phase is a hard gate — incremental load activation is blocked until this phase is signed off.

> ⚠️ **Prerequisite:** BLOCKER-1 (data volume) must be resolved before Step 1. If byte volume ≥ 50 GB, halt and escalate architecture review before proceeding.

1. **Trigger full load Cloud Run Job** with `LOAD_TYPE=FULL`. Monitor execution via Cloud Logging using the structured log filter `jsonPayload.pipeline_name="mysql-bq-employees" AND jsonPayload.load_type="FULL"`.

2. **Validate row count parity.**
   Query `SELECT COUNT(*) FROM agentichub.employees` (source) and `SELECT COUNT(*) FROM verizon_data_deah.stg_employees` (target). Both counts must be equal. Cross-check against the Phase 1 Step 5 baseline count.

3. **Validate PK uniqueness.**
   Execute `SELECT COUNT(*) - COUNT(DISTINCT employee_id) AS pk_violations FROM verizon_data_deah.stg_employees`. Result must be `0`.

4. **Validate NOT NULL enforcement.**
   Execute a BigQuery query asserting `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, and `created_date` have zero NULL values in the loaded dataset.

5. **Validate partition and clustering.**
   Run `SELECT hire_date, COUNT(*) FROM verizon_data_deah.stg_employees GROUP BY 1 ORDER BY 1` and confirm rows are distributed across `hire_date` month partitions as expected. Confirm clustering columns are set correctly via `bq show`.

6. **Validate audit log entry.**
   Query `pipeline_audit_log` and confirm one `FULL` run record exists with `status=SUCCESS`, matching `source_row_count` and `target_row_count`, and a populated `run_end_ts`.

7. **Sign off on historical load.**
   Obtain written stakeholder or tech lead sign-off confirming historical load is validated. Record sign-off reference in the project ticket SCRUM-123. Only after sign-off is recorded does Phase 5 activation proceed.

## Phase 5: Incremental Load Pipeline

**Objective:** Implement and validate the incremental load pipeline, activated only after Phase 4 sign-off.

> ⚠️ **Prerequisite:** Phase 4 sign-off must be recorded. BLOCKER-3 (`updated_date` reliability) must be resolved to confirm the delta strategy before Step 1.

1. **Confirm and document the incremental delta strategy.**
   Based on BLOCKER-3 resolution: if `updated_date` is server-managed, use `MAX(updated_date)` as the high-water mark written to the GCS state file after each successful run. If `updated_date` is unreliable, switch to primary key delta (`MAX(employee_id)`), document the limitation (in-place updates undetectable), and notify stakeholders.

2. **Set initial watermark value.**
   Write the watermark state file with `watermark_ts` set to `MAX(updated_date)` from the current `stg_employees` table (or `MAX(employee_id)` for PK strategy). Compute and embed the `sha256` checksum. This anchors the first incremental run at the end of the historical dataset.

3. **Execute first incremental load dry-run.**
   Trigger the Cloud Run Job with `LOAD_TYPE=INCREMENTAL` against a time window with known new/changed rows (use a test INSERT/UPDATE on a staging MySQL instance if production mutations are not available). Confirm only delta rows are extracted, merged, and that the watermark advances correctly.

4. **Validate MERGE deduplication.**
   Re-trigger the incremental job with the same watermark window (simulating a retry). Confirm that `SELECT COUNT(*) - COUNT(DISTINCT employee_id) AS pk_violations FROM verizon_data_deah.stg_employees` remains `0` — the MERGE must be idempotent.

5. **Validate watermark state file update.**
   After a successful incremental run, read the GCS state file and confirm `watermark_ts` (or `watermark_pk`) has advanced to the new high-water mark and the checksum is valid.

6. **Validate watermark corruption recovery.**
   Manually corrupt the GCS state file checksum. Trigger the Cloud Run Job and confirm it exits non-zero with a `WATERMARK_CORRUPT` log entry and does not load any data.

## Phase 6: Orchestration & Monitoring

**Objective:** Deploy the Cloud Composer 2 DAG for scheduled execution and configure Cloud Monitoring alerting policies.

1. **Implement the Cloud Composer 2 DAG.**
   Create an Airflow DAG `mysql_bq_employees_pipeline` with the following task sequence:
   - `check_source_connectivity` — a PythonOperator that verifies MySQL reachability before any load is triggered.
   - `run_full_load` (conditional, `LOAD_TYPE=FULL`) — a `CloudRunJobOperator` (or equivalent) targeting the `mysql-bq-employees` Cloud Run Job.
   - `run_incremental_load` (default, `LOAD_TYPE=INCREMENTAL`) — same operator, delta mode.
   - `validate_audit_log` — a BigQueryOperator asserting the most recent `pipeline_audit_log` entry for this run has `status=SUCCESS`.
   The DAG must be parameterised via Airflow Variables for `PROJECT_ID`, `DATASET_ID`, `WATERMARK_BUCKET`, and `PAGE_SIZE`.

2. **Configure DAG schedule.**
   Set the DAG schedule interval to the agreed cadence (to be confirmed with stakeholders — not specified in inputs). Set `catchup=False` to prevent backfill on deployment. Set `max_active_runs=1` to prevent concurrent executions.

3. **Configure retry policy.**
   Set `retries=2` and `retry_delay=timedelta(minutes=5)` on the Cloud Run Job tasks. Ensure that retries re-read the watermark from GCS (not from task state) so each retry uses the last committed watermark.

4. **Configure Cloud Monitoring alerting policies.**
   Create log-based alerting metrics on: (a) Cloud Run Job exit code ≠ 0 for the `mysql-bq-employees` job; (b) structured log entries where `jsonPayload.status="FAILED"` in the pipeline log stream; (c) `WATERMARK_CORRUPT` log entries. Set notification channels per the agreed SLA once BLOCKER-2 is resolved. **This step is blocked by BLOCKER-2.**

5. **Configure Looker Studio dashboard.**
   Connect Looker Studio to `verizon_data_deah.pipeline_audit_log`. Build a dashboard showing: last run status, source vs. target row counts per run, schema diff results, watermark progression over time, and run duration trend. Share with stakeholders for self-service health visibility.

6. **Validate end-to-end DAG execution.**
   Trigger the DAG manually from the Composer UI. Confirm all tasks complete with status `success`. Confirm the audit log entry is written, the watermark state file is updated, and the Looker Studio dashboard reflects the latest run.

## Phase 7: Testing

**Objective:** Execute concrete, requirements-derived test cases to validate correctness, reliability, and resilience of the pipeline.

---

### TC-01 — Source Connectivity via Auth Proxy
- **What is tested:** Network path from Cloud Run Job to MySQL source through Cloud SQL Auth Proxy.
- **Input:** Cloud Run Job launched with valid Auth Proxy sidecar config targeting `verizon-data:us-central1:mysql-druid-metadatastore`.
- **Expected outcome:** SQLAlchemy connection to `agentichub` at `34.70.79.163:3306` succeeds; `employees` table is visible in `INFORMATION_SCHEMA`.

### TC-02 — Historical Full Load Row Count Parity
- **What is tested:** All rows from MySQL `employees` are loaded to BigQuery `stg_employees` without loss or duplication.
- **Input:** Full load job executed against the production MySQL source with `LOAD_TYPE=FULL`.
- **Expected outcome:** `COUNT(*) FROM agentichub.employees` == `COUNT(*) FROM verizon_data_deah.stg_employees`. Audit log records matching `source_row_count` and `target_row_count`.

### TC-03 — Primary Key Uniqueness After Full Load
- **What is tested:** `employee_id` uniqueness constraint is enforced post-load.
- **Input:** Full load completed per TC-02.
- **Expected outcome:** `SELECT COUNT(*) - COUNT(DISTINCT employee_id) FROM verizon_data_deah.stg_employees` returns `0`.

### TC-04 — NOT NULL Column Enforcement
- **What is tested:** Columns defined as NOT NULL in the target schema contain no null values after load.
- **Input:** Full load completed per TC-02.
- **Expected outcome:** Zero NULL values in `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date` across all loaded rows.

### TC-05 — NUMERIC Precision Preservation
- **What is tested:** `salary` (precision 8, scale 2) and `commission_pct` (precision 4, scale 2) values are not truncated or rounded during load.
- **Input:** Source rows containing `salary = 12345678.99` and `commission_pct = 99.99`.
- **Expected outcome:** BigQuery target rows contain identical values with no loss of precision.

### TC-06 — Partition Correctness
- **What is tested:** Rows are physically placed in the correct `hire_date` MONTH partitions.
- **Input:** Loaded dataset with employees hired across at least two distinct calendar months.
- **Expected outcome:** `SELECT DATE_TRUNC(hire_date, MONTH), COUNT(*) FROM stg_employees GROUP BY 1` returns at least two distinct partition keys; a partition-pruning query with a specific month filter reads only that partition (confirmed via query bytes processed).

### TC-07 — Schema Drift — New Nullable Column
- **What is tested:** Auto-detection and auto-addition of a new nullable MySQL column before load.
- **Input:** A new nullable `VARCHAR(50)` column `middle_name` is added to MySQL `employees`. Incremental load job is triggered.
- **Expected outcome:** Schema diff detects the new column; `middle_name STRING NULLABLE` is added to `stg_employees` in BigQuery before load proceeds; audit log records the schema addition; load completes successfully.

### TC-08 — Schema Drift — Data Type Conflict
- **What is tested:** Pipeline halts and alerts on an irreconcilable type conflict between MySQL and BigQuery schemas.
- **Input:** The MySQL `salary` column is altered to `VARCHAR(50)` in the test environment. Load job is triggered.
- **Expected outcome:** Schema diff detects the type conflict; job exits non-zero before any data is loaded; audit log entry has `status=FAILED` with the conflict detail; Cloud Monitoring alert fires.

### TC-09 — Incremental Load — New Rows Only
- **What is tested:** Only rows with `updated_date` (or `employee_id`, per BLOCKER-3 resolution) beyond the current watermark are extracted and loaded.
- **Input:** Two new employees are inserted into MySQL with `updated_date = NOW()`, after the current high-water mark. Incremental job is triggered.
- **Expected outcome:** Exactly 2 new rows appear in `stg_employees`; existing rows are unchanged; watermark advances to the new `MAX(updated_date)`.

### TC-10 — Incremental Load Idempotency (MERGE Deduplication)
- **What is tested:** Re-running the incremental job with the same watermark window does not create duplicate rows.
- **Input:** Incremental job is triggered a second time with the same watermark low/high bounds as TC-09.
- **Expected outcome:** `stg_employees` row count does not increase; `SELECT COUNT(*) - COUNT(DISTINCT employee_id) FROM stg_employees` returns `0`.

### TC-11 — Watermark State Checksum Validation
- **What is tested:** A corrupted watermark state file is detected at job startup and the job halts without loading data.
- **Input:** GCS watermark state file `employees_watermark.json` checksum field is manually altered to an invalid value. Incremental job is triggered.
- **Expected outcome:** Job reads the state file, computes the checksum, detects the mismatch, logs `WATERMARK_CORRUPT`, exits non-zero, and loads zero rows. Audit log records `status=FAILED`.

### TC-12 — Self-Referential FK Integrity
- **What is tested:** All non-null `manager_id` values in `stg_employees` reference a valid `employee_id` in the same table.
- **Input:** Full load completed per TC-02.
- **Expected outcome:** `SELECT COUNT(*) FROM stg_employees e LEFT JOIN stg_employees m ON e.manager_id = m.employee_id WHERE e.manager_id IS NOT NULL AND m.employee_id IS NULL` returns `0`. Top-level employees (e.g. CEO) carry `manager_id = NULL`.

### TC-13 — Audit Log Completeness
- **What is tested:** Every load run (full and incremental) produces exactly one audit log entry with all required fields populated.
- **Input:** One full load and one incremental load executed.
- **Expected outcome:** `pipeline_audit_log` contains one `FULL` record and one `INCREMENTAL` record; both have non-null `run_id`, `run_start_ts`, `run_end_ts`, `status`, `source_row_count`, `target_row_count`, and `watermark_high`.

### TC-14 — Cloud Run Job Failure Alerting
- **What is tested:** A failing Cloud Run Job exit code triggers a Cloud Monitoring alert.
- **Input:** Cloud Run Job is forced to exit non-zero (e.g. by pointing at an invalid BigQuery dataset). *(BLOCKER-2 must be resolved to set the alert SLA threshold.)*
- **Expected outcome:** Cloud Monitoring alert fires within the agreed SLA window; notification is delivered to the configured channel.
