## ⚠️ Open Blockers

The following blockers must be resolved before the corresponding phases can be completed or signed off. Development may proceed on non-dependent steps in parallel, but **no production deployment gate may be cleared** while any blocker remains open.

> **BLOCKER 1 — Table Size & Volume Unknown**
> The approximate row count, total data size (GB), and daily row delta of `agentichub.employees` have not been provided. This information is required to: (a) size Cloud Run Job memory limits (default 512 MB, max 32 GB), (b) determine whether chunked paginated extraction is required for the historical load, and (c) produce a meaningful BigQuery storage and query cost estimate prior to deployment. **Owner must supply these figures before Phase 3 container sizing is finalised.**

> **BLOCKER 2 — `updated_date` Reliability Unconfirmed**
> It has not been confirmed that the `UPDATED_DATE` / `updated_at` column on `agentichub.employees` is reliably populated on every INSERT and UPDATE operation by the application. If this column is absent or inconsistently maintained, the entire incremental extraction strategy must be redesigned (primary key range scan or full-table hash comparison). **Owner must audit application write patterns and confirm or deny reliability before Phase 5 begins.**

> **BLOCKER 3 — Hard-Delete Strategy Undefined**
> It is unknown whether records in `agentichub.employees` are hard-deleted or soft-deleted (via a `deleted_at` or `is_active` / `status` flag). Hard deletes are invisible to timestamp-based incremental extraction and will cause BigQuery to retain stale rows, producing data divergence. **Owner must confirm the delete pattern before the incremental load DAG is deployed to production.**

Issue `SELECT 1` and log the round-trip latency. Confirm the proxy sidecar authenticates successfully via the service account IAM binding. Fail fast and raise a blocking alert if the connection cannot be established.

2. **Retrieve source schema from `information_schema`.** Execute the following query against the live MySQL instance and capture the result as the authoritative source schema artefact:
   ```sql
   SELECT COLUMN_NAME, DATA_TYPE, CHARACTER_MAXIMUM_LENGTH,
          NUMERIC_PRECISION, NUMERIC_SCALE, IS_NULLABLE, COLUMN_KEY
   FROM information_schema.COLUMNS
   ```

## Phase 3: Cloud Run Job — Historical Full Load

**Goal:** Build and validate the containerised Python job that performs a complete extraction of `agentichub.employees` and loads all rows into `stg_employees` with atomicity and auditability. *(Contingent on Blocker 1 for memory/chunk-size finalisation.)*

1. **Implement the Cloud Run Job container entrypoint** (`main.py`) with the following sequential execution steps, each wrapped in structured exception handling that writes to Cloud Logging as JSON:
   - Step A: Pre-load schema compatibility check (compare `information_schema` result against the type mapping registry; log schema diff JSON; abort if mismatches detected).
   - Step B: Paginated full extraction from `agentichub.employees` using `pymysql` server-side cursor with a configurable `CHUNK_SIZE` (default `50 000` rows per page; finalise based on Blocker 1 resolution).
   - Step C: Explicit Python type coercion per the mapping registry (e.g. `decimal` → `int` for `employee_id`, `decimal` → `Decimal` for `salary`/`commission_pct`).
   - Step D: Write each chunk to BigQuery using the **BigQuery Storage Write API** in `COMMITTED` mode for atomic, exactly-once delivery into `stg_employees`.
   - Step E: Post-load row count comparison — execute `SELECT COUNT(*) FROM agentichub.employees` and `SELECT COUNT(*) FROM verizon_data_deah.stg_employees`; assert equality; log both values to `audit_log`.
   - Step F: PK uniqueness enforcement — execute a BigQuery `MERGE` DML statement that identifies duplicate `employee_id` values and retains only the row with the latest `updated_date` (falling back to `created_date`).
   - Step G: Null constraint violation report — query `stg_employees` for rows where `phone_number IS NULL`, `commission_pct IS NULL`, or `updated_date IS NULL`; log counts as structured JSON to Cloud Logging; write `null_violation_count` to `audit_log`.
   - Step H: Write the final structured audit record to `audit_log` with status `SUCCESS`, all row counts, schema diff, null violation count, watermark values, and duration.

2. **Implement secrets retrieval at container startup.** Source the MySQL password and `instance_connection_name` exclusively from Google Secret Manager using the Secret Manager Python client. Do not accept credentials as plain environment variables.

3. **Containerise and push to Artifact Registry.** Write the `Dockerfile` using Python 3.11 slim base image. Install dependencies: `google-cloud-bigquery`, `google-cloud-bigquery-storage`, `google-cloud-secret-manager`, `pymysql`. Build the image and push to the designated Artifact Registry repository. Tag with the commit SHA and `latest`.

4. **Configure Cloud Run Job definition** via Terraform: set memory limit (to be finalised per Blocker 1; initial placeholder `2Gi`), CPU limit `2`, max retries `1`, timeout `3600s` (adjust after volume is known), parallelism `1`. Bind the Secret Manager secrets as environment variable secret references. Set `LOAD_TYPE=FULL` as a plain environment variable for this job definition.

5. **Execute a dry-run against a non-production dataset** (or a row-limited extract: `LIMIT 1000`) to validate the end-to-end container execution path, schema check, type coercion, Storage Write API write, and audit log write before running against the full table.

## Phase 4: Cloud Composer DAG — Historical Load Orchestration

**Goal:** Define and deploy the Airflow DAG that orchestrates the historical full load, enforces the pre-load gate checks, and prevents incremental load activation until this phase is signed off.

1. **Author the `dag_employees_full_load.py` DAG** with the following task sequence:
   - `task_schema_check` — `BashOperator` or `PythonOperator` invoking the schema compatibility check endpoint (or a dedicated pre-flight Cloud Run Job task variant).
   - `task_run_full_load` — `GCPCloudRunJobExecuteAsyncOperator` targeting the Cloud Run Job with `LOAD_TYPE=FULL`.
   - `task_validate_row_counts` — `BigQueryCheckOperator` asserting `rows_extracted == rows_loaded` from the latest `audit_log` record for `table_name = 'employees'` and `load_type = 'FULL'`.
   - `task_validate_no_pk_duplicates` — `BigQueryCheckOperator` asserting `SELECT COUNT(*) = 0 FROM (SELECT employee_id, COUNT(*) c FROM verizon_data_deah.stg_employees GROUP BY 1 HAVING c > 1)`.
   - `task_log_null_violations` — `BigQueryValueCheckOperator` reading `null_violation_count` from `audit_log`; does not block the DAG but logs a warning to Cloud Logging if count > 0.
   - `task_set_historical_load_complete_flag` — writes a sentinel record to `control_watermarks` with `load_type = 'FULL'` and `last_watermark` set to the maximum `updated_date` observed in `stg_employees` (falling back to max `created_date`). This record gates incremental load activation.

2. **Set DAG schedule to `None` (manual trigger only)** for the historical load. The DAG is not intended to run on a recurring schedule.

3. **Configure Airflow Connections.** Register the BigQuery and Cloud Run Job connections in Composer using `SecretManagerBackend` for any credential material. Do not store credentials in the Airflow metastore.

4. **Deploy DAG to the Composer GCS DAGs bucket** and confirm it parses without import errors in the Airflow UI. Verify the task dependency graph renders correctly.

5. **Trigger a manual DAG run** against the non-production dataset from Phase 3 step 5. Confirm all tasks reach `success` state. Review `audit_log` and `control_watermarks` entries in BigQuery to confirm values are correct.

## Phase 5: Cloud Run Job — Incremental Load

**Goal:** Extend the Cloud Run Job to support timestamp-based delta extraction using `updated_date` as the primary watermark and `created_date` as the fallback, with watermark state managed via `control_watermarks`. *(Contingent on Blocker 2 and Blocker 3 resolution.)*

1. **Implement watermark read at job startup.** At the start of each incremental run, read `last_watermark` from `control_watermarks` where `table_name = 'employees'` and `load_type = 'INCREMENTAL'`. If no row exists (first incremental run), read `last_watermark` from the `FULL` sentinel written in Phase 4 step 1. Store the value as `watermark_start`.

2. **Implement delta extraction query.** Construct the extraction query as:
   ```sql
   SELECT *
   FROM agentichub.employees
   WHERE COALESCE(updated_date, created_date) > :watermark_start
   ORDER BY COALESCE(updated_date, created_date) ASC
   ```
   Apply the same paginated server-side cursor pattern and chunk size as the full load.

3. **Apply the same processing pipeline** from Phase 3 step 1 (Steps A through H) to the incremental payload, with the following adjustments:
   - Step A schema check: run on every incremental execution.
   - Step E row count: compare MySQL `COUNT(*)` for `WHERE COALESCE(updated_date, created_date) > :watermark_start` against BigQuery rows written in this run.
   - Step F PK deduplication: apply the same `MERGE` DML after each incremental write to handle late-arriving updates and any duplicate keys introduced by the delta batch.
   - Step H audit record: populate `watermark_start` and `watermark_end` fields.

4. **Implement watermark write on successful completion.** On `SUCCESS` status only, upsert a record into `control_watermarks` setting `last_watermark` to the maximum `COALESCE(updated_date, created_date)` value observed in the current delta batch. Never advance the watermark on a failed or partial run.

5. **Implement the null constraint violation log** for `updated_date` specifically: before marking each incremental run complete, query the rows in the current delta batch where `updated_date IS NULL` and log the `employee_id` list as structured JSON to Cloud Logging, alongside the total count. Write this count to `null_violation_count` in `audit_log`.

6. **Set `LOAD_TYPE=INCREMENTAL`** as the environment variable distinguisher on the Cloud Run Job incremental variant definition. Rebuild and push the container image (the same image binary supports both modes via the `LOAD_TYPE` flag).

## Phase 6: Cloud Composer DAG — Incremental Load Orchestration

**Goal:** Define and deploy the recurring Airflow DAG for incremental loads, with an explicit gate that prevents execution until the historical load is confirmed complete.

1. **Author the `dag_employees_incremental_load.py` DAG** with the following task sequence:
   - `task_check_historical_gate` — `BigQueryCheckOperator` asserting that a row exists in `control_watermarks` where `table_name = 'employees'` and `load_type = 'FULL'`. If the assertion fails, the DAG fails immediately with a descriptive error message directing the operator to complete the historical load first. **This gate must never be removed or bypassed.**
   - `task_read_watermark` — `BigQueryGetDataOperator` reading `last_watermark` from `control_watermarks` and pushing it to XCom for use by downstream tasks.
   - `task_run_incremental_load` — `GCPCloudRunJobExecuteAsyncOperator` with `LOAD_TYPE=INCREMENTAL`, passing the watermark value as a job override environment variable.
   - `task_validate_row_counts` — `BigQueryCheckOperator` asserting `rows_extracted == rows_loaded` from the latest `audit_log` record for `load_type = 'INCREMENTAL'`.
   - `task_validate_no_pk_duplicates` — `BigQueryCheckOperator` asserting zero duplicate `employee_id` values in `stg_employees` post-load.
   - `task_log_null_violations` — `BigQueryValueCheckOperator` reading `null_violation_count` from `audit_log`; logs a warning without blocking if count > 0.

2. **Set DAG schedule.** Configure to run on the agreed cadence (hourly or daily — to be confirmed with the business owner). Use a Composer Variable or Airflow Variable to make the schedule configurable without a DAG code change.

3. **Set `catchup=False`** to prevent historical backfill runs on first deployment.

4. **Deploy DAG to the Composer GCS DAGs bucket** and confirm it parses cleanly. Verify the gate task renders in the dependency graph as the first node.

5. **Perform a controlled first incremental run** in a non-production environment: seed one new row and one updated row into a test MySQL instance, confirm only those rows appear in the BigQuery delta, confirm the watermark advances correctly, and confirm the audit log is populated accurately.

## Phase 7: Monitoring & Observability

**Goal:** Deliver the stakeholder self-service visibility layer and automated alerting that satisfies the architectural requirement for independent load status review without developer involvement.

1. **Create Cloud Logging log-based metrics** for the following structured log patterns emitted by the Cloud Run Job:
   - `etl/job_failure` — any Cloud Run Job execution that writes a `status = FAILURE` audit record.
   - `etl/null_violation` — any run where `null_violation_count > 0`.
   - `etl/schema_mismatch` — any pre-load schema check that flags a type mismatch, new column, or missing column.
   - `etl/row_count_mismatch` — any run where `rows_extracted != rows_loaded`.

2. **Create Cloud Monitoring alerting policies** on the log-based metrics from step 1:
   - Alert on `etl/job_failure`: notify via email and PagerDuty within the agreed SLA evaluation window.
   - Alert on `etl/schema_mismatch`: notify via email; mark as P2.
   - Alert on `etl/row_count_mismatch`: notify via email; mark as P1.
   - Alert on `etl/null_violation`: notify via email; mark as P3 (informational).

3. **Build the Looker Studio dashboard** on the `verizon_data_deah.audit_log` BigQuery table. The dashboard must include (sourced exclusively from `audit_log`):
   - Per-run status (`SUCCESS` / `FAILURE` / `PARTIAL`) with timestamps.
   - Row count trend chart (`rows_extracted` vs `rows_loaded` over time).
   - Last successful load timestamp for `employees`.
   - `null_violation_count` trend per run.
   - Schema diff flag indicator (non-empty `schema_diff_json` records).
   - Load duration trend (`duration_seconds` over time).
   Share the dashboard with stakeholder Yash and confirm they can access it independently without developer assistance.

4. **Confirm Airflow DAG run history UI access for stakeholder Yash.** Verify that Yash has an appropriate Composer / Airflow IAM role (e.g. `roles/composer.user`) to view DAG run history and task-level logs in the Airflow web UI without write access to DAG definitions.
