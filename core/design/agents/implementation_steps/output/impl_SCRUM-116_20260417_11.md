## ⚠️ Open Blockers

The following three blockers are unresolved. Each may alter implementation scope or require architectural adjustment and must be clarified before or during Phase 1.

1. **Network Connectivity Mechanism** — The precise path between the Cloud Composer worker environment and `catalog-db.corp.com:5432` is unknown (Cloud VPN, Cloud Interconnect, or public internet with TLS). This determines firewall rule scope, connectivity SLA, and whether a VPC-native Composer environment is required.

2. **Hard-Delete Propagation Requirement** — It is unconfirmed whether physical deletes from `products`, `inventory`, `categories`, or `suppliers` in PostgreSQL must be reflected in BigQuery. The current watermark-based incremental design cannot detect hard deletes. Stakeholder **John** must confirm. If required, a CDC mechanism (e.g., Debezium) will alter the scope of this engagement.

3. **Target GCP Region** — The GCP region for Cloud Composer, the GCS staging bucket, and BigQuery dataset `verizon-data.verizon_catalog_deah` has not been specified. All three resources must be co-located in a single region to eliminate cross-region transfer charges and minimise intra-pipeline latency.

## Phase 3: Extraction Pipeline Development

**Goal:** Implement the Airflow DAG tasks responsible for extracting data from PostgreSQL and writing Parquet files to GCS.

1. Register the Airflow PostgreSQL connection (connection ID: `postgres_catalog_db`) targeting `catalog-db.corp.com:5432`, database `product_db`, user `sa`. Store credentials in Airflow Secrets Backend (Secret Manager); do not hard-code in the DAG.

2. Implement the **schema fingerprint check** as the **first task in the DAG**: on every run, query `information_schema.columns` for all four source tables and compare column names, data types, and nullability against the GCS-stored baseline snapshot captured in Phase 1, Step 5. If any drift is detected, fail the DAG immediately with a P1 alert and do not proceed to extraction.

3. Implement a first-run detection mechanism using an Airflow Variable (e.g., `catalog_pipeline_first_run`). Set to `True` before the initial execution; the DAG sets it to `False` upon successful completion of the first full historical load.

4. Implement **full historical extraction** (first-run path) for all four tables (`products`, `categories`, `suppliers`, `inventory`) using psycopg2 server-side chunked cursor reads. Chunk size must be an externally configurable Airflow Variable to protect the operational source database. Do not load the full result set into Airflow worker memory.

5. Implement **watermark retrieval** for the incremental path: for `products` and `inventory`, query `MAX(updated_at)` from the respective BigQuery target table via `BigQueryHook` and pass the result forward via XCom as the lower-bound watermark for the current extraction window.

6. Implement **incremental extraction** for `products` using `WHERE updated_at > {watermark}` and for `inventory` using `WHERE updated_at > {watermark}`. Both use psycopg2 chunked cursor reads.

7. Implement **full daily reload extraction** for `categories` and `suppliers` on every non-first-run execution (all rows, no watermark filter), also using chunked cursor reads.

8. Serialise all extracted batches to Parquet format and write files to the GCS staging bucket using the path convention `gs://<staging-bucket>/<table_name>/<execution_date>/`. Confirm each file is written and non-zero in size before marking the extraction task as successful.

9. Apply retry configuration to all PostgresHook extraction tasks: **3 retries** with **60-second exponential backoff**.

## Phase 4: BigQuery Load & MERGE Pipeline Development

**Goal:** Implement the Airflow tasks that load GCS Parquet files into transient staging tables and atomically UPSERT records into production targets.

1. Implement a `BigQueryInsertJobOperator` **LOAD JOB** task for each table. Each task reads Parquet files from `gs://<staging-bucket>/<table_name>/<execution_date>/` and loads them into the corresponding `_stg_<table>` transient table using `WRITE_TRUNCATE` disposition on every run.

2. Author the parameterised MERGE SQL for **`stg_categories`**, keyed on `category_id`:
   - `WHEN MATCHED`: update `category_name`, `description`, `parent_category_id`, `created_at`.
   - `WHEN NOT MATCHED BY TARGET`: insert full row.
   - Handle `parent_category_id = NULL` (top-level categories) without error.

3. Author the parameterised MERGE SQL for **`stg_suppliers`**, keyed on `supplier_id`:
   - `WHEN MATCHED`: update `supplier_name`, `contact_email`, `country`, `is_active`, `created_at`.
   - `WHEN NOT MATCHED BY TARGET`: insert full row.

4. Author the parameterised MERGE SQL for **`stg_products`**, keyed on `product_id`:
   - `WHEN MATCHED`: update `product_name`, `sku`, `category_id`, `supplier_id`, `unit_price`, `description`, `is_active`, `created_at`, `updated_at`.
   - `WHEN NOT MATCHED BY TARGET`: insert full row.
   - Handle `category_id = NULL` and `supplier_id = NULL` without error.

5. Author the parameterised MERGE SQL for **`stg_inventory`**, keyed on `inventory_id`:
   - `WHEN MATCHED`: update `product_id`, `warehouse_code`, `quantity_on_hand`, `reorder_level`, `last_restocked_date`, `created_at`, `updated_at`.
   - `WHEN NOT MATCHED BY TARGET`: insert full row.
   - Handle `reorder_level = NULL` and `last_restocked_date = NULL` without error.

6. Implement `BigQueryInsertJobOperator` **MERGE execution tasks** for each of the four tables. Each MERGE task must be downstream of its corresponding LOAD JOB task.

7. Implement a **post-MERGE truncation task** for each `_stg_<table>` transient table. Truncation must execute only after a confirmed successful MERGE — wire task dependencies with appropriate Airflow trigger rules (`all_success`).

## Phase 5: Reconciliation, Audit Logging & Alerting

**Goal:** Implement operational integrity checks, audit logging, and alerting to ensure data quality and pipeline observability.

1. Implement a **reconciliation task** per table: after each MERGE, query `COUNT(*)` from the corresponding source table via PostgresHook and compare against `COUNT(*)` from the BigQuery target table.
   - For full-reload tables (`categories`, `suppliers`): counts must match exactly.
   - For incremental tables (`products`, `inventory`): BigQuery target count must be ≥ source count (the target accumulates history; the source count may decrease over time due to hard deletes — a known and accepted limitation).

2. Configure the reconciliation task to **fail the DAG** and trigger a PagerDuty/email alert on any count mismatch. Include `table_name`, `source_count`, `target_count`, `execution_date`, and `run_id` in the alert payload.

3. Implement the **audit log write task** using `BigQueryInsertJobOperator`: after each table's reconciliation step completes (pass or fail), write one row to `pipeline_audit.pipeline_run_log` containing `run_id`, `table_name`, `execution_date`, `status` (`SUCCESS` or `FAILED`), `source_count`, `target_count`, `duration_seconds`, and `error_message` (NULL on success).

4. Configure a DAG-level **SLA callback** for the 03:00 AM UTC deadline: implement an `sla_miss_callback` function that fires a PagerDuty/email alert if the DAG has not reached a terminal state by 03:00 AM UTC. Include DAG name, execution date, and elapsed duration in the alert payload.

5. Confirm the Cloud Monitoring **uptime check** on `catalog-db.corp.com:5432` is active with an alerting threshold configured to fire 30 minutes before the 01:00 AM UTC pipeline execution window, providing on-call engineers proactive notice of connectivity degradation.

6. Confirm that Cloud Logging captures task-level logs for all DAG tasks and that log entries are queryable by `run_id`, `table_name`, and `execution_date`.

## Phase 6: DAG Integration & Schedule Activation

**Goal:** Wire all tasks into a complete DAG, validate end-to-end flow in a non-production environment, and activate the production schedule.

1. Assemble the full DAG with the following task dependency chain:

   ```
   schema_fingerprint_check
     └─► [parallel] extract_categories, extract_suppliers
           └─► load_gcs_to_stg_categories, load_gcs_to_stg_suppliers
                 └─► merge_stg_to_target_categories, merge_stg_to_target_suppliers
                       └─► [unblocked] extract_products
                             └─► load_gcs_to_stg_products
                                   └─► merge_stg_to_target_products
                                         └─► extract_inventory
                                               └─► load_gcs_to_stg_inventory
                                                     └─► merge_stg_to_target_inventory
                                                           └─► truncate_stg_* (all 4, parallel)
                                                                 └─► reconcile_* (all 4, parallel)
                                                                       └─► write_audit_log_* (all 4, parallel)
   ```

   Rationale for ordering: `stg_products` has FK dependencies on `stg_categories` and `stg_suppliers`; `stg_inventory` has an FK dependency on `stg_products`. MERGE tasks for parent tables must complete before child table extraction begins.

2. Set the DAG schedule to `0 1 * * *` (01:00 AM UTC daily). Set `catchup=False`. Set `max_active_runs=1`.

3. Execute a **dry-run (first-run, full historical load)** against a non-production BigQuery dataset. Verify: all four tables are populated; row counts match the Phase 1 source baselines; `pipeline_audit.pipeline_run_log` contains one `SUCCESS` row per table; all `_stg_<table>` transient tables are empty after truncation.

4. Execute a **simulated incremental run** (second run) by advancing the XCom watermark values for `products` and `inventory`. Verify: only rows with `updated_at` greater than the watermark are extracted for those two tables; `categories` and `suppliers` return full row counts; UPSERT logic correctly updates existing BigQuery rows without creating duplicates.

5. Confirm the **GCS lifecycle policy** (7-day auto-delete) is active on the staging bucket and that Parquet files from the dry-run will be cleaned up within the retention window.

6. Promote the DAG to the production Cloud Composer 2 environment and activate the `0 1 * * *` schedule.
