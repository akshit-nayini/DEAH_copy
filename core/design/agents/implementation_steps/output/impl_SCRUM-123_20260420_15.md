> **These blockers must be resolved before Phase 3 (Ingestion) can begin. No sizing, watermark, or incremental design decisions are final until Blockers 1 and 2 are closed.**

1. **Data Volume Unknown** — Row count and uncompressed size (GB) of `agentichub.employees` have not been confirmed. 2. **Incremental Watermark Column Unconfirmed** — It is unknown whether `employees` contains a reliable `updated_date TIMESTAMP` column with `ON UPDATE CURRENT_TIMESTAMP` semantics, or whether a monotonically increasing `EMPLOYEE_ID` is available as a PK-based watermark fallback. If neither exists, the incremental delta strategy must be redesigned before Phase 5 begins. Resolved in Phase 1, Step 4.

3. **MySQL Binary Logging Status Unconfirmed** — Whether `binlog_format=ROW` and `binlog_row_image=FULL` are enabled on `verizon-data:us-central1:mysql-druid-metadatastore` has not been verified. This is a prerequisite only for a potential future Datastream migration but should be confirmed to fully close that option's feasibility.

## Phase 3: Ingestion Layer — Historical Full Load

**Goal:** Build and deploy the Cloud Run Job that performs a fully paginated extraction of `agentichub.employees` and writes all records to `stg_employees` via the BigQuery Storage Write API.

> **Prerequisite:** Volume profiling (Phase 1, Step 3) must be complete. Page size and memory limits must be set from confirmed volume data before this phase begins.

1. **Implement Cloud Run Job container (Python 3.12).** Configure the Cloud SQL Auth Proxy as a sidecar container targeting `verizon-data:us-central1:mysql-druid-metadatastore`. Use SQLAlchemy 2.x with the PyMySQL driver. Connection string: `mysql+pymysql://sa:{password}@127.0.0.1:3306/agentichub`. Set container memory limit based on the volume profiling result from Phase 1, Step 3.

2. **Implement pre-load schema compatibility check.** Before extracting any data, query MySQL `INFORMATION_SCHEMA.COLUMNS` for `agentichub.employees` and compare against the BigQuery `stg_employees` schema retrieved via the BigQuery REST API. Validate: (a) all expected column names are present in the source, (b) source data types are compatible with the defined BigQuery target types, (c) NOT NULL alignment is consistent, and (d) no new columns exist in the source that are absent from the BigQuery schema. If new columns are detected, write a `SCHEMA_HALT` record to `audit_log` and exit with a non-zero code — do not proceed to extraction.

3. **Implement keyset pagination extraction.** Extract `agentichub.employees` using keyset pagination ordered by `EMPLOYEE_ID`, with initial cursor value of `0`:
   ```sql
   SELECT *
   FROM agentichub.employees
   WHERE EMPLOYEE_ID > :last_seen_id
   ORDER BY EMPLOYEE_ID
   LIMIT :page_size;
   ```
   Advance the cursor to the maximum `EMPLOYEE_ID` of each page. Continue until no rows are returned. Accumulate `source_row_count` across all pages.

4. **Implement null constraint validation.** For each page of records, scan every row for null values in the NOT NULL columns: `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date`. Accumulate `null_violation_count`. Log each violation with its `EMPLOYEE_ID` identifier. Violations are non-fatal — record and continue; they are surfaced in `audit_log`.

5. **Implement type coercion.** Apply the following per-row type mappings before write:

   | MySQL Type | BigQuery Type | Columns |
   |---|---|---|
   | `DECIMAL(10,0)` | `INT64` | `employee_id`, `manager_id`, `department_id` |
   | `DECIMAL` / `NUMERIC` | `NUMERIC` | `salary`, `commission_pct` |
   | `DATETIME` / `TIMESTAMP` | `TIMESTAMP` | `created_date`, `updated_date` |
   | `DATE` | `DATE` | `hire_date` |
   | `VARCHAR` / `CHAR` / `TEXT` | `STRING` | `first_name`, `last_name`, `email`, `phone_number`, `job_id`, `status` |

6. **Write to BigQuery via Storage Write API (COMMITTED stream).** Use the `google-cloud-bigquery-storage` Python client with a COMMITTED stream to write each page of coerced records to `stg_employees`. COMMITTED mode provides atomic, exactly-once write semantics per stream, eliminating duplicate-write risk on job retry.

7. **Execute post-load MERGE for PK uniqueness and upsert enforcement.** After all pages are written, execute a BigQuery MERGE statement to enforce `employee_id` uniqueness:
   ```sql
   MERGE `{project}.{dataset}.stg_employees` T
   USING (
     SELECT * EXCEPT (row_num)
     FROM (
       SELECT *,
         ROW_NUMBER() OVER (
           PARTITION BY employee_id
           ORDER BY updated_date DESC NULLS LAST, created_date DESC
         ) AS row_num
       FROM `{project}.{dataset}.stg_employees`
     )
     WHERE row_num = 1
   ) S
   ON T.employee_id = S.employee_id
   WHEN MATCHED THEN UPDATE SET
     first_name = S.first_name, last_name = S.last_name, email = S.email,
     phone_number = S.phone_number, hire_date = S.hire_date, job_id = S.job_id,
     salary = S.salary, commission_pct = S.commission_pct,
     manager_id = S.manager_id, department_id = S.department_id,
     status = S.status, created_date = S.created_date, updated_date = S.updated_date
   WHEN NOT MATCHED THEN INSERT VALUES (
     S.employee_id, S.first_name, S.last_name, S.email, S.phone_number,
     S.hire_date, S.job_id, S.salary, S.commission_pct,
     S.manager_id, S.department_id, S.status, S.created_date, S.updated_date
   );
   ```

8. **Post-load row count validation.** Execute `SELECT COUNT(*) FROM stg_employees`. Assert that `target_row_count` equals `source_row_count` accumulated during extraction. Compute `row_delta = source_row_count - target_row_count`. Assert delta equals zero. Log the comparison result.

9. **Write historical run record to `audit_log`.** On success, insert one row into `audit_log` with `load_type = 'HISTORICAL'`, `status = 'SUCCESS'`, and all accumulated metrics: `source_row_count`, `target_row_count`, `row_delta`, `schema_diff_result`, `null_violation_count`, `watermark_start` (NULL for historical), `watermark_end` (MAX of `updated_date` or `created_date` from `stg_employees`), `wall_clock_duration_seconds`. On any failure, insert with `status = 'FAILURE'` and the error message in `schema_diff_result`.

10. **Write initial watermark to `run_state`.** On successful historical load completion, insert a row into `run_state` for `table_name = 'employees'` with `last_successful_watermark = MAX(updated_date)` from `stg_employees`. If `updated_date` is NULL for all rows, fall back to `MAX(created_date)`. Set `updated_at = CURRENT_TIMESTAMP()`.

## Phase 4: Orchestration — Cloud Composer DAG

**Goal:** Build the Airflow DAG that gates the historical load sign-off, handles schema-change halts, and controls incremental load activation.

1. **Define DAG skeleton.** Create a Cloud Composer 2 DAG with `dag_id = 'mysql_bq_employees_pipeline'`, `catchup = False`. Set `schedule_interval` to the agreed incremental cadence (TBD pending stakeholder input and volume confirmation from Phase 1).

2. **Implement historical load task.** Add a `CloudRunJobOperator` task (`task_id = 'run_historical_load'`) that triggers the Phase 3 Cloud Run Job. Configure `on_failure_callback` to write a DAG-level `FAILURE` record to `audit_log`.

3. **Implement historical validation SensorOperator gate.** Add a `BigQuerySensor` or `PythonSensor` task (`task_id = 'historical_load_validation_gate'`) that polls `audit_log` for a row where `table_name = 'employees'`, `load_type = 'HISTORICAL'`, `status = 'SUCCESS'`, and `row_delta = 0`. All downstream tasks are blocked until this sensor succeeds.

4. **Implement stakeholder sign-off gate.** Add a `PythonSensor` task (`task_id = 'stakeholder_signoff_gate'`) that polls for a manual sign-off entry in the `pipeline_signoffs` BigQuery table (or an Airflow Variable `employees_historical_signoff = true`). The incremental load task must not execute until this gate clears.

5. **Implement schema-change BranchOperator halt path.** Add a `BranchPythonOperator` task (`task_id = 'schema_change_branch'`) that queries `audit_log` for any `status = 'SCHEMA_HALT'` entry on the most recent run for `table_name = 'employees'`. If detected, branch to a `halt_and_alert` task that sends a notification (PagerDuty or email) and marks the DAG run as failed. Otherwise, proceed to the incremental load task.

6. **Implement incremental load task.** Add a `CloudRunJobOperator` task (`task_id = 'run_incremental_load'`) placed downstream of both the sign-off gate and the schema-change branch. This task triggers the Phase 5 incremental Cloud Run Job. Configure `on_failure_callback` to write a `FAILURE` record to `audit_log`.

7. **Implement DAG-level audit log write tasks.** Add `BigQueryInsertJobOperator` tasks for success and failure audit writes at the orchestration level, distinct from the Cloud Run Job's own writes. These cover orchestration events such as sensor timeouts and task-level retries.

8. **Validate DAG import and graph view.** Deploy the DAG to the Cloud Composer environment. Confirm it imports without parse errors (`airflow dags list`). Confirm task dependencies render correctly in the Airflow UI graph view with the expected gate → branch topology.

## Phase 5: Incremental Load

**Goal:** Build the watermark-based incremental load path that extracts only records changed since the last successful run and upserts them into `stg_employees`.

> **Prerequisite:** Phase 4 sign-off gate must have cleared. Blocker 2 must be resolved. The implementation below assumes `updated_date TIMESTAMP` is confirmed as a reliable watermark column. If only `EMPLOYEE_ID` is available, adapt the watermark to PK-based delta before implementing this phase.

1. **Read watermark from `run_state`.** At job startup, query `run_state` for `table_name = 'employees'` and retrieve `last_successful_watermark`. If no row exists, abort with a non-zero exit code and log an error — the historical load must complete before incremental runs can begin.

2. **Execute watermark-filtered extraction.** Query MySQL for all records where `updated_date > :last_watermark`, using a composite keyset cursor on `(updated_date, EMPLOYEE_ID)` to support pagination without gap or overlap:
   ```sql
   SELECT *
   FROM agentichub.employees
   WHERE
     updated_date > :last_watermark
     AND (updated_date > :last_page_updated_date
          OR (updated_date = :last_page_updated_date AND EMPLOYEE_ID > :last_page_id))
   ORDER BY updated_date ASC, EMPLOYEE_ID ASC
   LIMIT :page_size;
   ```
   Accumulate `source_row_count` across all pages.

3. **Re-apply pre-load schema compatibility check.** Execute the same schema compatibility check as Phase 3, Step 2. If new columns are detected, write `SCHEMA_HALT` to `audit_log` and exit non-zero. The Cloud Composer `schema_change_branch` task will route to the halt-and-alert path.

4. **Re-apply null validation, type coercion, and Storage Write API write.** Apply the same null constraint validation, type coercion mappings, and COMMITTED-stream write logic as Phase 3, Steps 4–6. Write the incremental batch to `stg_employees`.

5. **Execute MERGE for upsert and PK dedup.** Execute the same BigQuery MERGE statement as Phase 3, Step 7, scoped to this incremental batch. Existing employees are updated in place; new employees are inserted. PK uniqueness is enforced after every run.

6. **Post-load row count validation.** Assert that the number of records written to BigQuery equals `source_row_count` for this delta window. Compute and log `row_delta` in `audit_log`.

7. **Update `run_state` watermark.** On successful incremental run, update `run_state` for `table_name = 'employees'`: set `last_successful_watermark = MAX(updated_date)` from the current batch, `last_successful_run_id` to the current `run_id`, `updated_at = CURRENT_TIMESTAMP()`.

8. **Write incremental audit log record.** Insert a row into `audit_log` with `load_type = 'INCREMENTAL'`, `status`, all metrics, and the updated watermark values.

### TC-01 — MySQL Source Connectivity

### TC-02 — Historical Full Load: Row Count Integrity

- **What is being tested:** All rows present in `agentichub.employees` at extraction time are present in `stg_employees` after the historical load, satisfying the post-load row count validation constraint.
- **Input condition:** Execute historical Cloud Run Job against the full `employees` table.
- **Expected outcome:** `source_row_count` (MySQL `COUNT(*)`) equals `target_row_count` (BigQuery `COUNT(*)`); `row_delta = 0`; `audit_log` contains one row with `load_type = 'HISTORICAL'` and `status = 'SUCCESS'`.

---

### TC-03 — Primary Key Uniqueness Post-MERGE

- **What is being tested:** No duplicate `employee_id` values exist in `stg_employees` after the MERGE step, enforcing the PK uniqueness constraint.
- **Input condition:** Historical load completes and MERGE executes. Query BigQuery: `SELECT employee_id, COUNT(*) AS c FROM stg_employees GROUP BY employee_id HAVING c > 1`.
- **Expected outcome:** Query returns zero rows.

---

### TC-04 — NOT NULL Column Violation Detection and Logging

- **What is being tested:** Records with null values in NOT NULL columns are detected, counted, and logged without aborting the run.
- **Input condition:** Inject a synthetic test record into the extraction batch with `first_name = NULL` before the write step.
- **Expected outcome:** `null_violation_count >= 1` in `audit_log`; the violation is logged with the offending `employee_id`; the run completes with `status = 'SUCCESS'`; the record is not silently discarded.

### TC-05 — Type Coercion: MySQL DECIMAL(10,0) → BigQuery INT64

- **What is being tested:** `EMPLOYEE_ID` values stored as `DECIMAL(10,0)` in MySQL are correctly cast to `INT64` in BigQuery with no precision loss.
- **Input condition:** Source row with `EMPLOYEE_ID = 12345` (decimal representation).
- **Expected outcome:** `stg_employees.employee_id` stores `12345` as `INT64`; no coercion error; no write failure.

---

### TC-06 — Type Coercion: MySQL DATETIME/TIMESTAMP → BigQuery TIMESTAMP

- **What is being tested:** `CREATED_DATE` and `UPDATED_DATE` MySQL values are correctly coerced to BigQuery `TIMESTAMP` type.
- **Input condition:** Source row with `CREATED_DATE = '2024-03-15 08:30:00'`.
- **Expected outcome:** `stg_employees.created_date` stores `2024-03-15T08:30:00Z` as a valid BigQuery `TIMESTAMP`; no coercion error.

### TC-07 — Partitioning: hire_date Monthly Partition Assignment

- **What is being tested:** `stg_employees` correctly assigns records to monthly partitions based on `hire_date`.
- **Input condition:** Insert three records with `hire_date` values in distinct calendar months: `2023-01-15`, `2023-02-20`, `2023-03-10`.
- **Expected outcome:** `INFORMATION_SCHEMA.PARTITIONS` for `stg_employees` shows three distinct monthly partition IDs (`20230101`, `20230201`, `20230301`); each record resides in its correct partition.

### TC-08 — Schema Change Detection: Halt-and-Alert

- **What is being tested:** A new column in the MySQL source not present in the BigQuery schema triggers a `SCHEMA_HALT` and stops the write — satisfying the schema compatibility constraint.
- **Input condition:** Mock the schema compatibility check response to simulate a new column (e.g., `MIDDLE_NAME VARCHAR(50)`) present in MySQL but absent from `stg_employees`. Trigger the Cloud Run Job.
- **Expected outcome:** Job exits with a non-zero code; `audit_log` records `status = 'SCHEMA_HALT'` with `new_columns_detected = 'MIDDLE_NAME'`; no records are written to `stg_employees`; Cloud Composer DAG branches to `halt_and_alert` task.

---

### TC-09 — Incremental Load: Watermark Correctness (No Gap, No Overlap)

- **What is being tested:** Incremental load extracts only records with `updated_date` greater than the last successful watermark and does not re-ingest unchanged records.
- **Input condition:** Historical load completes with `run_state.last_successful_watermark = T0`. Update 3 rows in `agentichub.employees` setting `updated_date = T1` where `T1 > T0`. Leave all other rows unchanged. Trigger incremental load.
- **Expected outcome:** `source_row_count = 3` for the incremental run; only those 3 rows are written and merged; `run_state.last_successful_watermark` is updated to `T1`; `audit_log` records `load_type = 'INCREMENTAL'` and `status = 'SUCCESS'`.

### TC-10 — Incremental Load: Upsert Correctness (No Duplicate Insert)

- **What is being tested:** An updated record in MySQL is correctly reflected in `stg_employees` after an incremental load without creating a duplicate row.
- **Input condition:** After historical load, update `salary` for `employee_id = 100` in MySQL. Trigger incremental load.
- **Expected outcome:** `stg_employees` contains exactly one row for `employee_id = 100`; `salary` reflects the updated value; `SELECT COUNT(*) FROM stg_employees WHERE employee_id = 100` returns `1`.

### TC-11 — Self-Referencing FK: Root Manager Carries NULL manager_id

## Phase 8: Handover

**Goal:** Transfer ownership to the operations team with a complete runbook, documented procedures, risks, and escalation paths, and collect formal sign-off.

1. **Update pipeline design summary in runbook.** Document the following in the project runbook for SCRUM-123:
   - **Architecture:** Cloud Composer 2 + Cloud Run Jobs (Python 3.12, SQLAlchemy 2.x, PyMySQL, Cloud SQL Auth Proxy sidecar).
   - **Source:** MySQL `agentichub.employees` at `34.70.79.163:3306` (instance: `verizon-data:us-central1:mysql-druid-metadatastore`).
   - **Target:** BigQuery `stg_employees` in `verizon_data_deah`; partitioned by `hire_date` (MONTH); clustered by `[status, department_id, job_id]`.
   - **Load types:** Historical (keyset-paginated full load); Incremental (watermark-based delta on `updated_date`, fallback to `created_date`).
   - **Audit infrastructure:** `audit_log` and `run_state` tables in `verizon_data_deah`.
   - **Orchestration:** DAG `mysql_bq_employees_pipeline` in Cloud Composer; gated by historical validation sensor and stakeholder sign-off sensor before incremental activation; schema-change BranchOperator halt path active on every run.

2. **Document operational procedures.** Record the following in the runbook:
   - **Manual historical reload:** Truncate `stg_employees`, delete the `employees` row from `run_state`, and re-run the `run_historical_load` DAG task. Re-obtain stakeholder sign-off before the incremental gate will clear.
   - **Watermark reset:** Update `run_state.last_successful_watermark` directly in BigQuery. Document the risk of gap (missed records) or overlap (duplicate processing) and require a PK dedup MERGE after any manual reset.
   - **Clearing the sign-off gate:** Set Airflow Variable `employees_historical_signoff = true` or insert a row into `pipeline_signoffs` with the signer's name, role, and timestamp.
   - **Resolving a SCHEMA_HALT:** Review `new_columns_detected` in the most recent `audit_log` entry. Update the BigQuery `stg_employees` schema and the Cloud Run Job type-coercion mapping to accommodate the new column. Redeploy the container image. Re-trigger the DAG run.
   - **Reviewing pipeline health:** Direct stakeholders to the Looker Studio dashboard URL. For on-call triage, query: `SELECT * FROM verizon_data_deah.audit_log WHERE table_name = 'employees' ORDER BY run_timestamp DESC LIMIT 10`.

3. **Document known risks.** Record the following three risks in the runbook:
   - Cloud Run Jobs cannot auto-distribute processing across nodes. If `employees` grows to hundreds of millions of rows, the container may require a memory increase and keyset sharding refactor, or a migration to Dataflow. The migration path is defined in the architecture decision record.
   - If `updated_date` proves unreliable in the source (missing or not maintained by the application), the incremental strategy must be redesigned. Datastream CDC is the recommended fallback and requires `binlog_format=ROW` and `binlog_row_image=FULL` on the Cloud SQL instance.
   - The monitoring alerting evaluation period is provisionally set to 30 minutes. A formal monitoring SLA must be agreed with stakeholders and the Cloud Monitoring policy updated to reflect it. Until that SLA is confirmed, the 30-minute window should be treated as a temporary default.

4. **Document escalation paths.** Record the following in the runbook:
   - **Pipeline failure (non-zero Cloud Run exit code):** On-call engineer receives PagerDuty/email alert. Triage via Cloud Logging structured logs filtered on `run_id`. Query `audit_log` for the most recent `status = 'FAILURE'` entry and inspect the error field.
   - **Schema halt:** Airflow `halt_and_alert` task delivers notification. Escalate to the data engineering team to assess the new column in `audit_log.new_columns_detected` and update the pipeline before re-triggering.
   - **Row count delta non-zero:** Investigate `audit_log.row_delta` for the affected run. Determine whether a partial re-run (incremental) or full reload (historical) is required. Escalate to the data engineering team.
   - **Watermark gap detected:** Audit `run_state.last_successful_watermark` against expected cadence. Escalate to the data engineering team to determine whether a targeted re-run or full reload is required.
   - **Persistent or undiagnosed failures:** Escalate to the senior data engineer on duty. Reference the architecture decision record for rollback options and the Datastream migration path.

5. **Collect formal sign-off.** Obtain written sign-off from the following parties and record each in the `pipeline_signoffs` BigQuery table with fields `signer_name`, `role`, `sign_off_type`, and `signed_at`:
   - **Data Engineering lead** — confirming the pipeline implementation matches the architecture and data model specifications for SCRUM-123.
   - **Yash (stakeholder)** — confirming the Looker Studio dashboard meets observability requirements and that the historical load result has been reviewed and approved.
   - **Operations team** — confirming runbook completeness, alert delivery validation, and on-call readiness.

   The pipeline is not considered live until all three sign-offs are recorded in `pipeline_signoffs`.
