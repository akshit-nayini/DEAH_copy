## ⚠️ Open Blockers

The following items are unresolved and **block development work in the phases indicated**. No implementation step may proceed past its dependency gate until the blocker is formally closed.

1. **[Blocks Phase 1 Step 5 & Phase 3]** — Network path from GCP Dataflow workers to `orders-db.corp.com` is undefined. The answer (Cloud VPN HA, Cloud Interconnect, or mutual-TLS over public internet) determines Dataflow worker network configuration, JDBC connection string parameters, and the security review scope for the batch ingestion path.

2. **[Blocks Phase 1 Step 4 & Phase 3]** — Authoritative schema for Pub/Sub `orders.events` messages is unconfirmed (JSON with fixed schema, Avro with schema registry entry, or Protobuf). A formal schema contract is required before streaming pipeline development can begin.

3. **[Blocks Phase 3 Step 1]** — Acceptable deduplication semantics for the streaming path are undefined: best-effort at-least-once with downstream MERGE correction, or exactly-once guaranteed at ingest time via Storage Write API. This choice affects Dataflow pipeline design and BigQuery write quota consumption.

## ⚠️ Risks & Mitigations *(reference only — not implementation steps)*

| Risk | Mitigation Summary |
|------|--------------------|
| **Network Connectivity** | Provision Cloud VPN HA (two tunnels, two Cloud Routers, 99.99% SLA) or Cloud Interconnect to corporate DC. Configure Composer DAG retry: 3 retries, 15-minute exponential backoff. PagerDuty/email alert on DAG failure within 15 minutes of scheduled start. |
| **Deduplication Correctness** | Storage Write API exactly-once mode for streaming inserts. BigQuery MERGE keyed on `order_id` (orders), `item_id` (order_items) in batch path. Daily post-reconciliation Cloud Monitoring alert if duplicate primary key count exceeds zero. |
| **Schema Evolution** | Register `orders.events` schema in Pub/Sub Schema Registry (Avro or Protobuf). Route Dataflow parse errors to dead-letter topic. Store BigQuery schema contract as JSON in GCS; run schema drift check in Composer DAG before each batch extraction; alert on any deviation. |

## Phase 3: Streaming Ingestion Pipeline (Pub/Sub → `stg_order_events`)

*Prerequisite: Open Blockers 2 and 3 resolved; Phase 1 Steps 4–5 complete.*

1. **Implement the Dataflow streaming Apache Beam pipeline:**
   - **Source:** `PubSubIO.Read` from topic `orders.events` (project `verizon-data`), reading message attributes and payload.
   - **Parse & map:** Deserialise each message using the registered schema (Avro/Protobuf/JSON per Blocker 2 resolution). Map all payload fields to the `stg_order_events` column schema.
   - **Conditional timestamp population:** Based on `order_status` value, populate exactly one of `shipped_at`, `delivered_at`, or `cancelled_at`; set the other two to `NULL`. For `order_placed`, all three are `NULL`.
   - **Schema validation:** Validate required fields (`order_id`, `user_id`, `order_status`, `total_amount`, `currency`, `order_date`, `created_at`, `updated_at`) are non-null. Route any record failing validation to the designated dead-letter Pub/Sub topic without retrying.
   - **Per-key stateful deduplication:** Apply Beam State and Timer APIs keyed on `order_id` to suppress duplicate events within the deduplication window (confirm window duration with stakeholder; not specified in inputs).
   - **Sink:** Write deduplicated, validated records to `stg_order_events` via the BigQuery Storage Write API. Use exactly-once mode (confirm against Blocker 3 resolution before finalising).
   - **Audit append:** On each pipeline checkpoint, append a row to `pipeline_audit.pipeline_run_log` with `pipeline_name = 'streaming_order_events'`, `rows_processed`, and `status`.

2. **Configure dead-letter topic and subscription** — Create a separate Pub/Sub topic for streaming pipeline failures. Ensure the dead-letter subscription retains messages for at least 7 days. Set a Cloud Monitoring alert to fire when dead-letter message count exceeds 0.

3. **Deploy as an always-on Dataflow streaming job** — Enable autoscaling. Set the Dataflow worker network configuration per the confirmed network path (Open Blocker 1). Tag the job with label `pipeline=streaming-order-events` for monitoring and cost attribution.

4. **Validate sub-60-second end-to-end latency** — Publish a test message to `orders.events` and measure elapsed time to row availability in `stg_order_events` via the BigQuery streaming buffer. Confirm latency is under 60 seconds.

## Phase 4: Batch Ingestion Pipeline (PostgreSQL → `stg_orders`, `stg_order_items`)

*Prerequisite: Open Blocker 1 resolved; Phase 1 Steps 3 and 6 complete; Phase 2 complete.*

1. **Implement the watermark state table** — Create a GCS object or dedicated BigQuery table (`pipeline_audit.batch_watermarks`) to persist the last-successful `updated_at` watermark per source table (`orders`, `order_items`). Initialise both to a historical baseline timestamp that will capture the full existing dataset on first run.

2. **Implement the Dataflow batch Apache Beam pipeline for `orders`:**
   - **Source:** `JdbcIO.Read` from `orders-db.corp.com:5432`, database `orders_db`, username `sa`. Use the JDBC connection string confirmed in Phase 1 Step 6 (including SSL/TLS parameters if required).
   - **Incremental extraction:** Apply a `WHERE updated_at > :last_watermark` predicate, passing the current watermark from `pipeline_audit.batch_watermarks` as a runtime parameter.
   - **Map:** Transform source rows to the `stg_orders` column schema.
   - **Sink:** Write extracted rows to a GCS staging location, then execute a BigQuery `MERGE INTO stg_orders USING <staged_data> ON stg_orders.order_id = staged.order_id` statement to upsert records and enforce deduplication by `order_id`.
   - **Watermark update:** On successful completion, update `pipeline_audit.batch_watermarks` for `orders` to `MAX(updated_at)` from the extracted batch.
   - **Audit append:** Append a row to `pipeline_audit.pipeline_run_log` with `pipeline_name = 'batch_orders'`, `rows_processed`, and `status`.

3. **Implement the Dataflow batch Apache Beam pipeline for `order_items`:**
   - Identical pattern to Step 2, targeting `public.order_items`.
   - MERGE key: `item_id`.
   - Watermark column: `updated_at`.
   - Target table: `stg_order_items`.
   - Audit entry: `pipeline_name = 'batch_order_items'`.

4. **Implement late-arrival reconciliation** — After both MERGE operations complete, execute a reconciliation query that cross-references `stg_order_events.order_id` against `stg_orders.order_id` to surface any events whose parent order row is absent. Log reconciliation results to `pipeline_audit.pipeline_run_log` with `pipeline_name = 'reconciliation_check'`.

5. **Use GCS as Dataflow temporary and staging storage** — Configure the Dataflow batch job `--temp_location` and `--staging_location` to point to the designated GCS bucket. Confirm the Dataflow worker service account has `roles/storage.objectAdmin` on that bucket.

6. **Parameterise the pipeline** — Accept `--watermark_orders` and `--watermark_order_items` as runtime arguments so the Composer DAG can inject the correct watermark timestamps at trigger time without modifying pipeline code.

## Phase 5: Cloud Composer 2 Orchestration

1. **Implement the daily batch DAG** — Create a Cloud Composer 2 DAG with schedule `0 2 * * *` (02:00 UTC). Structure task dependencies in the following order:

   ```
   schema_drift_check
       → trigger_batch_orders
       → trigger_batch_order_items
       → verify_merge_completeness
       → post_reconciliation_dedup_check
   ```

2. **`schema_drift_check` task** — Read the current PostgreSQL schema for `public.orders` and `public.order_items` (via a JdbcOperator or PythonOperator with psycopg2). Compare against the GCS-stored schema contract JSON from Phase 2 Step 5. If any column name, type, or nullability differs, fail the DAG immediately and publish a Cloud Monitoring alert. Do not proceed to extraction until the drift is resolved.

3. **`trigger_batch_orders` task** — Read the current watermark for `orders` from `pipeline_audit.batch_watermarks`. Trigger the Dataflow batch pipeline (Phase 4 Step 2) via `DataflowStartFlexTemplateOperator`, passing `--watermark_orders` as a runtime parameter. Wait for job completion before advancing.

4. **`trigger_batch_order_items` task** — Identical pattern to Step 3, targeting the `order_items` pipeline with `--watermark_order_items`. Run after `trigger_batch_orders` completes to respect FK load order.

5. **`verify_merge_completeness` task** — Query `pipeline_audit.pipeline_run_log` to confirm `batch_orders` and `batch_order_items` runs for today's date have `status = 'SUCCESS'` and `rows_processed > 0`. Fail the DAG if either condition is not met.

6. **`post_reconciliation_dedup_check` task** — Execute the following BigQuery queries and publish a Cloud Monitoring alert if either returns a count greater than zero:
   - `SELECT COUNT(*) FROM stg_orders GROUP BY order_id HAVING COUNT(*) > 1`
   - `SELECT COUNT(*) FROM stg_order_items GROUP BY item_id HAVING COUNT(*) > 1`

7. **Configure DAG retry and SLA policy:**
   - Retries: 3, retry delay: 15 minutes, retry exponential backoff enabled.
   - SLA miss callback: if the DAG has not reached `success` state by 04:00 UTC, trigger a PagerDuty/email alert.
   - On any task failure after all retries are exhausted, send an alert within 15 minutes of the scheduled start time.

## Phase 6: Monitoring & Alerting

1. **Create Cloud Monitoring dashboards** with the following widgets:
   - *Streaming:* Pub/Sub oldest undelivered message age (`orders.events`), Dataflow throughput (elements/sec), BigQuery Storage Write API latency, dead-letter topic message count.
   - *Batch:* Composer DAG last run status and duration, rows processed per batch run (from `pipeline_audit.pipeline_run_log`), BigQuery MERGE statement duration.

2. **Configure Cloud Monitoring alerting policies:**
   - Dead-letter topic message count > 0 → alert within 5 minutes.
   - Streaming pipeline end-to-end data freshness > 60 seconds → alert.
   - Composer DAG not in `success` state by 04:00 UTC → SLA breach alert.
   - Duplicate primary key count > 0 (post-reconciliation dedup check) → alert.
   - Schema drift detected during pre-flight check → alert with diff detail.
   - Dataflow streaming job not running → alert within 5 minutes of detection.

3. **Configure Cloud Logging sinks** — Route Dataflow job logs (`resource.type="dataflow_job"`) and Cloud Composer task logs (`resource.type="cloud_composer_environment"`) to a centralised GCS log archive bucket. Set a 30-day retention policy on the bucket.

4. **Validate `pipeline_audit.pipeline_run_log` population** — Trigger a manual test run of the streaming pipeline and a manual invocation of the batch DAG. Confirm that both runs produce correctly populated rows in `pipeline_audit.pipeline_run_log` with accurate `start_time`, `end_time`, `rows_processed`, and `status` values.
