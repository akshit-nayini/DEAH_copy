## ⚠️ Open Blockers

> **These blockers must be resolved before the phases indicated. Do not proceed past the gate noted for each blocker.**

1. **[BLOCKER-1] Cloud SQL vs On-Premises Host** *(Gate: Phase 1, Step 1)*
   The `instance_connection_name` (`verizon-data:us-central1:mysql-druid-metadatastore`) implies Cloud SQL hosting, but the bare IP `34.70.79.163` has not been confirmed as a Cloud SQL private or public IP. If the host is on-premises, Cloud SQL Auth Proxy is not a valid connectivity path and VPC peering or Cloud Interconnect must be scoped instead. **No Dataflow pipeline work may proceed until connectivity approach is confirmed.**

2. **[BLOCKER-2] Full Load vs Incremental Ingestion Strategy** *(Gate: Phase 3, Step 2)*
   The current architecture assumes hourly full `WRITE_TRUNCATE`. If an incremental watermark-based or CDC strategy is required, a reliable high-watermark column (e.g., `updated_date`) must be confirmed on `agentichub.EMPLOYEES`, or a CDC tool (e.g., Debezium) must be introduced. The pipeline read query and BigQuery write disposition in Phase 3 depend on this answer.

3. **[BLOCKER-3] Target BigQuery Coordinates** *(Gate: Phase 2, Step 1)*
   The target BigQuery project ID, dataset name, and table naming convention have not been confirmed by the Customer360 data platform team. BigQuery dataset and table creation cannot proceed until these are known.

## Phase 3: Dataflow Pipeline Development

> **Prerequisite:** [BLOCKER-2] must be resolved before Step 2. The query and write disposition below assume full `WRITE_TRUNCATE`; adjust if incremental strategy is confirmed.

1. **Beam pipeline project structure:** Create an Apache Beam 2.x pipeline using a Dataflow Flex Template project layout. Implement the pipeline to accept the following runtime parameters:
   - `--jdbc_url` (resolved from Secret Manager at runtime)
   - `--jdbc_username` (resolved from Secret Manager at runtime)
   - `--jdbc_password` (resolved from Secret Manager at runtime)
   - `--bq_project`
   - `--bq_dataset`
   - `--bq_table` (default: `stg_employees`)
   - `--gcs_temp_location`
   - `--gcs_dead_letter_path`

2. **JDBC source transform:** Implement a `ReadFromJdbc` transform using the Beam JDBC IO connector:
   - **Driver:** `com.mysql.cj.jdbc.Driver`
   - **Connection URL:** `jdbc:mysql://127.0.0.1:3306/agentichub` (via Auth Proxy sidecar)
   - **Query:**
     ```sql
     SELECT
       EMPLOYEE_ID, FIRST_NAME, LAST_NAME, EMAIL, PHONE_NUMBER,
       HIRE_DATE, JOB_ID, SALARY, COMMISSION_PCT, MANAGER_ID,
       DEPARTMENT_ID, STATUS, CREATED_DATE, UPDATED_DATE
     FROM EMPLOYEES
     ```
   - Set fetch size to a value that avoids OOM on full-table reads; document the chosen value in pipeline comments.

3. **Schema mapping and type casting transform:** Implement a `DoFn` that maps each source row to the `stg_employees` target schema:
   - `EMPLOYEE_ID` (MySQL INT) → `employee_id` (INT64)
   - `FIRST_NAME`, `LAST_NAME`, `EMAIL`, `PHONE_NUMBER`, `JOB_ID`, `STATUS` → STRING; apply `TRIM()`; coerce empty string to NULL for NULLABLE fields
   - `HIRE_DATE` (MySQL DATE) → `hire_date` (DATE)
   - `SALARY`, `COMMISSION_PCT` (MySQL DECIMAL) → NUMERIC; preserve two decimal places
   - `MANAGER_ID`, `DEPARTMENT_ID` → INT64; allow NULL
   - `CREATED_DATE`, `UPDATED_DATE` (MySQL DATETIME/TIMESTAMP) → TIMESTAMP; normalise to UTC
   - **NOT NULL enforcement:** Reject and route to dead-letter any row where `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, or `created_date` is NULL. Include column name and rejection reason in the dead-letter record.

4. **Dead-letter sink:** Route all rejected rows to GCS:
   `gs://<bucket>/customer360/dead_letter/stg_employees/`
   Each dead-letter record must include the original source row (as JSON) and a `rejection_reason` string identifying the failing column and constraint.

5. **BigQuery sink transform:** Implement `WriteToBigQuery` with:
   - `write_disposition=WRITE_TRUNCATE`
   - `create_disposition=CREATE_NEVER`
   - Target: `<project>.<dataset>.stg_employees`
   - Use the BigQuery Storage Write API for throughput efficiency.

6. **Dataflow Flex Template packaging:** Package the pipeline as a Dataflow Flex Template. Store the template spec file at:
   `gs://<bucket>/customer360/templates/stg_employees_template.json`
   Document the full `gcloud dataflow flex-template run` launch command with all required runtime parameters in the pipeline repository `README`.

7. **Local unit test execution:** Run unit tests for the schema mapping and type casting `DoFn` (Phase 5, cases 5.2 and 5.3) against the pipeline codebase before building the Flex Template container image. All unit tests must pass before proceeding to Phase 4.

## Phase 4: Cloud Composer DAG Development

1. **DAG file creation:** Create an Airflow 2.x DAG named `customer360_stg_employees`:
   - **Schedule:** `0 * * * *` (hourly)
   - **`catchup`:** `False`
   - **`max_active_runs`:** `1` (prevent overlapping hourly runs)
   - **`default_args`:** include `retries=1`, `retry_delay=timedelta(minutes=5)`, `on_failure_callback` pointing to the alerting function defined in Step 6.

2. **Pre-flight schema check task:** Implement a `PythonOperator` (or `@task`-decorated callable) named `schema_check`:
   - Connect to `agentichub` MySQL at `34.70.79.163:3306` via Auth Proxy using credentials from Secret Manager.
   - Query `INFORMATION_SCHEMA.COLUMNS` for `agentichub.EMPLOYEES` (column names and data types).
   - Download the baseline snapshot from `gs://<bucket>/customer360/schema_snapshots/EMPLOYEES_snapshot.json`.
   - Compute a hash of the live schema column list and compare against the stored snapshot hash.
   - **On delta detected:** raise `AirflowException` with a message describing the changed columns. Do not proceed to the Dataflow task. Ensure the Cloud Logging alert (Step 6) fires.
   - **On match:** push the snapshot hash to XCom and allow downstream tasks to proceed.

3. **Dataflow trigger task:** Implement a `DataflowStartFlexTemplateOperator` named `dataflow_trigger`:
   - Template location: `gs://<bucket>/customer360/templates/stg_employees_template.json`
   - Runtime parameters: `bq_project`, `bq_dataset`, `bq_table=stg_employees`, `gcs_temp_location`, `gcs_dead_letter_path`; JDBC credentials resolved from Secret Manager via the Dataflow service account.
   - `wait_until_finish=True` — downstream tasks only proceed on confirmed Dataflow job `JOB_STATE_DONE`.

4. **Post-run row count validation task:** Implement a `BigQueryValueCheckOperator` named `row_count_check`:
   - SQL: `SELECT COUNT(*) FROM <project>.<dataset>.stg_employees`
   - Compare result against the baseline row count captured in Phase 1, Step 1 (stored in an Airflow Variable or GCS file).
   - If result is `0`: log a structured warning to Cloud Logging and trigger the alert callback, but do not fail the DAG (empty source is a possible legitimate state; confirm handling with product owner if desired behaviour differs).

5. **DAG dependency wiring:**
   ```
   schema_check >> dataflow_trigger >> row_count_check
   ```

6. **Monitoring and alerting hooks:** Implement an `on_failure_callback` function that publishes a structured log entry to Cloud Logging on any task failure. The log entry must include: DAG ID, task ID, logical run date, execution date, and exception message. Apply this callback at DAG level so all tasks inherit it.

7. **Airflow Variables and Connections documentation:** Document (do not automate via DAG code) the following prerequisites that must be manually configured in the Composer 2 environment before the first DAG run:
   - **Airflow Variables:** `bq_project`, `bq_dataset`, `gcs_temp_location`, `gcs_dead_letter_path`, `schema_snapshot_gcs_path`, `employees_baseline_row_count`
   - **Airflow Connections:** `mysql_agentichub` (MySQL connection to `agentichub` via Auth Proxy), `google_cloud_default` (GCP service account for BigQuery and GCS operations)

## Phase 5: Testing

### 5.1 Connectivity Test
- **What is being tested:** End-to-end JDBC connectivity from a Dataflow test worker to `agentichub.EMPLOYEES` via Cloud SQL Auth Proxy.
- **Input condition:** Dataflow Flex Template test job launched in Dev with `--cloud-sql-instances verizon-data:us-central1:mysql-druid-metadatastore`; JDBC URL `jdbc:mysql://127.0.0.1:3306/agentichub`; credentials from Secret Manager.
- **Expected outcome:** `SELECT COUNT(*) FROM EMPLOYEES` returns a non-zero integer with no TCP timeout or authentication error. Result is logged to Cloud Logging.

### 5.2 Schema Mapping — Happy Path (All Columns)
- **What is being tested:** All 14 source columns from `agentichub.EMPLOYEES` are correctly mapped and cast to the `stg_employees` target schema.
- **Input condition:** A representative set of MySQL rows containing values for all columns, including NULL-allowed fields (`phone_number`, `commission_pct`, `manager_id`, `department_id`, `updated_date`). Include a row where `phone_number = '+1-800-555-0199'` to validate string preservation.
- **Expected outcome:** Output BigQuery rows match the target schema exactly. NOT NULL columns contain non-null values. `salary` and `commission_pct` retain two decimal places. `created_date` and `updated_date` are UTC-normalised TIMESTAMPs. `phone_number` retains the value `+1-800-555-0199` without truncation or reformatting. Dead-letter path contains zero files.

### 5.3 NOT NULL Constraint Enforcement — Rejected Rows to Dead-Letter
- **What is being tested:** Rows with NULL values in required (NOT NULL) fields are rejected and routed to the dead-letter sink; no rejected row appears in BigQuery.
- **Input condition:** Nine separate test rows, each with one of the following fields set to NULL: `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date`. All other fields are valid.
- **Expected outcome:** All nine rows appear in `gs://<bucket>/customer360/dead_letter/stg_employees/` as JSON objects containing the original row and a `rejection_reason` field identifying the null column. Zero of these rows appear in `stg_employees` in BigQuery.

### 5.4 Self-Referencing Foreign Key Integrity
- **What is being tested:** `manager_id` values referencing `employee_id` within the same table are preserved; top-of-hierarchy employees with `manager_id = NULL` load without error.
- **Input condition:** A batch containing: one root employee (`manager_id = NULL`), two employees whose `manager_id` references the root employee's `employee_id`, and one employee whose `manager_id` references a non-existent `employee_id` (orphan reference — present in source, to confirm the pipeline does not perform FK validation).
- **Expected outcome:** All four rows are loaded to `stg_employees`. `manager_id` values are identical to source values. The orphaned `manager_id` is loaded as-is without rejection (FK enforcement is not a pipeline responsibility). Dead-letter path contains zero files.

### 5.5 Partitioning Correctness
- **What is being tested:** Rows are written to the correct DAY partition based on `created_date`.
- **Input condition:** Input rows with `created_date` values spanning exactly three distinct calendar days (e.g., `2025-01-01`, `2025-01-02`, `2025-01-03`), with at least two rows per day.
- **Expected outcome:** `SELECT DATE(_PARTITIONTIME), COUNT(*) FROM stg_employees GROUP BY 1` returns exactly three rows, one per calendar day, with per-day counts matching the input distribution.

### 5.6 Clustering Metadata Verification
- **What is being tested:** `stg_employees` in BigQuery is clustered on `department_id`, `job_id`, `status` in the correct order.
- **Input condition:** Inspect BigQuery table metadata after the first successful full load.
- **Expected outcome:** `SELECT clustering_fields FROM <project>.<dataset>.INFORMATION_SCHEMA.TABLES WHERE table_name = 'stg_employees'` returns `["department_id", "job_id", "status"]` in that exact order.

### 5.7 Pre-Flight Schema Check — Schema Drift Detected
- **What is being tested:** The Composer DAG `schema_check` task halts the pipeline when a column addition is detected in the source schema.
- **Input condition:** The GCS snapshot at `EMPLOYEES_snapshot.json` contains the original 14-column schema. The live MySQL `INFORMATION_SCHEMA.COLUMNS` for `agentichub.EMPLOYEES` is updated (simulated in Dev) to include an additional column `MIDDLE_NAME VARCHAR(50)`.
- **Expected outcome:** The `schema_check` task raises an `AirflowException` and enters a `failed` state. The `dataflow_trigger` task does not execute. A structured error log entry is written to Cloud Logging. `stg_employees` in BigQuery is unchanged from its pre-run state.

### 5.8 Pre-Flight Schema Check — No Drift Pass-Through
- **What is being tested:** The `schema_check` task succeeds and allows the pipeline to proceed when live schema matches the snapshot.
- **Input condition:** Live MySQL `INFORMATION_SCHEMA.COLUMNS` for `agentichub.EMPLOYEES` is identical (same columns, same types) to the GCS snapshot.
- **Expected outcome:** `schema_check` task completes in `success` state, writes the snapshot hash to XCom, and the `dataflow_trigger` task is triggered.

### 5.9 Post-Run Row Count Validation — Zero Row Warning
- **What is being tested:** The `row_count_check` task alerts when `stg_employees` contains zero rows after a Dataflow run.
- **Input condition:** Dataflow run completes with `WRITE_TRUNCATE` against a MySQL `EMPLOYEES` table that contains zero rows.
- **Expected outcome:** `row_count_check` task triggers the `on_failure_callback` alerting function and writes a structured warning to Cloud Logging. The DAG run itself does not enter a hard `failed` state.

### 5.10 End-to-End DAG Run — Successful Full Load
- **What is being tested:** A complete hourly DAG run from schema check through Dataflow execution to row count validation, producing a correctly loaded and partitioned BigQuery table.
- **Input condition:** MySQL `agentichub.EMPLOYEES` contains at least 10 rows with varied `department_id`, `job_id`, `status`, and `created_date` values spanning two distinct calendar days. All rows satisfy NOT NULL constraints.
- **Expected outcome:** All three DAG tasks (`schema_check`, `dataflow_trigger`, `row_count_check`) complete in `success` state. `stg_employees` row count equals the source row count. Two distinct partitions are visible in BigQuery. Dead-letter GCS path contains zero files. Cloud Logging contains a structured success log entry for the run.

## Phase 6: Handover

1. **Pipeline design summary:** Record the following in the runbook:
   - **Pipeline name:** Customer360 — `stg_employees` Hourly Batch Ingestion
   - **Ticket:** SCRUM-5
   - **Ingestion pattern:** Batch; hourly full `WRITE_TRUNCATE`
   - **Source:** MySQL `agentichub.EMPLOYEES` at `34.70.79.163:3306` via Cloud SQL Auth Proxy (instance: `verizon-data:us-central1:mysql-druid-metadatastore`)
   - **Target:** BigQuery `<project>.<dataset>.stg_employees` — `DAY`-partitioned on `created_date`; clustered on `department_id`, `job_id`, `status`
   - **Orchestrator:** Cloud Composer 2; DAG `customer360_stg_employees`; schedule `0 * * * *`
   - **Processing:** Dataflow Flex Template at `gs://<bucket>/customer360/templates/stg_employees_template.json` (Apache Beam 2.x)
   - **Connectivity:** Cloud SQL Auth Proxy sidecar on Dataflow workers
   - **Dead-letter path:** `gs://<bucket>/customer360/dead_letter/stg_employees/`
   - **Schema snapshot path:** `gs://<bucket>/customer360/schema_snapshots/EMPLOYEES_snapshot.json`
   - **Self-referencing relationship:** `manager_id → employee_id` within `stg_employees`; top-level employees have `manager_id = NULL`

2. **Operational procedures:** Document the following in the runbook:
   - **Manual backfill:** Trigger a manual DAG run from the Composer Airflow UI. Note that `WRITE_TRUNCATE` replaces the full table on every run — coordinate with downstream consumers (Looker, Looker Studio, dbt models) before initiating any backfill.
   - **Schema snapshot update:** When a planned upstream schema change is approved, re-run the snapshot capture script against the updated `agentichub.EMPLOYEES` schema and overwrite `EMPLOYEES_snapshot.json` in GCS before the next DAG execution. Update the snapshot hash in Airflow Variables.
   - **Dead-letter record inspection:** Browse `gs://<bucket>/customer360/dead_letter/stg_employees/` in GCS Console or via `gsutil ls`. Each file is newline-delimited JSON with `source_row` and `rejection_reason` fields. Review before re-running the pipeline on the same data.
   - **Pipeline health monitoring:** Use the Cloud Monitoring Dataflow dashboard to observe elements processed, system lag, and worker count per job. Use Cloud Logging log-based metric alerts for Airflow task failures (filter on DAG ID `customer360_stg_employees`).
   - **Credential rotation:** When the MySQL `sa` password is rotated, update the Secret Manager secret version. No pipeline code changes are required.

3. **Known risks:** Record in the runbook (cross-reference the Risks & Mitigations section):
   - **Connectivity dependency:** Cloud SQL Auth Proxy availability is tied to Cloud SQL maintenance windows and the GCP control plane. Any Auth Proxy disruption will cause hourly DAG runs to fail at the `schema_check` or Dataflow launch step.
   - **PII exposure:** `stg_employees` contains PII fields (`first_name`, `last_name`, `email`, `phone_number`, `salary`, `commission_pct`). Column-level access controls must be enforced before any consumer beyond the pipeline service account is granted access.
   - **Schema drift detection scope:** The pre-flight check detects column additions and removals. Intra-column type changes (e.g., `VARCHAR` width increases that do not alter the column name) may not be caught if the serialised column name list is unchanged. Consider extending the snapshot to include data types for fuller coverage.
   - **Full-load truncation risk:** `WRITE_TRUNCATE` means a truncated or empty upstream `EMPLOYEES` table will produce an empty `stg_employees` on the next run. There is no pipeline-level recovery from this scenario; a BigQuery table snapshot policy or dataset-level backup must be maintained separately.
   - **No native DQ framework:** Lightweight schema validation Beam assertions serve as the only in-pipeline data quality gate. A dedicated DQ framework must be introduced before the pipeline is promoted beyond Dev.

4. **Escalation paths:** Document in the runbook:
   - **Connectivity failure (Auth Proxy or TCP):** Escalate to GCP Networking / Cloud SQL admin team. Reference [BLOCKER-1] resolution notes for the confirmed connectivity approach.
   - **Schema drift alert (DAG failure in `schema_check`):** Escalate immediately to the source system owner (agentichub MySQL DBA) and the Customer360 data platform lead. Do not resume the pipeline until the root cause of the schema change is confirmed and the snapshot is updated under change control.
   - **Dead-letter spike (significant rejection volume):** Escalate to the Customer360 data engineering lead. Inspect dead-letter records before re-running. Determine whether the source data issue must be resolved upstream before the next pipeline run.
   - **BigQuery permission or quota error:** Escalate to the GCP IAM / project admin team.
   - **Unresolved open blockers ([BLOCKER-2] load strategy, [BLOCKER-3] BQ coordinates):** Escalate to the Customer360 product owner and data platform team lead. Document resolution in the runbook and update the pipeline accordingly.

5. **Sign-off confirmation:** Obtain the following sign-offs before promoting the pipeline to production:
   - [ ] **Customer360 Data Engineering Lead** — pipeline design, DAG logic, and Flex Template review
   - [ ] **GCP Security / Data Governance** — PII classification and column-level security controls confirmed on `stg_employees`
   - [ ] **Source System Owner (agentichub MySQL DBA)** — connectivity approach confirmed; schema change notification process agreed
   - [ ] **Cloud Platform / IAM Admin** — Dataflow and Composer service account permissions, Secret Manager configuration, and GCS bucket policies reviewed
   - [ ] **QA / Testing Lead** — all Phase 5 test cases (5.1–5.10) executed and passed in the Dev environment; results documented
