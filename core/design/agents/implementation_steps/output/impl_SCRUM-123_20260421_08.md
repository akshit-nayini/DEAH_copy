## ⚠️ Open Blockers

> **All three blockers must be resolved and documented before implementation advances past Phase 1. Do not proceed to Phase 2 with any blocker unresolved.**

1. **Binlog Enablement (Hard Prerequisite — Architecture Gate)**
   Is MySQL binary logging enabled on `verizon-data:us-central1:mysql-druid-metadatastore` with `binlog_format=ROW` and `binlog_row_image=FULL`? If not, the DBA team must confirm authority and an estimated enablement timeline. Without this, GCP Datastream CDC cannot function and the architecture must fall back to Cloud Run Jobs with a standard read-only user (Option 2).

2. **Replication User Grant (Hard Prerequisite — Architecture Gate)**
   Can a dedicated MySQL replication user be created on `agentichub` with `REPLICATION SLAVE`, `REPLICATION CLIENT`, and `SELECT` privileges for Datastream use? If organisational policy prohibits replication-level grants, the architecture must fall back to Option 2.

3. **Source Table Size & Change Rate (Required for Sizing)**
   What is the current approximate row count of `agentichub.employees` and the estimated daily change rate (INSERT, UPDATE, DELETE per day)? These figures are required to finalise Datastream cost estimates, BigQuery load job sizing, and to determine whether CDC or a daily full-snapshot is more cost-effective.

## ⚠️ Risks & Mitigations

> Reference material only — not embedded as implementation steps.

| Risk | Mitigation |
|---|---|
| **Credential Exposure** | Store all MySQL credentials in GCP Secret Manager. Configure Datastream and Cloud Composer to resolve Secret Manager resource IDs only — no plaintext credentials in DAG code, Terraform variables, or GCS objects. Use a dedicated service account with minimal IAM scope. Rotate the `sa` password before first production deployment; enforce a 90-day rotation policy. |
| **Data Completeness — Missing Change Indicator** | Audit the `employees` schema for a reliable incremental change indicator (`updated_date`) before finalising pipeline design. If absent or unreliable, implement a daily TRUNCATE + INSERT OVERWRITE as an interim measure and raise a request with the owning application team to add a proper change tracking column. |
| **Binlog Retention Gap** | Configure Cloud SQL `binlog_expire_logs_seconds` to a minimum of 7 days. Implement a Cloud Monitoring alert on Datastream stream lag exceeding 50% of the binlog retention window (e.g., alert at 3.5 days if retention is 7 days). Document and include the full resync runbook in the handover package. |

## Phase 3: BigQuery Target Schema Provisioning

1. **Create `stg_employees` table** — In the `verizon_data_deah` BigQuery dataset, create the `stg_employees` table with the following specification:

   | Column | Type | Nullable | Notes |
   |---|---|---|---|
   | `employee_id` | INT64 | NOT NULL | Primary key |
   | `first_name` | STRING | NOT NULL | |
   | `last_name` | STRING | NOT NULL | |
   | `email` | STRING | NOT NULL | Null-constraint validation pre-load |
   | `phone_number` | STRING | NULLABLE | NULLs preserved |
   | `hire_date` | DATE | NOT NULL | |
   | `job_id` | STRING | NOT NULL | Cluster column 2 |
   | `salary` | NUMERIC | NOT NULL | Accommodates source DECIMAL(8,2) |
   | `commission_pct` | NUMERIC | NULLABLE | NULL = not on commission plan |
   | `manager_id` | INT64 | NULLABLE | Self-ref FK; NULL for top-level |
   | `department_id` | INT64 | NULLABLE | Cluster column 1 |
   | `status` | STRING | NOT NULL | Cluster column 3 |
   | `created_date` | TIMESTAMP | NOT NULL | Partition column (DAY) |
   | `updated_date` | TIMESTAMP | NULLABLE | High-watermark delta key |

   Apply **DAY partitioning** on `created_date`. Apply **clustering** in the order: `department_id`, `job_id`, `status`.

2. **Create `audit_log` table** — In the `verizon_data_deah` dataset, create an `audit_log` table with columns: `run_id` (STRING), `pipeline_phase` (STRING — values: `HISTORICAL`, `INCREMENTAL`), `execution_ts` (TIMESTAMP), `source_row_count` (INT64), `target_row_count` (INT64), `status` (STRING — values: `VALIDATED`, `FAILED`), `failure_detail` (STRING, NULLABLE), `schema_delta_detected` (BOOL), `null_violations` (INT64).

3. **Create `schema_registry` table** — In the `verizon_data_deah` dataset, create a `schema_registry` table with columns: `run_id` (STRING), `captured_ts` (TIMESTAMP), `table_name` (STRING), `column_name` (STRING), `source_data_type` (STRING), `target_data_type` (STRING), `is_nullable` (BOOL), `drift_detected` (BOOL).

4. **Write initial schema baseline snapshot** — Execute a Cloud Composer SQL operator that queries MySQL `information_schema.COLUMNS` for `agentichub.employees` (via Cloud SQL hook) and `verizon_data_deah.INFORMATION_SCHEMA.COLUMNS` for `stg_employees`, then inserts one row per column into `schema_registry` with `drift_detected = FALSE`. This establishes the baseline for all future drift comparisons.

## Phase 4: Historical Load DAG (Cloud Composer)

1. **Develop BQ staging load task** — Write a `BigQueryInsertJobOperator` task that submits a BigQuery load job reading from the GCS backfill Avro prefix into a transient staging table `stg_employees_backfill_staging` in `verizon_data_deah`. Disable schema autodetect; apply the explicit `stg_employees` schema definition. Set write disposition to `WRITE_TRUNCATE`.

2. **Develop pre-load schema compatibility check task** — Write a Cloud Composer SQL operator that queries `verizon_data_deah.INFORMATION_SCHEMA.COLUMNS` for `stg_employees` and cross-references against a MySQL `information_schema.COLUMNS` query for `agentichub.employees` via a Cloud SQL hook. Check for: (a) target columns absent from the source, (b) source columns absent from the target (new columns), (c) data type mismatches. Insert a row per column into `schema_registry` per run, setting `drift_detected = TRUE` for any discrepancy. Fail the task on any breaking type mismatch; log non-breaking new columns as drift without failing.

3. **Develop null constraint validation task** — Write a `BigQueryCheckOperator` that executes:
   ```sql
   SELECT COUNT(*) FROM verizon_data_deah.stg_employees_backfill_staging
   WHERE employee_id IS NULL
      OR first_name IS NULL OR last_name IS NULL OR email IS NULL
      OR hire_date IS NULL OR job_id IS NULL OR salary IS NULL
      OR status IS NULL OR created_date IS NULL
   ```
   Expect result = 0. On failure, write the violating row count to `audit_log.null_violations` and fail the task, blocking the MERGE.

4. **Develop source-to-staging row count reconciliation task** — Write a task that: (a) executes `SELECT COUNT(*) FROM agentichub.employees` via MySQL Cloud SQL hook, (b) executes `SELECT COUNT(*) FROM verizon_data_deah.stg_employees_backfill_staging`, (c) asserts both counts are equal. Write both counts to `audit_log`. Fail the task if counts diverge, preventing promotion to the target table.

5. **Develop MERGE deduplication/upsert task** — Write a `BigQueryInsertJobOperator` task that executes a MERGE SQL statement against `verizon_data_deah.stg_employees` using `stg_employees_backfill_staging` as the source, matching on `employee_id`. On MATCHED: UPDATE all non-PK columns. On NOT MATCHED BY TARGET: INSERT all columns. This deduplicates on primary key and produces the authoritative target state.

6. **Develop historical audit sign-off write task** — Write a task that inserts one record into `verizon_data_deah.audit_log` with: `pipeline_phase = 'HISTORICAL'`, `status = 'VALIDATED'`, the source and target row counts from Step 4, `schema_delta_detected` from Step 2, and `null_violations = 0`. This record is the gate signal that activates the incremental pipeline.

7. **Assemble and deploy historical load DAG** — Assemble tasks 1–6 into a Cloud Composer DAG named `dag_employees_historical_load`. Set task dependencies in this order: `staging_load → schema_check → null_check → row_count_reconciliation → merge_upsert → audit_signoff`. Configure the DAG as non-scheduled (manual trigger only). Deploy to the Cloud Composer environment's GCS DAGs folder.

8. **Execute historical load DAG and confirm completion** — Trigger `dag_employees_historical_load` manually. Monitor all task states in the Airflow UI to confirm `success` for every task. Query `verizon_data_deah.audit_log` and confirm one `HISTORICAL / VALIDATED` record exists. Query `SELECT COUNT(*) FROM verizon_data_deah.stg_employees` and confirm it matches the source baseline recorded in Phase 1, Step 1.

## Phase 5: Incremental Pipeline DAG (Cloud Composer)

1. **Develop historical gate sensor task** — Write a `BigQueryCheckOperator` as the first task in the incremental DAG:
   ```sql
   SELECT COUNT(*) FROM verizon_data_deah.audit_log
   WHERE pipeline_phase = 'HISTORICAL' AND status = 'VALIDATED'
   ```
   Expect result > 0. If the check returns 0, the task fails and all downstream incremental tasks are blocked — enforcing the historical gate constraint.

2. **Develop high-watermark computation task** — Write a task that executes `SELECT MAX(updated_date) FROM verizon_data_deah.stg_employees` and passes the result as an XCom value to subsequent tasks for scoping GCS file selection to records with `updated_date > high_watermark`.

3. **Develop incremental GCS load task** — Write a `BigQueryInsertJobOperator` task that loads Avro change event files from the Datastream GCS landing prefix, scoped to the current cycle window using the high-watermark XCom from Step 2, into a transient staging table `stg_employees_incremental_staging`. Set write disposition to `WRITE_TRUNCATE`.

4. **Develop incremental schema compatibility check task** — Reuse the schema check logic from Phase 4, Step 2, applied to the current cycle. Insert a new `schema_registry` row per run. On `drift_detected = TRUE`, emit a structured Cloud Logging entry that triggers the configured log-based Cloud Monitoring alert (defined in Phase 6).

5. **Develop incremental null constraint validation task** — Reuse the null check logic from Phase 4, Step 3, applied to `stg_employees_incremental_staging`. Write violations to `audit_log.null_violations`. Fail the task on any violation.

6. **Develop incremental row count delta validation task** — Write a task that: (a) counts rows in `stg_employees_incremental_staging`, (b) counts records in the MySQL change window using `WHERE updated_date > :high_watermark` via Cloud SQL hook, (c) asserts equality. Write both counts to `audit_log`. Fail on divergence.

7. **Develop incremental MERGE upsert task** — Reuse the MERGE SQL pattern from Phase 4, Step 5, with `stg_employees_incremental_staging` as the source against `verizon_data_deah.stg_employees`. Match on `employee_id`. On MATCHED: UPDATE all non-PK columns. On NOT MATCHED BY TARGET: INSERT all columns.

8. **Develop incremental audit write task** — Write a task that inserts an `INCREMENTAL / VALIDATED` (or `FAILED` on error) record into `verizon_data_deah.audit_log` after each cycle, capturing `source_row_count`, `target_row_count`, `schema_delta_detected`, `null_violations`, and `execution_ts`.

9. **Assemble and deploy incremental DAG** — Assemble tasks 1–8 into a Cloud Composer DAG named `dag_employees_incremental_load`. Set task dependencies in this order: `gate_sensor → high_watermark → incremental_load → schema_check → null_check → row_count_delta → merge_upsert → audit_write`. Schedule at the agreed cadence (daily or hourly — to be confirmed after Open Blocker 3 is resolved). Deploy to the Composer GCS DAGs folder.

10. **Execute incremental DAG dry run** — Trigger `dag_employees_incremental_load` manually for one cycle. Confirm the gate sensor passes (historical sign-off present in `audit_log`). Confirm at least one changed record from MySQL is captured in the staging table, upserted into `stg_employees`, and a `INCREMENTAL / VALIDATED` record is written to `audit_log`.

## Phase 6: Monitoring & Alerting Configuration

1. **Configure Datastream stream error alert** — In Cloud Monitoring, create an alerting policy on the Datastream `datastream.googleapis.com/stream/errors` metric scoped to the `employees` stream. Set the threshold to any non-zero error count. Route notifications to the agreed data engineering on-call channel.

2. **Configure Datastream lag threshold alert** — Create a Cloud Monitoring alerting policy on Datastream stream lag. Set the threshold at 50% of the configured binlog retention window (e.g., 3.5 days for a 7-day retention). Route notifications to the data engineering team to trigger a resync response before events are at risk of expiry.

3. **Configure Cloud Composer DAG failure alert** — Create a Cloud Monitoring log-based alerting policy on Airflow task failure log entries scoped to DAG IDs `dag_employees_historical_load` and `dag_employees_incremental_load`. Route notifications to the data engineering on-call channel.

4. **Configure BigQuery load job error alert** — Create a Cloud Monitoring alerting policy on `bigquery.googleapis.com/job/num_failed_jobs` scoped to the `verizon_data_deah` dataset. Threshold: any failed load job triggers an alert.

5. **Configure schema drift log-based alert** — Create a Cloud Monitoring log-based metric on structured log entries emitted by the schema check task when `drift_detected = TRUE`. Create an alerting policy on this metric routing to both the data engineering team and the owning application team for `agentichub.employees`.

6. **Confirm Cloud Billing budget alert** — Confirm a Cloud Billing budget alert exists for Datastream bytes-processed costs and notifies the data engineering lead and stakeholder Yash on unexpected spend before go-live.

7. **Build Looker Studio audit dashboard** — Connect a Looker Studio report to `verizon_data_deah.audit_log`. Include: run status per cycle, source-to-target row count comparison, `schema_delta_detected` flag per run, last successful run timestamp, and `failure_detail` for failed runs. Grant access to stakeholder Yash. Confirm the dashboard renders the most recent `audit_log` record correctly.
