## ⚠️ Risks & Mitigations
- **Data Consistency**: Implement checksum validation and reconciliation processes with source system
- **SLA Compliance**: Set up proactive monitoring, alerts, and automatic retry mechanisms with sufficient buffer time
- **Security**: Use Secret Manager for credentials, VPC peering or Cloud SQL Auth Proxy for secure connectivity

## Phase 1: Source Connectivity and Ingestion Development

1. Test connectivity to MySQL source (host: 34.70.79.163, port: 3306, database: Agentichub)
2. Create Cloud Storage bucket for staging data with appropriate lifecycle policies
3. Develop Cloud Run ingestion job in Python to extract employees table data
4. Implement MySQL connection using Cloud SQL Auth Proxy or direct connection with SSL
5. Configure extraction logic to write data to GCS in newline-delimited JSON format
6. Add checksum validation and row count verification during extraction
7. Create Secret Manager entries for database credentials
8. Package and deploy ingestion Cloud Run job with appropriate IAM roles

## Phase 2: BigQuery Schema and Processing Development

1. Create BigQuery dataset for staging layer
2. Implement stg_employees table with specified schema and constraints
3. Configure table partitioning by created_date (daily) and clustering by employee_id, updated_date
4. Develop Cloud Run processing job for GCS-to-BigQuery data loading
5. Implement UPSERT logic using MERGE statements to handle data updates
6. Add data validation rules for required fields and referential integrity
7. Create error handling for data quality issues and failed loads
8. Package and deploy processing Cloud Run job

## Phase 3: Orchestration and Workflow Setup

1. Create Cloud Composer environment DAG for employee data pipeline
2. Configure DAG to trigger ingestion Cloud Run job first
3. Add dependency to trigger processing Cloud Run job after successful ingestion
4. Implement workflow error handling and retry mechanisms
5. Set up DAG scheduling based on business requirements
6. Configure workflow notifications for success and failure scenarios
7. Add workflow monitoring and logging integration

## Phase 4: Security and Access Controls

1. Configure IAM roles for Cloud Run jobs with least privilege access
2. Set up BigQuery dataset and table-level access controls
3. Implement row-level security if required for employee data
4. Configure VPC connectivity for secure database access
5. Set up audit logging for data access and pipeline operations
6. Test end-to-end security configuration

## Phase 5: Monitoring and Alerting

1. Configure Cloud Monitoring dashboards for pipeline health metrics
2. Set up alerts for job failures, data quality issues, and SLA breaches
3. Implement custom metrics for row count validation and processing time
4. Configure log-based alerts for error patterns and anomalies
5. Set up notification channels for operational teams
6. Create runbook templates for common operational scenarios

## Phase 6: Testing

1. **Connectivity Test**: Verify successful connection to MySQL source at 34.70.79.163:3306/Agentichub and ability to query employees table
2. **Data Extraction Test**: Extract sample employees data and verify all 14 columns are captured correctly with proper data types
3. **Schema Validation Test**: Load test data into stg_employees table and verify primary key constraint on employee_id functions correctly
4. **Partitioning Test**: Insert records with different created_date values and verify daily partitioning is working
5. **Clustering Test**: Query stg_employees by employee_id and updated_date to verify clustering performance benefits
6. **UPSERT Logic Test**: Insert initial record, then update same employee_id and verify MERGE statement correctly updates existing record
7. **Self-Referencing Relationship Test**: Insert employee with manager_id pointing to existing employee_id and verify many-to-one relationship constraint
8. **Null Handling Test**: Insert records with null values for optional fields (phone_number, commission_pct, manager_id, department_id, updated_date)
9. **End-to-End Pipeline Test**: Execute full DAG and verify data flows from MySQL → GCS → BigQuery with correct transformations
10. **Failure Recovery Test**: Simulate job failures and verify retry mechanisms and error handling work correctly
11. **Data Quality Test**: Inject invalid data and verify validation rules catch and handle data quality issues appropriately

## Phase 7: Handover

1. Update runbook with pipeline design summary: Cloud Composer orchestrating Cloud Run jobs for MySQL-to-BigQuery employee data pipeline with staging layer
2. Document operational procedures: DAG execution, monitoring dashboard access, manual job triggers, and troubleshooting steps
3. Record known risks: source database connectivity issues, data consistency challenges, and SLA compliance dependencies
4. Establish escalation paths: Level 1 (operations team for job failures), Level 2 (data engineering team for data quality issues), Level 3 (infrastructure team for connectivity problems)
5. Obtain sign-off confirmation from operations team, data governance team, and business stakeholders
6. Schedule knowledge transfer sessions and operational training
7. Archive implementation artifacts and update project documentation
