## ⚠️ Open Blockers

> **These blockers must be resolved before the corresponding pipeline phases can be finalised. Development may proceed in parallel but the affected steps cannot be signed off until each blocker is closed.**

1. **Watermark Column Existence** — It is unconfirmed whether the MySQL `agentichub.employees` table contains a reliable watermark column (e.g., `updated_at`, `modified_at`, `last_modified_timestamp`) suitable for incremental load filtering. If absent, the incremental strategy must switch to full-scan with MERGE/UPSERT, materially increasing daily extraction cost and duration. Resolution required before Phase 3 begins.

2. **Historical Load Sign-Off Mechanism** — The formal approval process for stakeholder Yash's historical load validation gate has not been defined. Cloud Workflows requires a programmatically evaluable artifact (e.g., a signed JSON object in GCS, a Firestore document). Confirm whether approval will be recorded in a JIRA ticket, email, or a pipeline-accessible artifact before Phase 3 gate implementation begins.

3. **VPC Service Controls Perimeter** — It is unconfirmed whether project `verizon-data` has a VPC Service Controls perimeter that could restrict Cloud Run serverless connector access to Cloud SQL private IP. If a perimeter exists, additional access policy rules or exceptions will be required. Resolution required before Phase 1 connectivity validation.

## Phase 1: Schema Discovery & Connectivity Validation

**Objective:** Confirm source reachability, capture the live schema of `agentichub.employees`, and validate it against the target data model before any extraction work begins.

1. **Connectivity smoke test** — Execute a Cloud Run job that connects to `agentichub` at `34.70.79.163:3306` via the Cloud SQL Auth Proxy serverless connector using instance connection name `verizon-data:us-central1:mysql-druid-metadatastore`. Run `SELECT 1 FROM employees LIMIT 1`. Assert a successful response. On failure, raise a Cloud Monitoring alert and halt all subsequent steps. Log outcome to Cloud Logging with structured fields `phase=schema_discovery`, `step=connectivity_smoke_test`, `status=<pass|fail>`.

2. **Live schema extraction** — Query `INFORMATION_SCHEMA.COLUMNS` for `agentichub.employees`. Capture field names, data types, `IS_NULLABLE` flags, and `COLUMN_KEY` values. Serialize the result as a JSON schema manifest and write it to GCS at a versioned path (e.g., `gs://<bucket>/schema-manifests/employees/SCRUM-145_<datestamp>.json`).

3. **Watermark column discovery** — Within the extracted schema manifest, inspect column names for candidates matching patterns `updated_at`, `modified_at`, `last_modified_timestamp`, or equivalent. Document the findings and resolve **Open Blocker 1** before proceeding to Phase 3 design.

4. **Schema assertion against target data model** — Compare the live MySQL schema manifest against the 14-column target definition for `stg_employees`. Assert the following for each expected column: field name present, data type compatible with the mapped BigQuery type, NOT NULL constraint alignment for `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, and `created_date`. Assert nullable allowance for `phone_number`, `commission_pct`, `manager_id`, `department_id`, and `updated_date`. Flag any unexpected additional columns as a schema drift alert.

5. **Schema drift handling definition** — Document the response policy for new column detection: pipeline halts, Cloud Monitoring alert fires, and a schema policy resolution is required before extraction resumes. Record this policy in the draft runbook.

6. **VPC Service Controls verification** — Confirm with the infrastructure team whether a VPC Service Controls perimeter exists on project `verizon-data` (resolves **Open Blocker 3**). Document the outcome and any required perimeter rule additions before proceeding.

## Phase 2: Historical Load Implementation

**Objective:** Extract the full `agentichub.employees` dataset (estimated 5 GB) to BigQuery `verizon_data_dea.stg_employees` using chunked extraction, GCS staging, and BigQuery native load with `WRITE_TRUNCATE`.

1. **Cloud Run job — chunked extractor** — Implement a containerised Python Cloud Run Job using `mysql-connector-python` and the Cloud SQL Auth Proxy serverless connector. Extraction uses keyset/cursor-based chunking ordered by `employee_id` with a configurable `chunk_size` defaulting to 100,000 rows. Each iteration issues `SELECT ... FROM employees WHERE employee_id > <last_cursor> ORDER BY employee_id LIMIT <chunk_size>`. Persist the cursor value and chunk sequence number in a GCS chunk manifest file after each successful chunk write.

2. **Per-chunk schema validation** — On read of each chunk, validate each row against the schema manifest captured in Phase 1. Assert that non-nullable columns (`employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date`) contain non-null values. Log any violations to Cloud Logging and halt the affected chunk.

3. **Per-chunk GCS write** — Serialize each validated chunk as newline-delimited JSON (NDJSON) and write atomically to GCS at path `gs://<bucket>/staging/historical/employees/chunk_<sequence>.ndjson`. Update the GCS chunk manifest to mark the chunk as `written`.

4. **Per-chunk retry logic** — Wrap each chunk's extraction and write in retry logic applying exponential backoff with a maximum of 3 attempts. On third failure, mark the chunk as `failed` in the manifest, raise a Cloud Monitoring alert referencing the chunk sequence number and cursor range, and halt extraction.

5. **BigQuery target table creation** — Create table `verizon_data_dea.stg_employees` with the 14-column schema defined in the data model. Apply `TIME` partitioning on `hire_date` at `DAY` granularity with 60-day partition expiry. Apply clustering on `department_id`, `job_id`, `status` (in that order). This step is a one-time DDL execution.

6. **BigQuery native load — historical** — After all chunks are marked `written` in the manifest, invoke a BigQuery load job targeting `verizon_data_dea.stg_employees` with `WRITE_TRUNCATE`, sourcing all `gs://<bucket>/staging/historical/employees/chunk_*.ndjson` files. Assert load job completion status.

7. **Post-load reconciliation** — Cloud Workflows invokes a BigQuery scripted SQL job. Take `SOURCE_COUNT` as the MySQL `COUNT(*)` snapshot captured at job start. Compare to `TARGET_COUNT` via `SELECT COUNT(*) FROM verizon_data_dea.stg_employees`. Assert equality within tolerance. Assert PK uniqueness via:
   ```sql
   SELECT employee_id, COUNT(*) AS cnt
   FROM verizon_data_dea.stg_employees
   GROUP BY employee_id
   HAVING cnt > 1;
   ```
   Assert zero rows returned.

8. **Audit log write — historical run** — Write an immutable record to `verizon_data_dea.audit_log` containing: `load_id`, `run_type=historical`, `chunk_count`, `source_count`, `target_count`, `validation_status`, `execution_start_timestamp`, `execution_end_timestamp`.

9. **GCS staging cleanup** — After a successful BigQuery load job confirmation, GCS lifecycle policy automatically deletes chunked NDJSON staging files. Verify the lifecycle rule is in effect and that the chunk manifest is retained for audit purposes.

## Phase 3: Incremental Load Implementation

**Objective:** Implement the daily scheduled incremental load, gated behind formal historical load sign-off. Incremental strategy is conditioned on the resolution of **Open Blocker 1**.

> **Note:** The Cloud Scheduler trigger is deployed in `PAUSED` state. Activation requires a manual Terraform apply with `approval_flag=true` only after historical load sign-off is formally recorded (resolves **Open Blocker 2**).

1. **Historical load sign-off gate** — Implement a Cloud Workflows step that reads a signed approval artifact from GCS (path: `gs://<bucket>/approvals/historical-load-signoff.json`). Assert the artifact contains valid `load_id`, `approver`, `timestamp`, and `confirmation_hash` fields matching the historical `audit_log` record. If the artifact is absent or invalid, Cloud Workflows exits with a non-zero code and fires a Cloud Monitoring alert. The incremental load does not proceed until this assertion passes.

2. **Incremental extraction — watermark path (if Open Blocker 1 resolved affirmatively)** — Cloud Run Job issues `SELECT ... FROM employees WHERE <watermark_column> > <last_high_watermark> ORDER BY <watermark_column>, employee_id`. Read the `last_high_watermark` value from the most recent `audit_log` record for `run_type=incremental`. Apply the same chunked extraction, per-chunk validation, and GCS write pattern as Phase 2 steps 2–4.

3. **Incremental extraction — full-scan path (if Open Blocker 1 resolved negatively)** — Cloud Run Job issues a full `SELECT * FROM employees ORDER BY employee_id` with the same chunked extraction pattern. Document that this path increases daily extraction cost and duration and record in the runbook.

4. **BigQuery MERGE — incremental** — After all incremental chunks are loaded to a transient GCS staging path, invoke a BigQuery MERGE job keyed on `employee_id`:
   ```sql
   MERGE verizon_data_dea.stg_employees T
   USING (SELECT * FROM staging_view) S
   ON T.employee_id = S.employee_id
   WHEN MATCHED THEN UPDATE SET ...
   WHEN NOT MATCHED THEN INSERT ...;
   ```
   All 14 columns are updated on match. `hire_date` partitioning and `department_id`, `job_id`, `status` clustering are preserved.

5. **Post-load reconciliation — incremental** — Repeat the row-count comparison and PK uniqueness assertion from Phase 2 step 7, scoped to the incremental run's affected partition(s) where possible.

6. **Audit log write — incremental run** — Write a record to `verizon_data_dea.audit_log` with `run_type=incremental`, `high_watermark_applied`, `chunk_count`, `source_count`, `target_count`, `validation_status`, and execution timestamps.

7. **Cloud Scheduler activation** — After historical load sign-off is confirmed and the gate artifact is in place, execute Terraform apply with `approval_flag=true` to transition the Cloud Scheduler trigger from `PAUSED` to `ENABLED` on the defined daily cron schedule.

## Phase 4: Orchestration & Monitoring

**Objective:** Wire Cloud Workflows, Cloud Scheduler, Cloud Monitoring alerting, and Looker Studio to complete the operational stack.

1. **Cloud Workflows definition** — Author the Cloud Workflows YAML definition encoding the linear execution sequence: sign-off gate check → connectivity smoke test → schema validation → extraction (Cloud Run Job invocation) → BigQuery load → post-load reconciliation → audit log write → GCS cleanup confirmation. Include retry and error-handling steps at each Cloud Run Job invocation using the platform's built-in retry syntax.

2. **Cloud Scheduler configuration** — Define the Cloud Scheduler job targeting the Cloud Workflows endpoint on the agreed daily cron expression. Deploy initially in `PAUSED` state via Terraform.

3. **Cloud Monitoring alerting policies** — Create alerting policies for: Cloud Run Job non-zero exit codes, Cloud Workflows execution failures, chunk-level retry exhaustion events, schema drift detection events, and sign-off gate assertion failures. Configure notification channels (e.g., PagerDuty or email) consistent with the operational escalation path.

4. **Cloud Logging structured sink** — Confirm that the Cloud Logging structured log sink captures all pipeline steps with consistent fields: `load_id`, `phase`, `step`, `status`, `chunk_sequence` (where applicable), and `timestamp`.

5. **Looker Studio dashboard** — Connect Looker Studio to `verizon_data_dea.audit_log`. Build a dashboard displaying: load success/failure per run, source vs. target row counts, validation status, chunk counts, and failure history. Provision stakeholder Yash with read access to the BigQuery dataset for independent ad-hoc SQL validation.
