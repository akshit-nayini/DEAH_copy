## âš ď¸Ź Open Blockers

Resolution of the following blockers is required before proceeding past Phase 2, Step 2. Each blocker owner must confirm in writing before the corresponding phase is signed off.

> **Blocker 1 â€” Confluent Schema Registry adoption (impacts Phase 2, Step 2 & 3)**
> Is Confluent Schema Registry active for topics `clickstream.page_views`, `clickstream.user_events`, and `clickstream.sessions`? If AVRO or Protobuf schemas are registered, the Beam pipeline must use the Confluent Schema Registry Beam transform for deserialization in place of generic JSON parsing. The deserialization strategy cannot be finalised until this is confirmed.

> **Blocker 2 â€” Schema evolution policy with upstream producers (impacts Phase 2, Step 3; Phase 5)**
> What is the agreed schema evolution policy with upstream Kafka producers? Specifically: are field additions guaranteed to be backward-compatible, and is there a formal schema change notification process to alert the pipeline team before breaking changes are deployed? Without this agreement, the null-safe field mapping fallback strategy and dead-letter routing logic cannot be fully specified.

> **Blocker 3 â€” Dead-letter topic retention period (impacts Phase 2, Step 7; Phase 6)**
> What retention period is configured on `clickstream.dead_letter`? A minimum of 7 days is required to support incident investigation and reprocessing after schema fixes. If the current retention is less than 7 days, this must be remediated before pipeline go-live.

## âš ď¸Ź Risks & Mitigations *(reference only â€” do not action as steps)*

| Risk | Mitigation Summary |
|---|---|
| **Data Loss** | Align KafkaIO checkpointing to Dataflow durable shuffle checkpoints so offsets are committed only after a successful BigQuery Storage Write API exactly-once commit. Set `auto.offset.reset=earliest`. Hourly reconciliation DAG compares lag-adjusted Kafka offset counts against BigQuery row counts; alert on >0.01% discrepancy. Maintain minimum 24-hour Kafka topic retention to support replay. |

| **Latency SLA Breach** | Enable Dataflow Streaming Engine. Set Storage Write API commit interval to 20 seconds, leaving 40 seconds of remaining budget against the 60-second SLA. ## Phase 2: Beam Pipeline â€” Core Development

1. **Scaffold Dataflow Flex Template project:** Create the Apache Beam 2.x pipeline project structure with Flex Template configuration. Define runtime parameters for: Kafka bootstrap server, GCP Secret Manager secret IDs for Kafka credentials, BigQuery project and dataset identifiers, and dead-letter topic name. No secrets or environment-specific values are to be embedded in the template image.

2. **Implement KafkaIO source connector:** Configure `KafkaIO.read()` targeting all three topics (`clickstream.page_views`, `clickstream.user_events`, `clickstream.sessions`) on broker `pkc-abc12.us-central1.gcp.confluent.cloud:9092`. Inject SASL/PLAIN credentials at runtime from GCP Secret Manager. Set consumer group `auto.offset.reset=earliest`. Enable Dataflow durable state store checkpointing, aligned to Storage Write API exactly-once commit boundaries so that Kafka offsets are only advanced after a confirmed BigQuery write commit. *Deserialization strategy (JSON parsing vs. Schema Registry Beam transform) is contingent on Blocker 1 resolution.*

3. **Implement topic-specific parsing DoFns:** Build a null-safe parsing `DoFn` for each of the three topic schemas. Apply explicit fallback values for all nullable optional fields as defined in the target schema. Map parsed fields to the target staging schemas (`stg_page_views`, `stg_user_events`, `stg_sessions`). For any record where JSON deserialization fails entirely, tag with `error_type=PARSE_ERROR` and route to the dead-letter `PCollection` with a structured error envelope containing: `raw_payload`, `error_type`, `topic`, `partition`, `offset`, `timestamp`.

4. **Implement schema validation DoFn:** After parsing, validate each record against the non-nullable field constraints defined in the target schema:
   - `stg_page_views`: `event_id`, `session_id`, `anonymous_id`, `page_url`, `event_timestamp`, `ingested_at`
   - `stg_user_events`: `event_id`, `session_id`, `anonymous_id`, `event_type`, `page_url`, `event_timestamp`, `ingested_at`
   - `stg_sessions`: `session_id`, `anonymous_id`, `session_start`, `ingested_at`

   Route any record failing non-nullable validation to the dead-letter `PCollection` with `error_type=SCHEMA_VALIDATION_FAILURE` and the full structured error envelope.

5. **Implement `event_timestamp` bounds check:** For records destined for `stg_page_views` and `stg_user_events`, validate that `event_timestamp` falls within Â±5 minutes of pipeline processing wall clock time. Route out-of-bounds records to the dead-letter `PCollection` with `error_type=TIMESTAMP_OUT_OF_BOUNDS`.

6. **Implement event deduplication:** Apply Beam stateful processing with a 10-minute sliding window and 1-minute slide interval, keyed on `event_id`, to deduplicate records across `stg_page_views` and `stg_user_events`. Retain the first occurrence of each `event_id` within the window; discard subsequent duplicates. Emit an audit count of duplicates dropped per window fire to the pipeline audit log.

7. **Implement dead-letter Kafka sink:** Route all tagged dead-letter `PCollection` records (from Steps 3, 4, and 5) to the `clickstream.dead_letter` Kafka topic using `KafkaIO.write()`. Serialize each error envelope as JSON. Confirm Kafka producer configuration includes appropriate retry and ack settings to ensure dead-letter delivery is not silently dropped.

8. **Implement pipeline audit logging:** After each deduplication window fires, emit a structured audit record to `pipeline_audit.pipeline_run_log` in BigQuery. Each record must include `schema_version` to satisfy the schema evolution risk mitigation traceability requirement.

## Phase 3: BigQuery Target Schema & Storage Write API

1. **Create staging tables:** Create tables `stg_page_views`, `stg_user_events`, and `stg_sessions` in dataset `verizon_clickstream_deah` in project `verizon-data`. Apply all column definitions, data types, and nullability constraints exactly as specified in the target schema. Do not apply any column not present in the target schema specification.

2. **Apply DAY partitioning:**
   - `stg_page_views`: partition by `event_timestamp` (DAY)
   - `stg_user_events`: partition by `event_timestamp` (DAY)
   - `stg_sessions`: partition by `session_start` (DAY)

3. **Apply clustering:**
   - `stg_page_views`: `session_id`, `device_type`, `country_code`
   - `stg_user_events`: `event_type`, `session_id`, `device_type`
   - `stg_sessions`: `user_id`, `channel`, `device_type`, `country_code`

4. **Create `pipeline_audit.pipeline_run_log` table:** Create the audit log table in BigQuery with columns sufficient to capture per-window pipeline run metadata, including `schema_version`, window start/end timestamps, record counts, and duplicate-drop counts.

5. **Configure BigQuery Storage Write API sink:** Wire the validated, deduplicated `PCollection`s to the three staging tables using the BigQuery Storage Write API in **exactly-once mode**. Set the commit interval to 20 seconds. Target dataset `verizon_clickstream_deah` in project `verizon-data`. Confirm that offset advancement in the KafkaIO consumer group is gated on successful commit confirmation from the Storage Write API.

## Phase 4: Orchestration & Reconciliation

1. **Implement hourly reconciliation DAG:** Create a Cloud Composer 2 (Airflow 2.x) DAG on an hourly schedule. The DAG must retrieve lag-adjusted consumer group offset counts from Confluent Kafka for topics `clickstream.page_views`, `clickstream.user_events`, and `clickstream.sessions`, then compare against BigQuery row counts in `stg_page_views`, `stg_user_events`, and `stg_sessions` for the same time window. Trigger a Cloud Monitoring alert if the discrepancy for any topic exceeds 0.01%.

2. **Implement peak-hours worker scaling task:** Create a Cloud Composer DAG task on a schedule aligned to defined business hours peak windows. The task must issue a Dataflow job update setting `minWorkers=4` at peak window open and restore the default `minWorkers` at window close.

3. **Build and register the Flex Template:** Build the Dataflow Flex Template container image and push it to Artifact Registry. Parameterise all runtime values (Kafka bootstrap server, Secret Manager secret IDs for credentials, BigQuery project/dataset, dead-letter topic name). Verify that no credentials or environment-specific values are embedded in the image.

4. **Deploy the Dataflow Streaming job:** Launch the Flex Template as a Dataflow Streaming Engine job with `minWorkers=4` for initial production deployment. Confirm that the Streaming Engine is enabled on the job. Verify Kafka consumer group offsets begin advancing from the baseline recorded in Phase 1, Step 4.

## Phase 5: Monitoring & Alerting

1. **Configure end-to-end latency alert:** Create a Cloud Monitoring alerting policy that fires when measured end-to-end event latency (Kafka ingest to BigQuery commit) exceeds **45 seconds**, providing a 15-second early warning before the 60-second SLA breach threshold. Route alert notifications to the designated on-call channel.

2. **Configure Dataflow worker health dashboard:** Create a Cloud Monitoring dashboard surfacing Dataflow worker health metrics, per-topic Kafka consumer lag, and end-to-end latency percentiles (p50, p90, p99) for all three source topics.

3. **Configure dead-letter topic alerting:** Set a Cloud Monitoring alert to fire whenever any record is written to `clickstream.dead_letter`. Route structured error envelope metadata from dead-letter records to Cloud Logging for forensic querying. Include `error_type`, `topic`, `partition`, and `offset` in log entries.

4. **Configure reconciliation failure alert:** Wire the hourly reconciliation DAG (Phase 4, Step 1) to emit a Cloud Monitoring custom metric on each run. Create an alerting policy that fires when row count discrepancy exceeds 0.01% for any topic, routed to the on-call channel with the per-topic discrepancy delta included in the alert payload.

5. **Validate Cloud Logging pipeline audit access:** Confirm that `pipeline_audit.pipeline_run_log` entries written by the Beam pipeline (Phase 2, Step 8) are queryable via Cloud Logging and that the Dataflow service account has sufficient permissions for operational log retrieval.
