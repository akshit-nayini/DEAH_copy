## ⚠️ Open Blockers

> These blockers must be resolved before the indicated phases can be marked complete. They are referenced in the relevant phase steps below.

1. **[BLOCKER-1] PII Column-Level Security Scope** — It has not been confirmed which fields in `employees` (e.g. `email`, `phone_number`, `salary`, and any fields not yet disclosed such as national ID, date of birth, or home address) require BigQuery Policy Tag enforcement, column-level masking, or tokenisation prior to data landing in `verizon_data_dea`. This is a **hard go-live gate** for Phase 2 (table creation) and Phase 6 (security controls). Owner: data governance / security team.

2. **[BLOCKER-2] Column-Addition Handling Strategy — Approval by Stakeholder yash** — The formal document defining how net-new MySQL columns are handled (schema evolution policy, propagation rules, blocking vs. non-blocking behaviour) has not been confirmed as drafted or approved. This is a **hard go-live gate** for Phase 5 (schema validation task) and Phase 7 (testing). Owner: stakeholder yash. Target approval date relative to SCRUM-153 delivery timeline is unconfirmed.

3. **[BLOCKER-3] Cloud SQL Instance Shared Workload Confirmation** — It is unconfirmed whether `verizon-data:us-central1:mysql-druid-metadatastore` is shared with Apache Druid or other operational systems, and what the agreed export window and maximum allowable read load are. This is a **hard gate** for scheduling the Cloud Composer DAG (Phase 5). Owner: DBA team.

## ⚠️ Risks & Mitigations

> Reference material only — not embedded as implementation steps.

| Risk | Mitigation |
|---|---|
| **PII Data Security** | Apply BigQuery Policy Tags on PII columns (`first_name`, `last_name`, `email`, `phone_number`, `salary`) before pipeline go-live; enforce GCS bucket encryption (CMEK or Google-managed); restrict dataset access via IAM to named service accounts and authorised analyst groups; document data classification in the security design artefact. |
| **Schema Drift** | Implement automated pre-load schema validation as a blocking Airflow task before every load; configure a Cloud Monitoring alert on any schema mismatch; enforce approved column-addition handling strategy (BLOCKER-2) as a mandatory go-live gate. |
| **Cloud SQL Instance Contention** | Schedule Cloud SQL Admin API export during agreed low-activity window (02:00–04:00 UTC, pending BLOCKER-3 confirmation); confirm with DBA team whether a read replica is required for analytics export isolation; document agreed window in SLA artefact. |
| **Hard-Delete Invisibility** | Rows deleted in MySQL will not propagate as deletes to BigQuery. Documented as an accepted trade-off in the data contract. A soft-delete reconciliation query may be introduced in a future iteration if required. |
| **Export Consistency Window** | Cloud SQL Admin API export is not guaranteed to be point-in-time consistent under active write load. Mitigated by scheduling during the low-activity window (pending BLOCKER-3) or by using a Cloud SQL read replica. |
| **Single-Threaded Export Scalability** | Adequate for current volume; architectural revisit required at approximately 30–50 GB table size. This is the documented scaling trigger. |

## Phase 2: BigQuery Table Definitions & PII Controls

**Goal:** Create all BigQuery tables with correct schema, partitioning, clustering, and column-level security before any data is loaded. **BLOCKER-1 must be resolved before step 4.**

1. **Create transient staging table `verizon_data_dea.employees_staging`.** This table is the GCS CSV landing target used exclusively by the BigQuery load job. Define the same 15-column schema as `stg_employees` (see data model). No partitioning or clustering required on this transient table. Set table expiration to 1 day to auto-clean residual loads.

2. **Create target table `verizon_data_dea.stg_employees`.** Apply the full schema from the data model:

   | Column | Type | Nullable | Notes |
   |---|---|---|---|
   | `employee_id` | INT64 | NOT NULL | Primary key; merge key component |
   | `first_name` | STRING | NOT NULL | PII |
   | `last_name` | STRING | NOT NULL | PII |
   | `email` | STRING | NOT NULL | PII — Policy Tag required |
   | `phone_number` | STRING | NULL | PII — Policy Tag required |
   | `hire_date` | DATE | NOT NULL | |
   | `job_id` | STRING | NOT NULL | Clustering col 3 |
   | `salary` | NUMERIC | NOT NULL | Sensitive compensation data |
   | `commission_pct` | NUMERIC | NULL | NULL = non-commissioned role |
   | `manager_id` | INT64 | NULL | Self-referential FK → `employee_id` |
   | `department_id` | INT64 | NULL | Clustering col 1 |
   | `status` | STRING | NOT NULL | Clustering col 2 |
   | `created_date` | TIMESTAMP | NOT NULL | UTC |
   | `updated_date` | TIMESTAMP | NULL | Partition col; NULL → `__NULL__` partition |

   Apply **TIME partitioning** on `updated_date` at **DAY** granularity.
   Apply **clustering** in order: `department_id`, `status`, `job_id`.

3. **Create audit log table `verizon_data_dea.pipeline_audit_log`.** Define schema to capture: `run_id` (STRING), `load_type` (STRING: FULL / INCREMENTAL), `dag_run_id` (STRING), `source_row_count` (INT64), `staging_row_count` (INT64), `target_row_count_pre_merge` (INT64), `target_row_count_post_merge` (INT64), `schema_validation_status` (STRING), `row_count_reconciliation_status` (STRING), `pk_uniqueness_status` (STRING), `load_status` (STRING: SUCCESS / FAILED), `failure_reason` (STRING, nullable), `execution_timestamp` (TIMESTAMP). Set table-level expiration to **365 days** (overriding the dataset 60-day default).

4. **Apply BigQuery Policy Tags to PII columns.** *(Blocked on BLOCKER-1.)* Once the PII field scope is confirmed, attach the appropriate taxonomy Policy Tags to `email`, `phone_number`, `salary`, `first_name`, and `last_name` on `verizon_data_dea.stg_employees`. Confirm that analyst IAM bindings include the `roles/datacatalog.categoryFineGrainedReader` grant for authorised groups only. Confirm GCS bucket encryption policy (CMEK or Google-managed) is in place on `gs://verizon-5gcore-staging/`.

5. **Validate all table DDL against the data model.** Run `bq show --schema` on each created table and confirm column names, data types, nullability, partitioning spec, and clustering spec match the data model exactly. Record validation output.

## Phase 3: Stored Procedure Development — `sp_mysqltobq_load.sql`

**Goal:** Implement and validate the BigQuery stored procedure that performs the MERGE upsert from `employees_staging` into `stg_employees`.

1. **Draft `sp_mysqltobq_load.sql`.** The procedure must:
   - Accept no required runtime parameters (dataset and table names may be parameterised as procedure arguments for reuse).
   - Execute a `MERGE` statement against `verizon_data_dea.stg_employees` (target) using `verizon_data_dea.employees_staging` (source).
   - Use the composite merge key: `target.employee_id = source.employee_id AND target.updated_date = source.updated_date`.
   - On **MATCH**: update all non-key columns.
   - On **NOT MATCHED BY TARGET**: insert the full row.
   - Do **not** include a `NOT MATCHED BY SOURCE` clause — hard-delete invisibility is an accepted trade-off (documented in data contract).
   - After the MERGE, execute a `SELECT COUNT(*) FROM verizon_data_dea.stg_employees` and surface the result for audit log capture.

2. **Deploy stored procedure to BigQuery.** Execute `CREATE OR REPLACE PROCEDURE verizon_data_dea.sp_mysqltobq_load(...)` via the BigQuery Jobs API or `bq query`. Confirm successful creation via `INFORMATION_SCHEMA.ROUTINES`.

3. **Unit test the stored procedure in isolation.** Load synthetic data into `employees_staging` covering: (a) a net-new record, (b) an update to an existing `employee_id` with a newer `updated_date`, (c) a duplicate row with the same composite key (should not insert duplicate). Execute the stored procedure, query `stg_employees`, and verify outcomes match expected upsert semantics. Document test results.

4. **Confirm post-MERGE `employee_id` uniqueness.** After unit tests, execute `SELECT employee_id, COUNT(*) FROM verizon_data_dea.stg_employees GROUP BY employee_id HAVING COUNT(*) > 1` and confirm zero rows returned.

5. **Clean up synthetic test data** from `stg_employees` and `employees_staging` before proceeding to DAG development.

## Phase 4: Cloud Composer DAG Development

**Goal:** Build, configure, and deploy both the full historical load DAG and the incremental load DAG. **BLOCKER-3 must be resolved before finalising DAG schedule.**

### 4a — Full Historical Load DAG

1. **Create DAG `dag_5gcore_employees_full_load`.** Configure as a one-time or manually-triggered DAG (no schedule interval). Set the DAG to write a completion marker to a pre-agreed GCS path (e.g. `gs://verizon-5gcore-staging/gates/full_load_complete/`) upon successful completion — this marker is consumed by the `ExternalTaskSensor` in the incremental DAG.

2. **Task 1 — `export_full_mysql_to_gcs` (CloudSQLExportOperator).** Export the full `employees` table from `agentichub` on instance `verizon-data:us-central1:mysql-druid-metadatastore` to `gs://verizon-5gcore-staging/employees/YYYY-MM-DD/employees_full.csv`. Use UTF-8 encoding, CSV format, no SQL `WHERE` clause restriction (full table export).

3. **Task 2 — `validate_schema_pre_load` (PythonOperator / BigQueryCheckOperator).** Query MySQL `INFORMATION_SCHEMA.COLUMNS` for `agentichub.employees` and compare field names, data types, nullability, and column count against the BigQuery `stg_employees` schema. If any mismatch is detected (type change, missing column, unexpected new column): set task state to FAILED, write a failure record to `pipeline_audit_log`, and raise a Cloud Monitoring alert. *(Column-addition handling behaviour is subject to BLOCKER-2 approval.)* Block downstream tasks on failure.

4. **Task 3 — `load_gcs_to_bq_staging` (BigQueryInsertJobOperator).** Execute a BigQuery load job importing the GCS CSV from Task 1 into `verizon_data_dea.employees_staging`. Configure: `WRITE_TRUNCATE` disposition (transient table), `SKIP_LEADING_ROWS=1` if CSV includes headers, `MAX_BAD_RECORDS=0` (zero tolerance), `ALLOW_QUOTED_NEWLINES=true`.

5. **Task 4 — `reconcile_row_counts` (BigQueryCheckOperator / PythonOperator).** Execute `COUNT(*)` on `agentichub.employees` (MySQL, via JDBC or Cloud SQL proxy query) and `COUNT(*)` on `verizon_data_dea.employees_staging` (BigQuery). Compare values. On mismatch: FAIL the task, write a failure record to `pipeline_audit_log`, and block downstream tasks.

6. **Task 5 — `invoke_merge_procedure` (BigQueryInsertJobOperator).** Call `CALL verizon_data_dea.sp_mysqltobq_load(...)` via the BigQuery Jobs API. Capture the returned post-MERGE row count for the audit log.

7. **Task 6 — `check_pk_uniqueness` (BigQueryCheckOperator).** Execute `SELECT COUNT(*) FROM (SELECT employee_id, COUNT(*) c FROM verizon_data_dea.stg_employees GROUP BY employee_id HAVING c > 1)`. Assert result equals zero. On failure: FAIL task and write to `pipeline_audit_log`.

8. **Task 7 — `write_audit_log` (BigQueryInsertJobOperator).** Insert a structured row into `verizon_data_dea.pipeline_audit_log` capturing all metrics from Tasks 2–6: load type (`FULL`), source/staging/target row counts, schema validation status, row count reconciliation status, PK uniqueness status, overall load status, and `execution_timestamp` (UTC).

9. **Task 8 — `write_full_load_gate_marker` (GCSCreateObjectOperator or PythonOperator).** Write a success marker file to `gs://verizon-5gcore-staging/gates/full_load_complete/` to signal readiness for incremental DAG activation.

10. **Set task dependencies in DAG** in the following strict sequence: `export → validate_schema → load_to_staging → reconcile_rows → invoke_merge → check_pk → write_audit_log → write_gate_marker`.

### 4b — Incremental Load DAG

11. **Create DAG `dag_5gcore_employees_incremental`.** Configure schedule interval pending BLOCKER-3 resolution (proposed window: 02:00–04:00 UTC daily). Set `catchup=False`.

12. **Task 0 — `gate_full_load_complete` (ExternalTaskSensor).** Sense successful completion of `dag_5gcore_employees_full_load`. Until the gate marker is confirmed, block the incremental DAG from executing. Set a reasonable timeout (e.g., 30 minutes) with a `poke_interval` of 60 seconds.

13. **Task 1 — `export_incremental_mysql_to_gcs` (CloudSQLExportOperator).** Export `employees` rows where `updated_date >= {last_successful_load_timestamp}` (retrieved from `pipeline_audit_log` MAX successful `execution_timestamp` for load type `INCREMENTAL` or `FULL`). Write CSV to `gs://verizon-5gcore-staging/employees/YYYY-MM-DD/employees_incremental.csv`.

14. **Tasks 2–8 — Repeat the same sequence as the full load DAG** (schema validation → GCS-to-BQ staging load → row count reconciliation → MERGE procedure → PK uniqueness check → audit log write), substituting `load_type = 'INCREMENTAL'` in the audit log record. The row count reconciliation in the incremental case compares the MySQL delta `COUNT(*)` (with the same `WHERE updated_date >=` filter) against the `employees_staging` row count.

15. **Set task dependencies** in the same strict sequence as the full load DAG, preceded by the `gate_full_load_complete` sensor.

## Phase 5: Monitoring & Alerting

**Goal:** Ensure all failure modes surface to the correct audience without requiring manual log inspection.

1. **Configure Cloud Monitoring alert — Composer task failure.** Create a metric-based alert policy targeting Airflow task failure events for DAGs `dag_5gcore_employees_full_load` and `dag_5gcore_employees_incremental`. Notification channel: confirmed team escalation channel (PagerDuty / email / Slack — confirm with ops team). Set alert threshold: any single task failure triggers immediate notification.

2. **Configure Cloud Monitoring alert — Schema validation failure.** Create a log-based metric on structured log entries emitted by the `validate_schema_pre_load` task where `schema_validation_status = FAILED`. Attach to the same notification channel as step 1. Confirm alert fires within 5 minutes of log emission.

3. **Configure Cloud Monitoring alert — Row count reconciliation failure.** Create a log-based metric on `pipeline_audit_log` entries or structured logs where `row_count_reconciliation_status = FAILED`. Attach to the same notification channel.

4. **Configure structured pipeline execution logging.** Confirm all DAG tasks emit structured JSON logs to Cloud Logging with fields: `dag_id`, `task_id`, `run_id`, `status`, `row_count`, `timestamp`. Validate log entries are queryable in Cloud Logging.

5. **Build Looker Studio pipeline health dashboard.** Connect Looker Studio to `verizon_data_dea.pipeline_audit_log`. Include the following views: last successful run timestamp, load type, source vs. target row counts, schema validation status history, failure history (last 30 days), and current pipeline status. Share with stakeholder yash and confirm self-serve access without engineering involvement.

6. **Validate end-to-end alerting.** Deliberately trigger a schema validation failure in a non-production Composer run and confirm the Cloud Monitoring alert fires and the notification is received by the correct channel. Document alert fire time.
