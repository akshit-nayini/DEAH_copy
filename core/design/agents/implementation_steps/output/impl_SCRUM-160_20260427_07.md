## ⚠️ Open Blockers

> **These blockers must be resolved before any phase beyond Phase 1 proceeds. Development agents must not assume resolution.**

1. **VPC Connectivity (Hard Deployment Blocker):** It has not been confirmed that the Cloud Composer 2 environment has network connectivity to the Cloud SQL instance `verizon-data:us-central1:mysql-druid-metadatastore` via VPC peering or Private Service Connect. This must be validated before any pipeline provisioning proceeds.

2. **Incomplete Source Table Inventory:** The `source_tables` field lists only `employees`. The full, approved list of MySQL tables in `agentichub` requiring historical and incremental loads is unknown. This inventory is required to design the Composer DAG fan-out pattern, size the GCS staging bucket, and determine per-table scheduling cadences.

3. **`sp_mysqltobq_load.sql` Idempotency Not Validated:** It has not been confirmed that the stored procedure is safe for re-execution on Composer retry. The MERGE must not produce duplicate records, corrupt `updated_date` watermarks, or apply out-of-order updates. A formal idempotency test must be documented and signed off before production deployment.

## Phase 3: Extraction & GCS Staging

1. **Implement Cloud SQL Auth Proxy connection helper:** Create a reusable Composer PythonOperator utility that initialises the Auth Proxy sidecar connection to `verizon-data:us-central1:mysql-druid-metadatastore` and opens a `pymysql` connection to `agentichub` using credentials for username `sa`.

2. **Implement full historical extraction operator:** Write a PythonOperator that:
   - Executes `SELECT * FROM employees` with no watermark filter.
   - Streams results to `gcs://5gcore-staging/agentichub/employees/full/YYYYMMDD_HHMMSS_employees.csv` as a delimited CSV.
   - Applies explicit quoting rules: all STRING fields quoted, DATE/TIMESTAMP fields emitted in ISO 8601 format, NUMERIC fields emitted with two decimal places, NULLs written as the literal empty string `""`.
   - Records the source row count extracted to an Airflow XCom variable for downstream reconciliation.

3. **Implement incremental extraction operator:** Write a PythonOperator that:
   - Reads `max_updated_date` from `verizon_data_audit.load_watermark` where `table_name = 'employees'`.
   - Executes `SELECT * FROM employees WHERE updated_date > {max_updated_date}` using the watermark.
   - Streams results to `gcs://5gcore-staging/agentichub/employees/incremental/YYYYMMDD_HHMMSS_employees.csv` applying the same quoting rules as the full extract.
   - Records source row count to XCom.
   - If the watermark is NULL (guard condition), abort with a clear error — the incremental DAG must not execute before the historical load is complete.

4. **Post-extraction CSV row count validation:** After each extraction, verify that the GCS object is non-empty and that the line count (minus header) matches the XCom-stored source row count. Fail the task if counts do not match; do not proceed to the load phase.

5. **GCS lifecycle policy confirmation:** After a successful confirmed load (Phase 4 Step 6), trigger a lifecycle policy confirmation check to ensure the staging CSV object is marked for deletion per the auto-delete-after-confirmed-load policy. Log the GCS object URI and deletion eligibility status to `load_log`.

## Phase 4: BigQuery Load & Stored Procedure Execution

1. **Pre-load schema drift check:** Before every load run (full or incremental), execute a Composer task that:
   - Fetches `agentichub.employees` schema from MySQL `INFORMATION_SCHEMA.COLUMNS`.
   - Compares against the current `stg_employees` BigQuery schema.
   - If **additive drift** detected: execute `ALTER TABLE verizon_data_dea.stg_employees ADD COLUMN ...`, write to `schema_drift_log` with `drift_type = 'ADDITIVE'`, emit a non-blocking warning alert via Cloud Monitoring.
   - If **destructive drift** detected: write to `schema_drift_log` with `drift_type = 'DESTRUCTIVE'`, raise a P1 alert, and **block** the load task pending engineering sign-off. Do not proceed.

2. **LOAD DATA from GCS into raw staging:** Use `BigQueryInsertJobOperator` to execute a `LOAD DATA OVERWRITE` (or `INTO`) statement targeting a transient raw table `verizon_data_dea.raw_employees_staging` from the GCS CSV file URI passed via XCom. Apply explicit schema mapping to ensure datatype fidelity for all 14 columns.

3. **Execute `sp_mysqltobq_load.sql` stored procedure:** Use `BigQueryInsertJobOperator` to invoke the stored procedure. The procedure must execute a `MERGE` on `verizon_data_dea.stg_employees` using merge keys `employee_id` AND `updated_date`, applying upsert semantics (UPDATE matched rows, INSERT new rows). Confirm Blocker 3 (idempotency sign-off) is resolved before this step is run in production.

4. **Row count reconciliation:** After stored procedure completion, execute:
   - `SELECT COUNT(*) FROM verizon_data_dea.stg_employees` (BigQuery).
   - Compare against the MySQL source count retrieved via the extraction operator XCom.
   - Write result to `verizon_data_audit.reconciliation_log` with `check_type = 'ROW_COUNT'`.
   - If the delta exceeds the defined threshold, fail the task and emit a Cloud Monitoring alert.

5. **Primary key uniqueness check:** Execute `SELECT COUNT(*) - COUNT(DISTINCT employee_id) AS duplicates FROM verizon_data_dea.stg_employees`. Expected result: `0`. Write result to `reconciliation_log` with `check_type = 'PK_UNIQUENESS'`. Fail the task on any non-zero value.

6. **Null constraint verification:** Assert that the following columns contain zero NULL values in `stg_employees`: `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date`. Write each assertion result to `reconciliation_log` with `check_type = 'NULL_CONSTRAINT'`. Fail the task on any violation.

7. **Watermark advancement:** On full validation pass, update `verizon_data_audit.load_watermark` SET `max_updated_date = (SELECT MAX(updated_date) FROM verizon_data_dea.stg_employees)`, `last_updated_ts = CURRENT_TIMESTAMP()` for `table_name = 'employees'`.

8. **Write load completion record:** Insert a completed row into `verizon_data_audit.load_log` with `run_id`, `load_type`, `source_row_count`, `target_row_count`, `status = 'SUCCESS'`, start and end timestamps.

## Phase 5: Orchestration — Cloud Composer DAGs

1. **Implement the historical load DAG (`dag_5gcore_employees_full_load`):**
   - Task order: `schema_drift_check` → `extract_full_to_gcs` → `validate_csv_row_count` → `bq_load_data` → `exec_sp_mysqltobq_load` → `check_row_count` → `check_pk_uniqueness` → `check_null_constraints` → `set_historical_load_complete`.
   - All tasks implement retry logic with exponential back-off. Retry stages must not re-query MySQL if the GCS file is already staged (use XCom GCS URI to resume from `bq_load_data` on retry).
   - The final task `set_historical_load_complete` sets `historical_load_complete = TRUE` in `verizon_data_audit.load_watermark` **exclusively** after all three validation checks (`ROW_COUNT`, `PK_UNIQUENESS`, `NULL_CONSTRAINT`) have written `passed = TRUE` to `reconciliation_log` in the same `run_id`.

2. **Implement the incremental load DAG (`dag_5gcore_employees_incremental_load`):**
   - First task: `ExternalTaskSensor` targeting `dag_5gcore_employees_full_load.set_historical_load_complete`. DAG will not proceed until this sensor resolves `True`.
   - Task order: `sensor_historical_complete` → `schema_drift_check` → `extract_incremental_to_gcs` → `validate_csv_row_count` → `bq_load_data` → `exec_sp_mysqltobq_load` → `check_row_count` → `check_pk_uniqueness` → `check_null_constraints` → `advance_watermark` → `write_load_log`.
   - Same retry-without-MySQL-requery semantics as the full load DAG.

3. **Configure scheduling cadences:** Set `schedule_interval = '@hourly'` and `schedule_interval = '@daily'` as independent schedule options on the incremental DAG. Confirm the correct cadence for `employees` with stakeholders before enabling the scheduler (pending Blocker 2 resolution for any additional tables).

4. **Implement schema drift alert routing:** Wire the destructive drift P1 alert and the additive drift warning alert from Phase 4 Step 1 to Cloud Monitoring notification channels (email and/or webhook). Confirm alert recipients with engineering lead.

5. **Implement DAG failure alerting:** Configure an `on_failure_callback` on both DAGs that publishes a Cloud Monitoring custom metric and triggers the defined alerting policy for DAG-level failures.

## Phase 6: Monitoring, Access Control & Stakeholder Visibility

1. **Configure Cloud Monitoring alerting policies:** Define policies for: (a) DAG failure on either DAG, (b) `reconciliation_log` `passed = FALSE` on any `check_type`, (c) row count mismatch delta exceeding defined threshold. Assign notification channels per the alerting runbook.

2. **Configure Cloud Logging log-based metrics:** Create log-based metrics for schema drift events and load completion events sourced from Cloud Composer task logs, to enable dashboard integration.

3. **Build Looker Studio dashboards:** Connect Looker Studio data sources exclusively to `verizon_data_audit` dataset tables (`load_log`, `reconciliation_log`, `schema_drift_log`). Do **not** connect to `verizon_data_dea`. Provide views covering: load history by date, row count trends, validation pass/fail rates, schema drift event timeline.

4. **Enforce stakeholder access boundaries:** Confirm that stakeholder IAM bindings grant `bigquery.dataViewer` on `verizon_data_audit` only. Confirm zero direct `verizon_data_dea` bindings for non-engineering accounts. Document access grants in the runbook.

5. **Apply column-level security on sensitive fields:** Apply BigQuery column-level access policies restricting `salary`, `commission_pct`, `email`, and `phone_number` columns in `stg_employees` to authorised data engineering and approved analyst roles only.

6. **Document IAM assignments:** Record all service account and user IAM bindings for `verizon_data_dea`, `verizon_data_audit`, the GCS staging bucket, and Cloud Composer in the runbook. Schedule quarterly access review.

## Phase 7: Testing

### TC-01 — MySQL Connectivity via Auth Proxy
- **What:** Cloud SQL Auth Proxy successfully authenticates and connects to `agentichub`.
- **Input:** Composer PythonOperator initiates `pymysql` connection to `34.70.79.163:3306` using username `sa`.
- **Expected:** Connection established, `SELECT 1` returns `1`, no timeout or auth error within 10 seconds.

### TC-02 — Full Historical Extract Row Count Fidelity
- **What:** CSV row count written to GCS matches MySQL source count exactly.
- **Input:** Full extract of `employees` with no watermark filter.
- **Expected:** `wc -l gcs_file - 1 (header) == SELECT COUNT(*) FROM employees` in MySQL. XCom values match. Task succeeds.

### TC-03 — Historical Load Three-Stage Validation Gate
- **What:** All three checkpoints pass before `historical_load_complete` is set.
- **Input:** Full load of `employees` into `stg_employees`.
- **Expected:** `reconciliation_log` contains three rows for the same `run_id` with `check_type IN ('ROW_COUNT', 'PK_UNIQUENESS', 'NULL_CONSTRAINT')` all with `passed = TRUE`. `load_watermark.historical_load_complete = TRUE` only after all three pass.

### TC-04 — Incremental DAG ExternalTaskSensor Gate
- **What:** Incremental DAG is blocked until `historical_load_complete = TRUE`.
- **Input:** Trigger incremental DAG while `historical_load_complete = FALSE` in the watermark table.
- **Expected:** `ExternalTaskSensor` does not resolve; incremental DAG remains in `running/sensing` state and no extraction or load tasks execute.

### TC-05 — Watermark-Driven Incremental Filter
- **What:** Incremental extraction correctly filters rows by `updated_date > max_updated_date`.
- **Input:** Insert 5 new rows into MySQL `employees` with `updated_date` values after the current watermark. Trigger incremental DAG.
- **Expected:** GCS CSV contains exactly 5 rows (plus header). BigQuery `stg_employees` row count increases by 5. `load_watermark.max_updated_date` advances to the maximum of the 5 inserted `updated_date` values.

### TC-06 — MERGE Idempotency on Retry
- **What:** Re-executing `sp_mysqltobq_load.sql` on the same GCS file does not produce duplicate records or corrupt the watermark.
- **Input:** Execute the stored procedure twice consecutively with the same GCS staging file and the same `run_id`.
- **Expected:** `COUNT(DISTINCT employee_id) = COUNT(*)` in `stg_employees` after both executions. Row count is identical after the second execution. `max_updated_date` in `load_watermark` is unchanged by the second execution. *(Confirm Blocker 3 is resolved before executing this test case.)*

### TC-07 — NULL `updated_date` Partition Routing
- **What:** Rows with `updated_date = NULL` land in the BigQuery `__NULL__` partition.
- **Input:** Insert a row into MySQL `employees` with `updated_date = NULL`. Run full or incremental load.
- **Expected:** `SELECT COUNT(*) FROM verizon_data_dea.stg_employees WHERE _PARTITIONTIME IS NULL` returns at least 1. Row is present in `stg_employees` with all non-nullable fields populated.

### TC-08 — PK Uniqueness Failure Triggers Task Failure
- **What:** A duplicate `employee_id` in the staging data causes the PK uniqueness check task to fail.
- **Input:** Artificially insert a duplicate `employee_id` row into `raw_employees_staging` before the stored procedure runs (test environment only).
- **Expected:** `check_pk_uniqueness` task fails. `reconciliation_log` records `passed = FALSE` for `check_type = 'PK_UNIQUENESS'`. DAG is marked failed. No watermark advancement occurs.

### TC-09 — Additive Schema Drift — Auto-Remediation
- **What:** A new nullable column added in MySQL is auto-detected, `stg_employees` is altered, and a warning alert is emitted without blocking the load.
- **Input:** Add a nullable column `middle_name VARCHAR(50)` to MySQL `employees`. Trigger a load run.
- **Expected:** `schema_drift_log` contains a new row with `drift_type = 'ADDITIVE'`, `column_name = 'middle_name'`. `stg_employees` schema in BigQuery gains a `middle_name STRING NULLABLE` column. A Cloud Monitoring warning alert fires. Load completes successfully.

### TC-10 — Destructive Schema Drift — Load Blocked
- **What:** A column drop in MySQL blocks the load and raises a P1 alert.
- **Input:** Drop column `phone_number` from MySQL `employees`. Trigger a load run.
- **Expected:** `schema_drift_check` task fails. `schema_drift_log` records `drift_type = 'DESTRUCTIVE'`, `column_name = 'phone_number'`. A P1 alert fires. No `bq_load_data` or `exec_sp_mysqltobq_load` tasks execute.

### TC-11 — Null Constraint Violation on NOT NULL Field
- **What:** A NULL value in a NOT NULL column is detected and the load is failed.
- **Input:** Insert a row into MySQL `employees` with `first_name = NULL` (bypass MySQL constraint in test environment). Run load.
- **Expected:** `check_null_constraints` task fails. `reconciliation_log` records `passed = FALSE` for `check_type = 'NULL_CONSTRAINT'` referencing `first_name`. No watermark advancement occurs.

### TC-12 — Self-Referencing FK — Top-Level Employee (NULL `manager_id`)
- **What:** Top-level employees with no manager are loaded without error.
- **Input:** Source `employees` row where `manager_id = NULL`.
- **Expected:** Row lands in `stg_employees` with `manager_id = NULL`. No referential integrity error (BigQuery does not enforce FK constraints). PK/null checks pass.

### TC-13 — 60-Day Partition Expiry Enforcement
- **What:** Partitions older than 60 days are expired and not queryable.
- **Input:** Confirm `partition_expiration_days = 60` on `stg_employees`. Query a synthetic partition dated 61+ days ago.
- **Expected:** BigQuery returns zero rows for the expired partition or the partition does not exist. Current partitions are unaffected.

### TC-14 — Stakeholder Access Boundary Enforcement
- **What:** Stakeholder accounts cannot directly query `verizon_data_dea.stg_employees`.
- **Input:** Attempt to execute `SELECT * FROM verizon_data_dea.stg_employees LIMIT 1` using a stakeholder IAM identity.
- **Expected:** BigQuery returns a `403 Access Denied` error. Looker Studio dashboard (connected to `verizon_data_audit` only) remains accessible to the same identity.
