## Phase 2: Cloud Run Job — Ingestion Layer

**Goal:** Build and containerise the Python ingestion job that extracts from MySQL and writes Parquet to GCS.

1. **Implement the pre-flight schema comparison step.** At job startup, fetch live column definitions from `agentichub.INFORMATION_SCHEMA.COLUMNS` for `employees` and diff against the stored schema baseline. Classify each delta:
   - *Additive* (new nullable column present in MySQL, absent in BigQuery): issue `ALTER TABLE verizon_data_dea.stg_employees ADD COLUMN …`, write a `WARNING` record to `bq_load_audit`, and continue execution.
   - *Breaking* (type mutation, column drop, new `NOT NULL` on existing column): write an `ERROR` record to `bq_load_audit` with `schema_validation_status = FAIL`, exit with non-zero code, and do not proceed to extraction.

2. **Implement historical (full) extraction.** Execute `SELECT * FROM agentichub.employees` with no filter. Stream results into a pandas DataFrame. Cast source types to the target schema: MySQL `varchar` → `STRING`, `DECIMAL(8,2)` → `NUMERIC`, `DECIMAL(4,2)` → `NUMERIC`, `TIMESTAMP` → `TIMESTAMP` (UTC). Validate `NOT NULL` columns contain no nulls before writing. Capture the MySQL source row count and store it in the job context for audit.

3. **Implement incremental extraction.** Read the last watermark from `bq_load_audit` for `table_name = 'stg_employees'` and `status = 'SUCCESS'` (latest `completed_at`, `load_type = 'incremental'`). Execute `SELECT * FROM agentichub.employees WHERE updated_date > {last_watermark}`. Apply the same type casting and NOT NULL validation as the full load. Capture source row count.

4. **Write Parquet to GCS.** Serialise the extracted DataFrame to Parquet (snappy compression) and upload to `gs://verizon-data-5gcore-staging/employees/{run_id}/part-*.parquet`. Use `run_id` = `{table}_{load_type}_{YYYYMMDDTHHMMSSZ}`.

5. **Write pre-load audit record.** Insert a record into `bq_load_audit` with `status = 'RUNNING'`, `load_type` (`full` or `incremental`), `source_row_count`, `run_id`, and `started_at`. This record is updated to `SUCCESS` or `FAIL` by the processing phase.

6. **Containerise and deploy.** Package the job as a Python 3.12 container image. Confirm the image is deployed to the Cloud Run Jobs service with the correct service account, Secret Manager volume mount, VPC connector, and environment variables (`PROJECT_ID`, `DATASET`, `GCS_BUCKET`, `TABLE_NAME`).

## Phase 3: BigQuery Processing Layer

**Goal:** Load staged Parquet files into BigQuery and execute the MERGE stored procedure.

1. **BigQuery Load Job.** Trigger a BigQuery Load Job that reads all Parquet files from `gs://verizon-data-5gcore-staging/employees/{run_id}/` into `verizon_data_dea.stg_employees_stage`. The staging table is truncated (`WRITE_TRUNCATE`) before each load. Confirm the load job completes successfully and captures `target_row_count` (staging row count).

2. **Row-count reconciliation assertion (pre-MERGE).** Execute a BigQuery SQL assertion:
   ```sql
   SELECT COUNT(*) AS staging_count FROM verizon_data_dea.stg_employees_stage;
   ```
   Compare `staging_count` against `source_row_count` recorded in `bq_load_audit` for the current `run_id`. If the counts do not match, write `FAIL` to `bq_load_audit` with both counts recorded and halt. Do not proceed to MERGE.

3. **Execute stored procedure `sp_mysqltobq_load`.** Issue:
   ```sql
   CALL verizon_data_dea.sp_mysqltobq_load('stg_employees', 'stg_employees_stage');
   ```
   The stored procedure must implement a `MERGE` from `stg_employees_stage` into `stg_employees` using the composite merge key `(employee_id, updated_date)`. `WHEN MATCHED` → `UPDATE SET` all non-key columns. `WHEN NOT MATCHED` → `INSERT` all columns.

4. **Post-MERGE validation assertions.** Execute the following BigQuery SQL assertions in sequence. Write each result (`PASS` / `FAIL` with counts and timestamps) to `bq_load_audit`:
   - **Assertion A — Row-count match:** `SELECT COUNT(*) FROM verizon_data_dea.stg_employees` where `updated_date` falls within the current load partition range must be ≥ the `source_row_count` from the audit record for this run.
   - **Assertion B — Duplicate composite key check:** `SELECT employee_id, updated_date, COUNT(*) FROM verizon_data_dea.stg_employees GROUP BY 1, 2 HAVING COUNT(*) > 1` must return zero rows.
   - **Assertion C — NOT NULL constraint check:** `SELECT COUNT(*) FROM verizon_data_dea.stg_employees WHERE employee_id IS NULL OR first_name IS NULL OR last_name IS NULL OR email IS NULL OR hire_date IS NULL OR job_id IS NULL OR salary IS NULL OR status IS NULL OR created_date IS NULL` must return zero.

5. **Update audit record.** Update the `bq_load_audit` record for the current `run_id` with `status = 'SUCCESS'` (or `'FAIL'` if any assertion failed), `target_row_count`, `schema_validation_status`, and `completed_at`.

## Phase 4: Orchestration — Cloud Composer DAGs

**Goal:** Wire the ingestion and processing steps into Composer-managed DAGs with correct dependency gating.

> ⚠️ **Blocked by Open Blocker 1:** Schedule intervals (`hourly` vs `daily`) cannot be finalised until confirmed with Yash. Use placeholder schedule variable `{{ var.value.employees_schedule_interval }}` until resolved.

1. **Define Airflow Variables.** In Cloud Composer 2 (Airflow 2.9), register the following Airflow Variables:
   - `employees_schedule_interval` — placeholder until cadence is confirmed.
   - `employees_last_watermark` — seeded to `NULL` for the first run; managed by the incremental DAG thereafter.
   - `gcs_staging_bucket` — `verizon-data-5gcore-staging`.
   - `bq_dataset` — `verizon_data_dea`.

2. **Implement `dag_5gcore_full_load`.** This DAG runs once (no recurring schedule). Task sequence:
   1. `check_connectivity` — validates TCP connection to `34.70.79.163:3306`.
   2. `trigger_cloud_run_full` — triggers the Cloud Run Job with `LOAD_TYPE=full`.
   3. `poll_cloud_run_full` — polls until job completion or failure (non-zero exit → DAG failure).
   4. `trigger_bq_load_job` — runs the BigQuery Load Job from GCS into `stg_employees_stage`.
   5. `call_sp_mysqltobq_load` — executes the stored procedure via BigQuery operator.
   6. `validate_assertions` — runs post-MERGE assertions A, B, C and reads results from `bq_load_audit`.
   7. `gate_full_load_success` — a `BranchPythonOperator` that checks `bq_load_audit` for `status = 'SUCCESS'` on all assertions. On `FAIL`, it routes to `notify_failure` and halts. On `SUCCESS`, it sets a Composer Variable `full_load_validated = true` and routes to `notify_success`.

3. **Implement `dag_5gcore_incremental_load`.** This DAG runs on the confirmed schedule interval. Task sequence:
   1. `gate_check_full_load_validated` — reads Airflow Variable `full_load_validated`. If not `true`, skip all downstream tasks and exit cleanly with a log warning. This enforces the architectural constraint that the incremental pipeline cannot start until the historical load is confirmed.
   2. `check_connectivity` — validates TCP connection.
   3. `read_watermark` — reads latest successful watermark from `bq_load_audit`.
   4. `trigger_cloud_run_incremental` — triggers Cloud Run Job with `LOAD_TYPE=incremental` and the watermark value.
   5. `poll_cloud_run_incremental` — polls until job completion.
   6. `trigger_bq_load_job` — BigQuery Load Job from GCS into `stg_employees_stage`.
   7. `call_sp_mysqltobq_load` — stored procedure execution.
   8. `validate_assertions` — post-MERGE assertions A, B, C.
   9. `update_watermark` — on `SUCCESS`, update `employees_last_watermark` Airflow Variable to `MAX(updated_date)` from the current load batch.
   10. `notify_outcome` — write final status notification.

4. **Configure SLA alerts on DAGs.** Attach SLA miss callbacks to both DAGs referencing the schedule interval once confirmed. SLA breach triggers a Cloud Monitoring alert (email + optional Pub/Sub).

## Phase 5: Monitoring & Alerting

**Goal:** Instrument the pipeline so failures surface automatically without engineering polling.

1. **Cloud Monitoring — Cloud Run Job non-zero exit.** Create an alerting policy on the Cloud Run Jobs log metric filtered for `severity=ERROR` and `jsonPayload.status != 0` on job names matching `5gcore-employees-*`. Notification channel: email; optional Pub/Sub topic for downstream paging.

2. **Cloud Monitoring — BigQuery stored procedure ERROR logs.** Create an alerting policy on BigQuery `severity=ERROR` log entries containing `sp_mysqltobq_load` in the resource labels. Notification channel: email.

3. **Cloud Monitoring — Row-count reconciliation mismatch.** Create a log-based metric on `bq_load_audit` writes where `status = 'FAIL'` and `schema_validation_status = 'FAIL'`. Alert when this metric count exceeds zero within a 1-hour evaluation window. Configure a configurable tolerance threshold for row-count discrepancy exceeding 0.1%.

4. **Looker Studio dashboard.** Connect Looker Studio to `verizon_data_dea.bq_load_audit`. Build the following views accessible without engineering involvement:
   - Load success history (table_name, run_id, load_type, status, started_at, completed_at).
   - Row-count reconciliation trend chart (source_row_count vs. target_row_count over time).
   - Schema validation event log (schema_validation_status per run_id).
   - SLA compliance summary (% of runs completing within SLA window, once cadence is confirmed).

5. **Weekly full-reconciliation job.** Schedule a weekly Cloud Run Job (separate from the daily/hourly incremental) that executes `SELECT COUNT(*) FROM agentichub.employees` and compares against `SELECT COUNT(*) FROM verizon_data_dea.stg_employees`. Alert if discrepancy exceeds 0.1%. Write result to `bq_load_audit` with `load_type = 'reconciliation'`.

## Phase 7: Handover

**Goal:** Deliver complete operational documentation and obtain sign-off.

1. **Update the runbook — Pipeline Design Summary.**

   Document the following procedures:
   - **Re-running a failed full load:** Set `full_load_validated = false` in Airflow Variables, clear the `stg_employees_stage` table, trigger `dag_5gcore_full_load` manually, and confirm all assertions pass before re-enabling the incremental DAG.
