## ⚠️ Open Blockers

> **These blockers must be resolved before development begins. Do not proceed with Phase 2 or later until each is answered.**

1. **Network Connectivity Pattern (Critical)** — Is `34.70.79.163` a Cloud SQL instance (as implied by `instance_connection_name: verizon-data:us-central1:mysql-druid-metadatastore`) or an external/on-premise host? The answer determines whether the Dataflow pipeline uses the Cloud SQL Connector library or requires VPC peering / firewall allowlisting. The JDBC connection strategy in Phase 2 cannot be finalised until this is resolved.

2. **Incremental Load Column (Critical)** — Does the `EMPLOYEES` table expose a reliable high-watermark column (`updated_at TIMESTAMP`, `created_at DATETIME`, or an auto-increment ID) to support delta extraction? If not, full-table extraction (`WRITE_TRUNCATE`) is required on every hourly run, with significant cost and storage implications. This determines the BigQuery write mode and pipeline merge logic in Phase 2–3.

3. **PII & Data Classification (Critical)** — What are the data sensitivity and access-control requirements for the `EMPLOYEES` table? Column-level encryption, BigQuery row-level security policies, VPC Service Controls perimeter membership, and audit logging obligations are unspecified and must be addressed before real employee data flows through the Dev pipeline.

## ⚠️ Risks & Mitigations

*Reference only — do not action as steps.*

| Risk | Mitigation |
|---|---|
| **Credential Exposure** | All MySQL credentials for user `sa` are stored exclusively in Google Secret Manager. Reference secrets at runtime via Airflow Connections backed by Secret Manager, or direct Secret Manager SDK calls in Dataflow pipeline code. Rotate the `sa` password before Dev pipeline deployment; enforce least-privilege on the MySQL account (SELECT on `agentichub.EMPLOYEES` only). |
| **Network Connectivity** | Confirm Cloud SQL vs. external host (see Blocker 1). For Cloud SQL: use the Cloud SQL Connector library or Auth Proxy sidecar on Dataflow workers. For external hosts: establish VPC peering or Cloud Interconnect with firewall rules scoped to the Dataflow worker subnet CIDR range. |
| **Data Quality** | Implement row-count reconciliation (source `COUNT(*)` vs. BigQuery row count post-load), NOT NULL checks on `employee_id`, `first_name`, `last_name`, `email`, `hire_date`, `job_id`, `salary`, `status`, `created_date`, and schema drift detection in the Beam pipeline. Publish pipeline quality metrics to Cloud Monitoring with alerting thresholds before Dev sign-off. |
| **Self-Referencing FK Integrity** | `manager_id` is a nullable self-reference to `employee_id`. Rows where `manager_id IS NOT NULL` must have a corresponding `employee_id` in the same load batch. Validate referential consistency post-load in BigQuery. |
| **Dataflow Startup Latency** | Dataflow incurs a 2–3 minute startup overhead per job. The hourly cron (`0 * * * *`) provides adequate headroom for a 10 GB dataset at auto-scaled throughput, but monitor P95 job durations and set a DAG execution timeout of 45 minutes to detect regressions. |

## Phase 3: Dataflow Apache Beam Pipeline Development

1. **Scaffold the pipeline.** Create a Python (or Java) Apache Beam pipeline project under `src/pipelines/employees_ingestion/`. Include a `pipeline.py` entry point, a `transforms/` package, and a `config/` directory for pipeline options.

2. **Implement the JDBC MySQL source.** Configure the Dataflow JDBC I/O connector (or `ReadFromJdbc`) to connect to `34.70.79.163:3306`, database `agentichub`, table `EMPLOYEES`, user `sa`, with the password retrieved at runtime from Secret Manager. Apply the network pattern resolved in Phase 1 (Cloud SQL Connector or direct JDBC).

   - If **incremental mode**: parameterise the JDBC query with a `WHERE updated_date > :last_watermark` predicate. Accept `last_watermark` as a pipeline option passed by the Airflow DAG.
   - If **full-table mode**: `SELECT * FROM EMPLOYEES` with no predicate.

3. **Implement the schema validation transform.** For each record, assert:
   - `EMPLOYEE_ID` is non-null and castable to INT64.
   - `FIRST_NAME`, `LAST_NAME`, `EMAIL`, `HIRE_DATE`, `JOB_ID`, `SALARY`, `STATUS`, `CREATED_DATE` are non-null.
   - `STATUS` is one of `{ACTIVE, INACTIVE}`.
   - Route invalid records to a dead-letter GCS path (`gs://<bucket>/dead-letter/employees/`). Do not fail the pipeline on individual bad rows.

4. **Implement the type-casting transform.** Cast MySQL source types to BigQuery target types:
   - `EMPLOYEE_ID` (decimal 10,0) → INT64
   - `HIRE_DATE` (MySQL DATE string) → BigQuery DATE
   - `CREATED_DATE`, `UPDATED_DATE` (MySQL DATETIME/TIMESTAMP) → BigQuery TIMESTAMP
   - `SALARY`, `COMMISSION_PCT` (MySQL decimal) → BigQuery NUMERIC
   - All remaining string columns: strip trailing whitespace.

5. **Implement the GCS raw landing write.** Write the validated, cast records to `gs://<bucket>/raw/employees/YYYY/MM/DD/HH/` in Parquet format before the BigQuery write step. This preserves a raw audit copy independent of the BigQuery layer.

6. **Implement the BigQuery write.** Use the Beam `WriteToBigQuery` connector targeting `customer360_dev.stg_employees`. Apply the write mode determined in Phase 2, Step 3. Set `create_disposition=CREATE_NEVER` to prevent accidental schema creation.

7. **Implement row-count reconciliation.** After the BigQuery write completes, emit a pipeline metric (`rows_written_to_bq`) and compare it against the source `COUNT(*)` captured at pipeline start. If the delta exceeds a configurable threshold (default 0), write an anomaly event to Cloud Logging and fail the pipeline with a non-zero exit code.

8. **Implement schema drift detection.** At pipeline startup, retrieve the current MySQL `EMPLOYEES` column list via JDBC metadata. Compare it against the expected schema defined in pipeline config. If any column is added, removed, or has a changed type, log a `SCHEMA_DRIFT_DETECTED` event to Cloud Logging and fail the pipeline before reading any data.

9. **Parameterise all pipeline options.** Ensure the following are pipeline options (not hardcoded): GCS bucket URI, BigQuery table reference, Secret Manager secret name, MySQL host/port/database, last watermark value (incremental mode only), dead-letter path, and row-count anomaly threshold.

## Phase 4: Cloud Composer Airflow DAG Development

1. **Author the `customer360_employees_dag.py` DAG.** Place it in the Composer DAGs bucket. Configure:
   - `schedule_interval='0 * * * *'` (hourly)
   - `catchup=False`
   - `max_active_runs=1`
   - `dagrun_timeout=timedelta(minutes=45)`
   - `tags=['customer360', 'employees', 'SCRUM-5']`

2. **Implement the Dataflow trigger task.** Use `DataflowCreatePythonJobOperator` (or equivalent) to submit the Beam pipeline from Phase 3. Pass all pipeline options as operator parameters. Retrieve the MySQL password from Secret Manager via the Airflow Connection backed by Secret Manager — do not pass the password as a plaintext DAG parameter.

3. **Implement the watermark management task (incremental mode only).** Before triggering Dataflow, read the last successful watermark from a BigQuery metadata table (`customer360_dev.pipeline_watermarks`) or an Airflow Variable. After a successful Dataflow run, update the watermark to `MAX(updated_date)` from `stg_employees`.

4. **Implement the post-load validation task.** After the Dataflow job completes successfully, execute a BigQuery query to assert:
   - Row count in `stg_employees` for the current load window is greater than zero.
   - No `employee_id` values are NULL.
   - The `MAX(updated_date)` in `stg_employees` is ≥ the watermark used for the current run (incremental mode).

5. **Implement DAG-level failure alerting.** Configure an `on_failure_callback` that writes a structured failure event to Cloud Logging and triggers a Cloud Monitoring alert notification (email / PagerDuty channel — use the channel already provisioned).

6. **Test the DAG import.** Run `python customer360_employees_dag.py` locally (or in a CI step) to confirm zero import errors before uploading to the Composer bucket.

## Phase 5: Monitoring & Alerting Configuration

1. **Define Cloud Monitoring dashboard.** Create a dashboard (via Terraform or the Console) covering: Dataflow job status (running / succeeded / failed), Dataflow worker CPU and memory utilisation, pipeline throughput (elements/sec, bytes/sec), `rows_written_to_bq` custom metric, and DAG success/failure counts from Cloud Logging.

2. **Configure alerting policies.** Create alerting policies for:
   - Dataflow job in `JOB_STATE_FAILED` for more than 0 occurrences in a 1-hour window.
   - `rows_written_to_bq` metric deviates by more than the configured anomaly threshold from a rolling 7-day baseline.
   - DAG `customer360_employees_dag` has not completed successfully within 90 minutes of its scheduled start time.
   - `SCHEMA_DRIFT_DETECTED` log entry appears in Cloud Logging (any occurrence).

3. **Validate log sinks.** Confirm that Dataflow worker logs, Beam pipeline logs, and DAG task logs are all routing to the designated Cloud Logging log bucket for the Customer360 project. Spot-check a test run's logs are queryable via `gcloud logging read`.
