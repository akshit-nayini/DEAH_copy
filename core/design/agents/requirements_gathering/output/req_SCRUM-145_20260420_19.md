## Objective
Design and implement a reliable, scalable pipeline to load the employees table from the MySQL agentichub database (Cloud SQL: verizon-data:us-central1:mysql-druid-metadatastore) into BigQuery (verizon-data.verizon_data_dea.employees), with a one-time full historical load followed by ongoing daily scheduled incremental loads, supporting analytics, reporting, and long-term storage for the 5gcore programme.

## Business Context
The 5gcore programme requires a reliable MySQL-to-BigQuery data pipeline to make employee data available for analytics, reporting, and long-term storage. Stakeholder Yash requires full historical data completeness as a validated baseline before incremental loads are enabled. The dataset is estimated at 5 GB with 1 GB/month growth and a 60-day retention policy. Key business outcomes include stakeholder confidence in data quality and independent stakeholder access to load validation reports without engineering assistance.

## Functional Requirements
- Build a batch data ingestion pipeline from MySQL agentichub.employees to BigQuery verizon_data_dea.employees
- Perform a one-time full historical load of all records from MySQL agentichub.employees into BigQuery verizon_data_dea.employees
- Implement chunking or batching strategy to handle 5 GB data volume and prevent timeouts and memory failures during historical load
- Implement retry logic for failed extraction chunks during the historical load
- Set up daily scheduled incremental loads from MySQL to BigQuery following historical load sign-off
- Block incremental pipeline activation until historical load validation sign-off is formally documented
- Implement pre-load schema validation covering field names, data types, and null handling
- Perform post-load row count reconciliation between MySQL and BigQuery after every load execution
- Enforce primary key uniqueness checks in BigQuery employees table after every load
- Validate null and datatype constraints against schema definitions before each load
- Define and enforce a handling strategy for new or unexpected MySQL columns prior to their inclusion in loads
- Maintain audit logs for every historical and incremental load execution
- Configure monitoring and alerting for pipeline failures with early detection capability
- Provide stakeholders with visibility into load success and data quality via logs or a self-service dashboard
