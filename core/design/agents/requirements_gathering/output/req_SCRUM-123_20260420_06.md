## Objective
Design and implement a reliable and scalable pipeline to load MySQL tables (agentichub database, employees table) into Google BigQuery (verizon_data_deah dataset) on a scheduled basis, supporting both a one-time full historical load and ongoing scheduled incremental loads (hourly or daily per table) to enable analytics, reporting, and long-term data storage.

## Business Context
This initiative enables analytics and reporting teams to access MySQL source data reliably within BigQuery at scale. By migrating and continuously syncing data, the organization achieves long-term data storage, 100% historical completeness, and self-service analytics capabilities. Stakeholder Yash requires independent visibility into load health without developer involvement, making operational transparency a first-class requirement alongside data correctness.

## Functional Requirements
- Execute a one-time full historical load of the MySQL employees table (agentichub) into BigQuery verizon_data_deah dataset with 100% data completeness and no data loss
- Implement a scheduled incremental data load pipeline that continuously syncs new and updated records from MySQL employees table to BigQuery on a per-table cadence (hourly or daily)
- Track incremental changes using timestamp-based or primary key delta strategies, consistently applied across all runs
- Perform pre-load schema compatibility checks covering data types, null handling, field names, and new column handling strategy before each load execution
- Perform post-load row count validation between MySQL source and BigQuery target for every load cycle
- Enforce primary key uniqueness in BigQuery after every load execution
- Define and implement a predefined handling strategy for new columns appearing in MySQL before they are loaded into BigQuery
- Generate and store audit logs for every load execution, accessible after each run
- Implement a monitoring dashboard or log-based visibility layer enabling stakeholders to independently review load success without developer involvement
- Trigger monitoring alerts on pipeline failures within a defined SLA
- Gate incremental load activation: incremental pipeline must not begin until historical load validation is fully passed and signed off
- Log and report null constraint violations before marking any load as complete
- Prevent data duplication and gaps across all incremental load cycles
